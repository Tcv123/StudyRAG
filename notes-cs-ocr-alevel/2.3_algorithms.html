2.3.1 Algorithm Analysis and Design

What is an Algorithm?

An algorithm is a finite, ordered sequence of unambiguous, executable instructions that transforms one or more inputs into one or more well-defined outputs in a finite amount of time.

The four essential properties of a formal algorithm are:

  • Finiteness — the algorithm must terminate after a finite number of steps. An infinite loop is therefore not an algorithm.
  • Definiteness (determinism) — each step must be precisely and unambiguously defined; given the same input it produces the same output.
  • Clearly defined inputs — the algorithm takes zero or more specified inputs drawn from a specified set.
  • Clearly defined outputs — it produces one or more outputs that have a specified relationship to the inputs.
  • Effectiveness — each step must be basic enough that it can in principle be carried out exactly by a person using paper and pencil.
In a "define algorithm" question, do not just write "a set of steps". Mention finite, unambiguous, and inputs produce outputs for full marks.

Why Analyse Algorithms?

Two algorithms may both solve the same problem correctly, but have radically different performance. Analysing algorithms lets us:

  • Predict how running time and memory usage will scale as the input size grows.
  • Compare competing algorithms before implementing them.
  • Identify bottlenecks and guarantees for worst-case behaviour (critical in real-time, medical or safety-critical systems).
  • Choose the right algorithm for the constraints of a particular platform (embedded, mobile, server).

There are two resources we usually measure:

  • Time complexity — how the number of basic operations grows with input size n.
  • Space complexity — how the memory used grows with input size n, excluding the input itself (i.e. the auxiliary space).

We do not measure raw wall-clock time, because that depends on the CPU, language, compiler and competing load. Counting operations and expressing the count as a function of n gives a platform-independent comparison.

Big-O Notation

Big-O notation describes an asymptotic upper bound on the growth of a function. We say f(n) = O(g(n)) if there exist constants c > 0 and n0 > 0 such that 0 ≤ f(n) ≤ c · g(n) for all n ≥ n0. Intuitively, g(n) is a ceiling that f(n) never rises above (up to a constant factor) once n is large enough.

Two key simplifications follow from this definition:

  • Drop constants: O(3n) = O(n); O(n/2) = O(n). Constant factors reflect hardware, not the algorithm.
  • Keep only the dominant term: O(n2 + 5n + 9) = O(n2). For large n the highest-order term overwhelms everything else.

We care primarily about the worst case because it provides a guarantee: no matter what input arrives, the algorithm will never be slower than this. Average-case analysis is also used but usually requires assumptions about input distribution that may not hold in practice.

Big-O is an upper bound; Big-Ω (Omega) is a lower bound; Big-Θ (Theta) is a tight bound. OCR questions use Big-O, but knowing it is specifically the worst-case upper bound will stop you losing marks for saying "the algorithm runs in O(n)" when you mean on average.

Best, Average and Worst Case

The running time of most algorithms depends not just on the size of the input but also on its shape. We therefore distinguish three cases:

  • Best case — the input arrangement that makes the algorithm finish fastest (e.g. target at position 0 for linear search).
  • Average case — the expected running time over all "typical" inputs. Often requires a probabilistic assumption.
  • Worst case — the input that maximises running time (e.g. target not in the list at all for linear search).

A classic example where the three differ dramatically is quick sort:

CaseInput shapeComplexity
BestPivot always splits list roughly in halfO(n log n)
AverageRandom input, random or median-of-three pivotO(n log n)
WorstAlready sorted (or reverse-sorted) list with first-element pivotO(n2)
When asked "why is bubble sort best-case O(n)?", your answer must mention the early-termination flag: if no swap happens on a pass, the list is sorted and the algorithm terminates after one linear pass.

Space Complexity

Space complexity measures the amount of auxiliary memory an algorithm uses as a function of the input size n, excluding the memory required for the input itself.

Two useful distinctions:

  • In-place algorithms use O(1) or O(log n) extra memory — they rearrange data within the original array without needing a large second copy. Quick sort and insertion sort are in-place.
  • Out-of-place algorithms need O(n) or more auxiliary space. Merge sort is out-of-place because merging two halves requires a temporary buffer of length n.

On memory-constrained devices (microcontrollers, mobile, embedded) a slightly slower in-place algorithm is often preferred over a faster out-of-place one.

Analysing a Simple Code Fragment

To practise extracting a big-O from code, count the number of times the innermost statement executes as a function of n.

FOR i := 0 TO n - 1          // outer loop runs n times
    FOR j := 0 TO n - 1      // inner loop runs n times
        x := x + 1           // executes n * n = n² times
    ENDFOR
ENDFOR

Total work: n × n = n2, hence O(n2).

i := 1
WHILE i < n
    print(i)
    i := i * 2              // i doubles each pass
ENDWHILE

The value of i goes 1, 2, 4, 8, …, and the loop exits when i ≥ n. That takes log2 n iterations, giving O(log n).

FOR i := 1 TO n
    j := 1
    WHILE j < n
        j := j * 2
    ENDWHILE
ENDFOR

Outer loop n times, inner loop log n times per iteration, so O(n log n) overall. This pattern is exactly what happens in merge sort and heap sort.

When asked to "find the complexity" of a code fragment, state clearly: (1) identify the innermost statement; (2) count how many times it runs as a function of n; (3) simplify (drop constants and lower-order terms).

2.3.2 Searching Algorithms

Linear Search

Linear (sequential) search examines each element of a list in turn from the start until either the target value is found or the end of the list is reached.

Linear search works on any list — the list does not need to be sorted. This is its only real advantage.

FUNCTION linearSearch(list, target)
    FOR i := 0 TO length(list) - 1
        IF list[i] = target THEN
            RETURN i                 // found — return index
        ENDIF
    ENDFOR
    RETURN -1                        // not found
ENDFUNCTION

Worked example. Searching for 17 in [4, 22, 17, 11, 8]:

  1. i=0: compare 4 vs 17 — no match.
  2. i=1: compare 22 vs 17 — no match.
  3. i=2: compare 17 vs 17 — match! Return 2.

Linear search on a linked list

Because a linked list does not provide O(1) random access, binary search cannot be used on it directly — finding the middle element takes O(n) in itself. For linked lists, linear search is therefore the default:

FUNCTION linearSearchLinkedList(head, target)
    current := head
    position := 0
    WHILE current != NULL
        IF current.value = target THEN
            RETURN position
        ENDIF
        current := current.next
        position := position + 1
    ENDWHILE
    RETURN -1
ENDFUNCTION

Average performance of linear search (expected number of comparisons) is about n / 2 when the target is present and uniformly distributed, which still simplifies to O(n).

Binary Search

Binary search finds a value in a sorted list by repeatedly halving the search space. At each step the middle element is compared to the target; the half that cannot contain the target is discarded.
Binary search requires sorted data. Every long-answer question must state this assumption. If the data is not sorted, you must sort it first (costing O(n log n)) or use linear search.

Three pointers are maintained: low, high, and the midpoint mid = (low + high) DIV 2.

Iterative version

FUNCTION binarySearchIter(list, target)
    low  := 0
    high := length(list) - 1
    WHILE low <= high
        mid := (low + high) DIV 2
        IF list[mid] = target THEN
            RETURN mid
        ELSEIF list[mid] < target THEN
            low := mid + 1
        ELSE
            high := mid - 1
        ENDIF
    ENDWHILE
    RETURN -1
ENDFUNCTION

Recursive version

FUNCTION binarySearchRec(list, target, low, high)
    IF low > high THEN
        RETURN -1                    // base case — not found
    ENDIF
    mid := (low + high) DIV 2
    IF list[mid] = target THEN
        RETURN mid
    ELSEIF list[mid] < target THEN
        RETURN binarySearchRec(list, target, mid + 1, high)
    ELSE
        RETURN binarySearchRec(list, target, low, mid - 1)
    ENDIF
ENDFUNCTION

Why O(log n)?

After 1 comparison we have n/2 elements left, after 2 we have n/4, after k comparisons we have n/2k. The search finishes when only one element remains: n/2k = 1, i.e. k = log2 n. Doubling the list size adds only one more comparison.

Worked example — find 17 in [1, 4, 7, 11, 14, 17, 22, 28, 31, 35]

Binary Tree Search

A binary search tree (BST) is a binary tree with the property that for every node, all values in the left subtree are smaller and all values in the right subtree are larger. Searching a BST resembles binary search but on a linked structure rather than an array.

FUNCTION bstSearch(node, target)
    IF node = NULL THEN
        RETURN NULL                  // not found
    ENDIF
    IF target = node.value THEN
        RETURN node
    ELSEIF target < node.value THEN
        RETURN bstSearch(node.left, target)
    ELSE
        RETURN bstSearch(node.right, target)
    ENDIF
ENDFUNCTION
CaseTree shapeComplexity
Best / AverageBalanced tree (height ≈ log n)O(log n)
WorstDegenerate tree — values inserted in sorted order, height = n (effectively a linked list)O(n)
Explain why the worst case is O(n): if nodes are inserted in already-sorted order (e.g. 1, 2, 3, 4, ...), every insertion goes right, producing a tree of height n where search degenerates to linear search. Self-balancing trees (AVL, red–black) are designed to prevent this.

Bubble Sort

Bubble sort makes repeated passes through a list, comparing each adjacent pair and swapping them if they are out of order. Larger elements "bubble up" to the end of the list one position per pass. 
PROCEDURE bubbleSort(list)
    n := length(list)
    FOR i := 0 TO n - 2
        swapped := FALSE
        FOR j := 0 TO n - 2 - i      // last i items already sorted
            IF list[j] > list[j + 1] THEN
                swap(list[j], list[j + 1])
                swapped := TRUE
            ENDIF
        ENDFOR
        IF NOT swapped THEN
            RETURN                   // early termination — already sorted
        ENDIF
    ENDFOR
ENDPROCEDURE

Worked example — one full pass through [5, 2, 8, 1, 9]

Insertion Sort

Insertion sort builds a sorted sub-array at the start of the list. For each subsequent element it shuffles that element leftwards past larger values until it sits in the correct position of the sorted prefix. 
PROCEDURE insertionSort(list)
    FOR i := 1 TO length(list) - 1
        key := list[i]
        j   := i - 1
        WHILE j ≥ 0 AND list[j] > key
            list[j + 1] := list[j]   // shift right
            j := j - 1
        ENDWHILE
        list[j + 1] := key           // insert into gap
    ENDFOR
ENDPROCEDURE

Merge Sort

Merge sort is a divide-and-conquer algorithm: the list is recursively split in half until sub-lists contain one element; pairs of sorted sub-lists are then merged back together to produce longer sorted sub-lists. 
FUNCTION mergeSort(list)
    IF length(list) ≤ 1 THEN
        RETURN list                  // base case — already sorted
    ENDIF
    mid   := length(list) DIV 2
    left  := mergeSort(list[0 .. mid - 1])
    right := mergeSort(list[mid .. length(list) - 1])
    RETURN merge(left, right)
ENDFUNCTION

FUNCTION merge(left, right)
    result := []
    i := 0
    j := 0
    WHILE i < length(left) AND j < length(right)
        IF left[i] ≤ right[j] THEN
            append result, left[i]
            i := i + 1
        ELSE
            append result, right[j]
            j := j + 1
        ENDIF
    ENDWHILE
    append result, remaining elements of left
    append result, remaining elements of right
    RETURN result
ENDFUNCTION

Why O(n log n)?

Splitting halves the sub-list size each recursion, giving log2 n levels of recursion. At each level the merge step does O(n) work in total (the sub-lists at a level together cover all n elements). Total: n × log n.

Merge sort trades O(n) extra memory for its guaranteed O(n log n) bound. If memory is tight, prefer quick sort (in-place).

Why merge sort is stable

During the merge step, the comparison left[i] ≤ right[j] (note the "≤", not "<") prefers the left element when values are equal. Because the left sub-list came from earlier in the original array, equal keys keep their original relative order. This is exactly what stability means — and it is why many database and spreadsheet sorts are based on merge sort, since users often sort by one column and then another, relying on the earlier sort to survive the later one.

External sorting

"External" sorting means sorting data that is too large to fit in main memory, e.g. a 500 GB log file on a machine with 16 GB of RAM. Merge sort is the algorithm of choice: split the file into chunks each small enough to fit in RAM, sort each chunk with an in-memory sort, write back to disk, then k-way merge the sorted chunks by streaming from each file and repeatedly picking the smallest current element. This minimises disk seeks and respects the streaming nature of the data.

Quick Sort

Quick sort is a divide-and-conquer algorithm. A pivot element is chosen and the list is partitioned into elements < pivot, the pivot itself, and elements ≥ pivot. Quick sort is then applied recursively to the two sub-lists.

Lomuto partition scheme

FUNCTION quickSort(list, low, high)
    IF low < high THEN
        p := partition(list, low, high)
        quickSort(list, low, p - 1)
        quickSort(list, p + 1, high)
    ENDIF
ENDFUNCTION

FUNCTION partition(list, low, high)
    pivot := list[high]              // Lomuto: pivot = last element
    i := low - 1
    FOR j := low TO high - 1
        IF list[j] ≤ pivot THEN
            i := i + 1
            swap(list[i], list[j])
        ENDIF
    ENDFOR
    swap(list[i + 1], list[high])    // place pivot in final position
    RETURN i + 1
ENDFUNCTION

Hoare scheme (briefly): two pointers move inward from opposite ends, swapping elements they find on the wrong side of the pivot until they cross. Hoare does fewer swaps on average but is trickier to code correctly.

Pivot selection strategies

  • First or last element — simplest; degenerates to O(n2) on already-sorted input.
  • Random element — expected O(n log n) regardless of input shape; eliminates adversarial inputs.
  • Median-of-three — pick the median of first, middle, last element. Cheap, usually good, used in most production implementations (e.g. C++ std::sort).
  • Middle element — simple and works well on sorted input.

If you are asked why quick sort is usually preferred over merge sort in practice, mention: (1) in-place — only O(log n) space, (2) better cache locality from contiguous swaps, (3) smaller hidden constants. Merge sort wins only when worst-case guarantees or stability are required.

Why the O(n2) worst case happens

If the pivot is always the smallest (or largest) element of the current sub-array — which happens with a "first element" or "last element" pivot when the input is already sorted or reverse-sorted — then each partition shaves just one element off the sub-array. Instead of halving the work at every step we decrement by one, giving a recursion depth of n rather than log n. Total work becomes n + (n-1) + (n-2) + … + 1 = n(n+1)/2, i.e. O(n2).

Randomised pivot selection or median-of-three makes this worst case vanishingly unlikely. Production libraries such as the C++ STL's std::sort use introsort, which starts as quick sort but falls back to heap sort if recursion goes too deep — giving the best of both worlds (quick sort's speed with an O(n log n) worst-case guarantee).

Tail-call optimisation and stack depth

A naive recursive implementation of quick sort can use O(n) stack space in the worst case. Production implementations recurse only on the smaller sub-array and iterate over the larger one (or use tail-call optimisation where available). This bounds the stack depth at O(log n) even in adverse cases.

2.3.4 Graph Algorithms

Graph Recap

(See 1.4.2 for internals.) A graph is a set of vertices (nodes) connected by edges. Edges can be directed or undirected, and weighted (edges carry a cost, e.g. distance, travel time, bandwidth). Two common representations:

  • Adjacency matrix — a V×V grid; fast edge lookup, O(V2) space even if sparse.
  • Adjacency list — each vertex stores a list of neighbours; space-efficient for sparse graphs.

Traversal algorithms (BFS, DFS) visit every reachable vertex in a specific order, running in O(V + E) time. OCR treats these in 1.4; here we focus on weighted-graph shortest-path algorithms.

Dijkstra's Shortest-Path Algorithm

Dijkstra's algorithm (1956) finds the shortest path from a single source vertex to every other vertex in a graph with non-negative edge weights. It is a greedy algorithm that repeatedly extracts the unvisited vertex with the smallest known distance and relaxes all its outgoing edges.
Dijkstra's algorithm requires non-negative edge weights. If negative weights are present, use Bellman–Ford instead. This is a favourite exam gotcha.

The algorithm

  1. Assign every vertex a tentative distance: 0 for the source, ∞ for every other vertex.
  2. Mark all vertices unvisited; put the source into a priority queue keyed by distance.
  3. Loop while the priority queue is non-empty:
    1. Extract the vertex u with the smallest tentative distance.
    2. Mark u as visited (its distance is now final).
    3. For each unvisited neighbour v of u:
      • Compute alt = dist[u] + weight(u, v).
      • If alt < dist[v], set dist[v] := alt, set prev[v] := u, and add/update v in the priority queue. This step is called relaxation.
  4. When the queue is empty, dist[] holds the shortest distance from source to every vertex, and prev[] lets you reconstruct the path.
FUNCTION dijkstra(graph, source)
    FOR each vertex v IN graph
        dist[v] := INFINITY
        prev[v] := UNDEFINED
    ENDFOR
    dist[source] := 0
    pq := new PriorityQueue()
    pq.insert(source, 0)

    WHILE pq is not empty
        u := pq.extractMin()
        FOR each neighbour v OF u
            alt := dist[u] + weight(u, v)
            IF alt < dist[v] THEN
                dist[v] := alt
                prev[v] := u
                pq.insertOrDecreaseKey(v, alt)
            ENDIF
        ENDFOR
    ENDWHILE
    RETURN dist, prev
ENDFUNCTION

Reconstructing the Shortest Path

Dijkstra produces a prev[] array where prev[v] is the predecessor of v on the best-known path from source. To recover the actual vertex sequence, start at the destination and follow prev pointers back to the source, then reverse.

FUNCTION reconstructPath(prev, destination)
    path := []
    current := destination
    WHILE current != UNDEFINED
        prepend path, current
        current := prev[current]
    ENDWHILE
    RETURN path
ENDFUNCTION

In the worked example above, starting from E: prev[E]=F, prev[F]=C, prev[C]=A, prev[A]=undefined. Reversed: A→C→F→E, total distance 20.

A* (A-Star) Algorithm

A* is an informed search algorithm that extends Dijkstra with a heuristic h(n) that estimates the remaining cost from node n to the goal. Instead of selecting the node with the smallest g(n), it selects the one with the smallest f(n) = g(n) + h(n). A* directs search towards the goal rather than exploring outward uniformly.
  • g(n) — the known cost of the shortest path from start to n.
  • h(n) — the estimated remaining cost from n to the goal (problem-specific).
  • f(n) = g(n) + h(n) — the estimated total path cost through n.

Heuristic properties

  • Admissible — h(n) never overestimates the true remaining cost. Admissibility guarantees A* returns an optimal path.
  • Consistent (monotonic) — satisfies the triangle inequality: for every edge from n to m, h(n) ≤ weight(n,m) + h(m). Consistency implies admissibility and lets A* settle each node's g the first time it is expanded (no re-expansion needed).

Common heuristics

  • Straight-line (Euclidean) distance — √((x₁-x₂)2 + (y₁-y₂)2). Admissible in continuous 2D space.
  • Manhattan distance — |x₁-x₂| + |y₁-y₂|. Admissible on a grid where you can only move up/down/left/right.
  • Chebyshev distance — max(|dx|, |dy|). Admissible when diagonal moves cost the same as orthogonal.
  • Zero heuristic — h(n) = 0 for all n. Always admissible; reduces A* exactly to Dijkstra.
The standard marking point is: "When h(n) = 0, A* is equivalent to Dijkstra's algorithm." If the heuristic is a poor estimate (large overestimate) A* can return a sub-optimal path.

Pseudocode

FUNCTION aStar(graph, start, goal, h)
    openSet  := PriorityQueue()
    openSet.insert(start, h(start))
    gScore[start] := 0
    fScore[start] := h(start)

    WHILE openSet is not empty
        current := openSet.extractMin()        // lowest f-score
        IF current = goal THEN
            RETURN reconstructPath(prev, current)
        ENDIF
        FOR each neighbour n OF current
            tentativeG := gScore[current] + weight(current, n)
            IF tentativeG < gScore[n] THEN
                prev[n]   := current
                gScore[n] := tentativeG
                fScore[n] := tentativeG + h(n)
                IF n NOT IN openSet THEN
                    openSet.insert(n, fScore[n])
                ENDIF
            ENDIF
        ENDFOR
    ENDWHILE
    RETURN FAILURE                             // no path
ENDFUNCTION

When to use A*

  • Video games — pathfinding for NPCs and units on tile/grid maps.
  • Robotics — motion planning through obstacle fields.
  • GPS / satnav — routing through road networks with distance/time heuristics.
  • Puzzle solving — 8-puzzle, 15-puzzle (heuristic: Manhattan distance of each tile from its goal position).

Trade-offs of A*

  • Memory — A* stores every expanded node in the open and closed sets; this can balloon in large search spaces. Memory-bounded variants (IDA*, SMA*) exist.
  • Heuristic quality — a tight, accurate admissible heuristic dramatically reduces the number of expansions. A loose heuristic (e.g. h = 0) makes A* no better than Dijkstra.
  • Inadmissible heuristics — sometimes faster but may return a sub-optimal path. Games often use a scaled-up heuristic (h' = 1.5 h) to trade off a little optimality for much faster searches.
  • Dynamic environments — if edge costs change (e.g. traffic updates), the path must be re-planned. D* (dynamic A*) and D* Lite incrementally repair the existing solution rather than re-running from scratch.

Worked Mini-Questions

Q1. State the time complexity of the following algorithm in Big-O notation, and justify.

FUNCTION mystery(n)
    total := 0
    FOR i := 1 TO n
        FOR j := i TO n
            total := total + 1
        ENDFOR
    ENDFOR
    RETURN total
ENDFUNCTION

Answer: The inner loop runs n times when i = 1, n-1 times when i = 2, …, 1 time when i = n. Total iterations = n + (n-1) + … + 1 = n(n+1)/2. Dropping constants and lower-order terms: O(n2).

Q2. An unsorted list of 1 024 integers is searched for a value. On average, how many comparisons does a linear search make? What about binary search?

Answer: Linear search performs on average about n/2 = 512 comparisons. Binary search cannot be used on an unsorted list — it would have to be sorted first (O(n log n) = ~10 000 operations) before achieving log2(1024) = 10 comparisons per search. Binary search only pays off if many searches are performed against the same list.

Q3. Name one scenario where quick sort's worst case occurs and suggest a mitigation.

Answer: Sorting an already-sorted (or reverse-sorted) list with a first-element or last-element pivot produces maximally unbalanced partitions, giving O(n2). Mitigations: random pivot, median-of-three pivot, or switching to introsort (quick sort + heap sort fallback).

Q4. Explain briefly why A* returns an optimal path when the heuristic is admissible.

Answer: An admissible heuristic never overestimates the true remaining cost, so f(n) = g(n) + h(n) is always a lower bound on the cost of any path through n. A* expands the node with the smallest f-value first; the first time it expands the goal, no unexplored node can possibly improve on that f-value, so the path found is optimal.

Q5. Trace one pass of bubble sort on [4, 1, 3, 2].

Answer:

  • Compare 4 vs 1 → swap → [1, 4, 3, 2]
  • Compare 4 vs 3 → swap → [1, 3, 4, 2]
  • Compare 4 vs 2 → swap → [1, 3, 2, 4]

After one pass the maximum (4) is in its final position. The list still requires more passes to finish (swap 3 and 2).

Classic long-answer question: "Compare merge sort and quick sort. Explain which you would use and why." Summary

  • An algorithm is a finite, unambiguous procedure that transforms inputs into outputs.
  • Big-O notation expresses asymptotic worst-case upper bounds on time or space.
  • Linear search is O(n) and works on any list; binary search is O(log n) but requires sorted data.
  • Bubble and insertion sort are O(n2) quadratic sorts; merge sort guarantees O(n log n) at the cost of O(n) extra memory; quick sort is O(n log n) on average, O(n2) worst, but in-place and usually fastest in practice.
  • Dijkstra's algorithm finds shortest paths in graphs with non-negative weights by greedy relaxation.
  • A* extends Dijkstra with a heuristic h(n); when h = 0, A* reduces to Dijkstra. A* is widely used in games, robotics and GPS.
  • Space/time trade-offs, stability and adaptivity are all legitimate criteria when choosing between algorithms.
