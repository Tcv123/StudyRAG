1.1.1 Von Neumann Architecture and CPU Components

The Von Neumann architecture is a computer design where data and instructions are stored in the same memory and are processed sequentially (one instruction at a time). This is the basis of virtually all modern general-purpose computers. The key insight is that programs are data — they can be loaded, modified, and replaced without physically rewiring the machine.

Before Von Neumann's design (proposed in 1945), early computers like ENIAC had to be physically rewired to perform different tasks. Von Neumann realised that if the program itself were stored in memory alongside the data, the computer could be reprogrammed simply by loading different instructions — no rewiring needed. This concept is called the stored program concept and remains the foundation of computing today.

The alternative is the Harvard architecture, where instructions and data have separate memory stores and separate buses. This allows instructions and data to be fetched simultaneously, which is why it is used in some embedded systems and digital signal processors where speed is critical. However, the Von Neumann design is simpler and more flexible, which is why general-purpose computers use it.

Key CPU Components

The CPU (Central Processing Unit) is the brain of the computer. It fetches instructions from memory, decodes them to understand what action is required, and then executes them. The CPU is made up of several key components that work together:

ComponentFunctionDetail
ALU (Arithmetic Logic Unit)Performs all arithmetic calculations and logical comparisonsArithmetic: add, subtract, multiply, divide. Logic: AND, OR, NOT, comparisons (equal, greater than, less than). Results are stored in the accumulator. The ALU is the only part of the CPU that actually performs calculations on data.
Control Unit (CU)Coordinates all CPU operationsFetches instructions from memory, decodes them into operations, and directs other components (ALU, registers, buses) to carry them out. It sends control signals via the control bus. The CU does not process data itself — it orchestrates the work of other components, like a conductor directing an orchestra.
RegistersUltra-fast storage inside the CPUSmall number of storage locations that hold data, addresses, or instructions currently being used. Much faster than RAM (accessed in a single clock cycle). Each register typically holds one word of data (e.g., 32 or 64 bits). There are only a handful of registers, each with a specific purpose.
CacheFast memory between CPU and RAMStores copies of frequently accessed data and instructions from RAM so the CPU does not have to wait for slower RAM. L1 cache is fastest and smallest (per core, ~32-64 KB); L2 is larger but slightly slower (~256 KB-1 MB); L3 is largest and shared between cores (~4-32 MB). Cache is made of SRAM (faster than the DRAM used in main memory).
ClockSynchronises all CPU operationsAn electronic oscillator that generates a regular pulse. Each pulse is one clock cycle, and one or more stages of an instruction happen per cycle. Clock speed is measured in GHz (billions of cycles per second). A 3.5 GHz processor performs 3.5 billion clock cycles every second. The clock ensures all components work in step with each other.

Buses — The CPU's Communication Highways

The CPU communicates with memory and I/O devices via a system bus consisting of three sub-buses. Think of buses as motorways connecting different parts of the computer:

BusCarriesDirectionAnalogy
Address BusMemory addresses — tells memory which location to read from or write toUnidirectional (CPU → memory only)Like a postcode on a letter — it tells the postal system WHERE to deliver, but the letter itself travels separately
Data BusActual data being transferred between CPU, memory, and I/O devicesBidirectional (data flows both ways)Like the letter inside the envelope — the actual content being sent or received
Control BusControl signals: read, write, clock, interrupt, bus request/grantBidirectionalLike traffic lights controlling the flow — it tells components when to send, receive, read, or write

The width of the address bus determines how many unique memory locations the CPU can address. A 32-bit address bus can address 232 = 4,294,967,296 locations (4 GB). A 64-bit address bus can theoretically address 264 locations (16 exabytes). The width of the data bus determines how much data can be transferred in a single operation — a wider data bus means more data per transfer, improving performance.

The address bus is unidirectional (CPU sends addresses to memory, never the other way). The data bus is bidirectional (data can be read FROM memory or written TO memory). This is a commonly examined distinction. Remember: the address bus width determines how much memory can be addressed; the data bus width determines how much data can be transferred at once.

1.1.2 Registers

Registers are the fastest storage locations in a computer, located directly inside the CPU. They are used to hold data, addresses, and instructions that the CPU is currently working with. Each register has a specific role in the fetch-decode-execute cycle.

RegisterFull NameHoldsPurpose During the FDE Cycle
PCProgram CounterAddressHolds the address of the NEXT instruction to be fetched. Incremented by 1 after each fetch (unless a branch instruction changes it to a different address). The PC is what allows the CPU to know where it is in a program.
MARMemory Address RegisterAddressHolds the memory address the CPU wants to read from or write to. Connected to the address bus. The MAR is like a request slip sent to memory: "give me the data at this location". The MAR is loaded from the PC during fetch, or from the CIR operand during execute.
MDRMemory Data RegisterDataHolds the data just read from memory, or the data about to be written to memory. Connected to the data bus. Also called the Memory Buffer Register (MBR). It acts as a buffer between the CPU and memory — all data passing between them goes through the MDR.
ACCAccumulatorDataStores the result of the most recent ALU operation (arithmetic or logic). Acts as the primary working register for calculations. For example, after an ADD instruction, the sum is stored here.
CIRCurrent Instruction RegisterData (instruction)Holds the instruction currently being decoded and executed. The CU splits its contents into the opcode (what to do, e.g., ADD, LOAD, STORE, BRANCH) and the operand (the data or address to use). The CIR is needed so the MDR is free to be used for data access during the execute phase.
Always distinguish between registers that hold addresses (PC, MAR) and those that hold data (MDR, ACC, CIR). The PC holds the address of the next instruction, not the current one. The MAR holds addresses; the MDR holds data. A common mistake is confusing the MAR and MDR — remember: MAR = Address, MDR = Data. This distinction is required for full marks.

How Registers Connect to the Buses

Understanding which registers connect to which buses is important:

  • The MAR connects to the address bus — it sends the address of the memory location to be accessed
  • The MDR connects to the data bus — it receives data read from memory or sends data to be written
  • The CU uses the control bus to send signals such as "memory read" or "memory write"
  • The PC, ACC, and CIR are internal registers — they do not connect directly to external buses but communicate with other registers inside the CPU

Why Each Register Exists — The Architectural Reason

Understanding the purpose behind each register, rather than just its function, gives you a much stronger answer in 4- and 6-mark questions:

  • PC exists because the CPU must always know the address of the next instruction to fetch. Without the PC, the CPU would have no way to keep track of where it is in a program. The PC is modified by branch instructions, which is the hardware mechanism behind IF statements, loops, and function calls.
  • MAR exists to hold the address being sent on the address bus long enough for memory to respond. Memory access takes time (RAM takes ~50-100 ns); the MAR decouples the CPU from memory timing — the CPU can load the MAR and then do other internal work while waiting. Without the MAR, the address bus would need to be driven continuously, tying up the CPU.
  • MDR exists as a buffer between the fast CPU and the relatively slow RAM. Data arriving from memory over the data bus is placed in the MDR first, then transferred internally within the CPU at CPU speed. This decoupling means the CPU does not have to wait at the exact moment data arrives on the bus. Without the MDR, every memory access would require the CPU to halt and hold the bus until the transfer was complete.
  • CIR exists so that the instruction being decoded and executed is preserved while the MDR is reused. During the EXECUTE phase of a LOAD instruction, the MAR and MDR are needed again to access the data address. If the instruction were still in the MDR, it would be overwritten. Moving the instruction to the CIR during FETCH frees the MDR for data transfers during EXECUTE.
  • ACC exists as the primary working register for the ALU. The ALU is wired to take input from the ACC and return results to the ACC. Without a dedicated accumulator, every intermediate calculation result would need to be written back to memory, which would be extremely slow. The ACC keeps intermediate results on-chip, inside the CPU, at register speed.
The MAR and MDR exist to decouple the CPU from memory timing. Memory is much slower than the CPU; these registers act as buffers so the CPU does not have to pause while memory responds. This is why the instruction is moved from MDR to CIR during fetch — to free the MDR for use during execute. This level of reasoning (not just "what" but "why") is what distinguishes a full-mark answer from a partial one.

1.1.3 The Fetch-Decode-Execute (FDE) Cycle

The CPU continuously repeats three stages for every instruction. This cycle runs billions of times per second and is the fundamental operation of every Von Neumann computer. Everything a computer does — from displaying text to playing games — is ultimately the result of billions of FDE cycles.

1. Fetch

The purpose of the fetch phase is to retrieve the next instruction from memory and bring it into the CPU.

  1. The address in the PC is copied to the MAR. (The CPU now knows which memory address to read from.)
  2. The PC is incremented by 1 (PC = PC + 1), so it now points to the next instruction. (This happens immediately so the PC is ready for the next cycle.)
  3. The address in the MAR is sent along the address bus to memory.
  4. The CU sends a memory read signal on the control bus. (This tells memory to send back the data at the requested address.)
  5. The instruction at that memory address is transferred along the data bus into the MDR.
  6. The contents of the MDR are copied to the CIR. (This frees the MDR for use during the execute phase.)

Summary: PC → MAR → Memory → MDR → CIR. PC incremented.

2. Decode

The purpose of the decode phase is to work out what the instruction means and prepare the CPU to execute it.

  1. The Control Unit examines the instruction in the CIR.
  2. It splits the instruction into the opcode (the operation to perform, e.g., ADD, LOAD, STORE, BRANCH) and the operand (the data or address to use).
  3. The CU determines which control signals to send and to which components. For example, if the opcode is ADD, the CU knows it needs to activate the ALU and route data appropriately.

3. Execute

The purpose of the execute phase is to carry out the decoded instruction.

  1. The decoded instruction is carried out. This could be:
    • An arithmetic/logic operation — the ALU performs a calculation (e.g., ADD, SUBTRACT, COMPARE) and the result goes to the ACC
    • A data transfer — load data from memory into a register (LOAD), or store register data to memory (STORE). For LOAD: the operand address goes to the MAR, memory is read, data comes back via data bus to MDR, then MDR contents go to ACC.
    • A branch — load a new address into the PC, changing the sequence of execution. This is how loops and IF statements work at the hardware level.
  2. The result may be stored in the ACC or written back to memory via the MDR.
  3. The cycle then repeats from the fetch phase — the PC already points to the next instruction.

1.1.4 CPU Performance Factors

Three main factors affect how fast a CPU can process instructions. In exam questions, you must be able to explain how each factor improves performance AND its limitations.

FactorHow It Improves PerformanceLimitations
Clock SpeedMeasured in GHz. Higher clock speed = more FDE cycles per second. A 4 GHz processor does 4 billion cycles/second. Doubling clock speed roughly doubles throughput for single-threaded tasks. Going from 2 GHz to 4 GHz means twice as many instructions can be processed per second.Higher clock speed generates more heat and uses more power. Physical limits exist (signal propagation, heat dissipation). Diminishing returns — modern CPUs hit thermal walls above ~5 GHz. Requires better cooling solutions (larger heatsinks, liquid cooling), which increases cost and size.
Number of CoresEach core is an independent processor with its own ALU, CU, and registers. A quad-core CPU can run 4 threads simultaneously. Good for multi-tasking (running multiple programs) and multi-threaded applications (video editing, gaming, web servers).Software must be written to use multiple threads to benefit. A single-threaded program uses only one core regardless of how many cores exist. Not all tasks can be divided into parallel sub-tasks. Overhead from coordinating between cores can reduce the expected speedup. Doubling cores does NOT double speed.
Cache SizeCache stores frequently accessed data closer to the CPU. Larger cache = more data available without slow RAM access = fewer delays. L1 is fastest/smallest (~1 nanosecond access); L3 is slowest/largest (~10-20 ns). RAM access takes ~50-100 ns. A cache hit avoids the ~50-100 ns penalty of going to RAM.Cache is very expensive per byte (SRAM vs DRAM). Diminishing returns as cache grows larger (larger caches are physically slower to search). Takes up physical space on the CPU die, leaving less room for other components. The relationship between cache size and performance is logarithmic, not linear.

Performance Factors — Summary for Comparisons

Question: "A user wants to improve their computer's performance for
video editing. Explain how each CPU factor would help."

Clock speed: Higher clock speed means more FDE cycles per second,
so each individual operation completes faster. Video encoding
involves billions of calculations, each processed faster.

Cores: Video editing software is designed to be multi-threaded.
A quad-core processor can encode 4 sections of video simultaneously,
roughly quadrupling the speed compared to a single core. This is
where cores provide the most benefit.

Cache: Larger cache stores more of the video frame data close
to the CPU, reducing the number of slow RAM accesses needed.
This reduces delays between calculations.

Best answer: For video editing, MORE CORES would provide the
greatest benefit because video editing software is highly
multi-threaded and can effectively utilise all available cores.
When explaining CPU performance, always state how the factor helps AND its limitations. For example: "More cores allow multiple threads to execute simultaneously, improving performance for multi-threaded applications such as video editing. However, if the software is single-threaded, additional cores provide no benefit." This balanced answer scores higher than just stating the positive effect. Also know that increasing one factor cannot compensate for all workloads — a high clock speed with one core is great for single-threaded tasks but poor for multi-tasking.
Cache holds copies of frequently accessed data and instructions from RAM. When the CPU requests data, it first checks the cache (cache hit = returned very quickly, ~1 ns). If not found (cache miss), the CPU must access slower RAM (~50-100 ns) and a copy is stored in cache for future use. Larger cache means more data stored close to the CPU, reducing the frequency of slow RAM accesses.
Common exam question: "Suggest two ways to increase CPU performance." Never just say "faster processor" — name specific factors and explain how they help. Strong answers: (1) "Increase the clock speed — more FDE cycles are completed per second, so each instruction is processed faster." (2) "Increase the cache size — more data is stored close to the CPU, reducing the number of slow main memory accesses." Saying "use a faster computer" or "upgrade the CPU" will score zero marks.

Worked Exam Question — CPU Performance

Question: "A computer has a 3.2 GHz dual-core processor with 4 MB L3 cache.
Explain two ways its performance could be improved." (4 marks)

Answer:

(1) Increase the clock speed (e.g., to 4.0 GHz).
    The clock speed determines how many FDE cycles are completed per second.
    At 4.0 GHz, each core completes 4 billion cycles per second compared to
    3.2 billion — an increase of 25%. Each individual instruction is processed
    faster, so the CPU completes more work in a given time.
    Limitation: higher clock speed generates more heat and uses more power,
    requiring better (and more expensive) cooling.

(2) Increase the cache size (e.g., to 16 MB L3 cache).
    Larger cache can store more frequently accessed data and instructions
    from RAM. When the CPU needs data, it is more likely to find it in fast
    cache (cache hit) rather than needing to access much slower RAM (cache
    miss). Reducing RAM accesses reduces delays between instructions.
    Limitation: cache is expensive per byte, and very large caches have
    diminishing returns as they become physically slower to search.

Alternative valid answers: add more cores (enables parallel execution of
multiple threads); upgrade to faster RAM (reduces the penalty of a cache miss).

1.1.5 Embedded Systems

An embedded system is a computer built into a larger device to perform one specific dedicated function. The program is stored in ROM (or flash memory) and runs automatically when the device is powered on. Unlike a general-purpose computer, an embedded system is designed for a single task and typically has no user-accessible operating system.

Embedded systems are everywhere — there are far more embedded processors in the world than general-purpose computers. Your home likely contains dozens: in the microwave, washing machine, central heating thermostat, car, TV remote, and even in your toothbrush if it is electric. They are purpose-built, inexpensive, and reliable because they only need to do one thing well.

Characteristics of Embedded Systems

  • Performs one specific dedicated function — it does not run arbitrary user programs. A washing machine controller will only ever control the washing machine.
  • Program stored in ROM or flash memory — runs automatically on power-on without booting an OS. The program is permanent and begins executing immediately.
  • Limited or no user interface — may have buttons, switches, or LEDs but not a keyboard, mouse, or monitor. Interaction is minimal and specific to the device's function.
  • Real-time response — must respond within a guaranteed time frame. In a car's anti-lock braking system, a delay of even a fraction of a second could be dangerous. This is why embedded systems often use a Real-Time Operating System (RTOS) or no OS at all.
  • Low power, compact, low cost — designed to be small, energy-efficient, and inexpensive to mass-produce. Many run on battery power for years (e.g., a pacemaker).
  • Highly reliable — must work continuously without crashing. A pacemaker cannot "blue screen" or need rebooting. Embedded systems are often tested far more rigorously than desktop software.
  • Difficult to update — firmware updates may be infrequent or impossible once deployed. Modern embedded systems with internet connectivity (IoT devices) can receive updates, but many older devices cannot.

For embedded systems questions, always give a specific example and state its dedicated function. Do not just say "a washing machine" — say "a washing machine controller that manages water temperature, spin speed, and cycle timing." Also know that embedded systems use ROM (non-volatile, runs automatically) rather than loading software from a hard drive. A common exam question gives a scenario and asks you to explain why an embedded system is more suitable than a general-purpose computer — focus on: single function, low power, low cost, high reliability, real-time response, and compact size.

1.1.6 Secondary Storage Types

Secondary storage is non-volatile (data is retained when power is removed) and is used for long-term storage of the OS, programs, and user files. The CPU cannot access secondary storage directly — data must first be loaded into RAM. There are four main types, each with different technologies, speeds, costs, and suitable uses.
