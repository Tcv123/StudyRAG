4.2 Fundamentals of Data Structures

Computer Science — AQA A-Level (7517) — Paper 1: On-screen Programming

Topic 4.2 sits at the heart of Paper 1. Data structures provide the shapes in which data is stored so that algorithms can operate efficiently. Expect AQA to ask you to describe a data structure, trace its operations, compare alternative implementations, and justify which one to pick for a given scenario. You must know the difference between an abstract data type (the logical contract) and its possible implementations (how it is actually stored in memory).

4.2.1 Data Structures — General

A data structure is an organised collection of related data elements, together with a set of operations that may be performed on them. It describes how data is stored so that it can be accessed and modified efficiently.

Static vs Dynamic Data Structures

AspectStaticDynamic
SizeFixed at compile time; cannot grow or shrink.Grows and shrinks at run time as items are added or removed.
Memory allocationAllocated in one contiguous block on the stack or in the data segment.Allocated on the heap, often in separately-linked nodes.

Memory wasteWastes space if only partially filled; overflows if the declared size is exceeded.Uses exactly what it needs, but pays per-node overhead for pointers.

Abstract Data Type vs Implementation

An abstract data type (ADT) is a mathematical / logical model of a data structure, defined by the operations that may be performed on it and the contract those operations obey — independent of how they are programmed. An implementation is the concrete arrangement of memory (and code) that realises an ADT.

For instance, the stack ADT is specified by four operations (push, pop, peek, isEmpty) and the LIFO contract. That same ADT can be implemented using a fixed-size array plus an integer top-pointer, or using a singly-linked list of dynamically-allocated nodes. Client code that uses only the ADT operations will work with either implementation unchanged — this is the power of abstraction.

If AQA asks "what is an abstract data type?" state: (1) a logical description of a data structure, (2) defined by its operations and their behaviour, (3) with the implementation hidden from the user. A two-mark answer needs at least two of these three points.

4.2.2 Single- and Multi-Dimensional Arrays

An array is a static data structure consisting of a fixed-size, ordered collection of elements of the same data type, stored in a single contiguous block of memory and accessed by an integer index.

Indexing and Bounds

  • Most languages use zero-based indexing: a 5-element array has valid indices 0, 1, 2, 3, 4.
  • AQA pseudocode is commonly one-based or allows the programmer to declare bounds explicitly: DECLARE a : ARRAY[1..10] OF Integer.
  • Accessing an index outside the declared bounds raises an IndexOutOfRange exception (or produces undefined behaviour in C).
  • Access is O(1): the address of element a[i] is computed as base + i × elementSize.

Memory Layout of 2D Arrays — Row-Major vs Column-Major

Computer memory is one-dimensional, so a 2D array must be flattened. There are two widely-used conventions:

  • Row-major order (C, C++, Java, Python NumPy default) — rows are stored one after another. For an m × n array, element a[i][j] sits at offset (i × n + j) × elementSize.
  • Column-major order (Fortran, MATLAB, R) — columns are stored one after another. Element a[i][j] sits at offset (j × m + i) × elementSize.
Array:   a[0][0] a[0][1] a[0][2]
         a[1][0] a[1][1] a[1][2]

Row-major memory:    [a00][a01][a02][a10][a11][a12]
Column-major memory: [a00][a10][a01][a11][a02][a12]

The distinction matters for cache performance: iterating in the same order as storage minimises cache misses. A loop that traverses a row-major array row-by-row is much faster than column-by-column.

Higher-Dimensional Arrays

A 3D array generalises naturally — e.g. voxel[x][y][z] for a cube of values. In row-major layout the address of a[i][j][k] in a d1 × d2 × d3 array is base + (i × d2 × d3 + j × d3 + k) × elementSize. AQA occasionally asks students to compute such an offset.

// Declaring and using a 2D array in AQA pseudocode
DECLARE grid : ARRAY[1..3, 1..3] OF Integer

FOR r ← 1 TO 3
    FOR c ← 1 TO 3
        grid[r, c] ← (r - 1) * 3 + c
    ENDFOR
ENDFOR

OUTPUT grid[2, 3]     // prints 6
If an AQA question gives you a 2D array and asks for the address of an element, always state the formula base + (row × cols + col) × size (row-major) or state your assumption explicitly. Examiners want to see the offset calculation, not just the final number.

4.2.3 Fields, Records and Files

A record is a composite (heterogeneous) data type consisting of a fixed collection of named fields, each of which may be of a different data type. A file is a named collection of records held on a backing store (disk, SSD, tape) and persisting beyond program execution.

Records

TYPE Student = RECORD
    studentID : Integer
    surname   : String
    forename  : String
    dob       : Date
    average   : Real
ENDRECORD

s : Student
s.studentID ← 14072
s.surname   ← "Lovelace"

Contrast with an array, which is homogeneous (every element has the same type). Records group logically-related information into a single entity that can be passed, returned and stored as a unit.

Serial and sequential files differ only in ordering: both are read in order. Random files trade insert-order simplicity for near-constant look-up time at the cost of potential collisions (see hash tables below).

4.2.5 Queues

A queue is a First-In-First-Out (FIFO) abstract data type. Items are added at the rear (enQueue) and removed from the front (deQueue). The first element added is always the first one removed — mirroring a real-world queue at a supermarket till.

Standard Operations

  • enQueue(item) — add item to the rear. Fails if isFull().
  • deQueue() — remove and return the front item. Fails if isEmpty().
  • peek() / front() — return the front item without removing it.
  • isEmpty() — return True if the queue has no items.
  • isFull() — return True if the queue has no space (bounded implementations only).

Array-Backed Linear Queue — the Problem

The naive implementation uses an array with two pointers (front and rear). Each deQueue increments the front pointer, so the queue drifts rightwards through the array and wastes the space at the left. Eventually rear hits the end even though the left of the array is empty.

Circular Queue (Ring Buffer)

The standard fix is to treat the array as circular: when a pointer passes the last index it wraps around to index 0 using the modulo operator.

CONSTANT SIZE = 5
DECLARE q     : ARRAY[0..SIZE-1] OF Integer
DECLARE front : Integer ← 0
DECLARE rear  : Integer ← 0
DECLARE count : Integer ← 0

PROCEDURE enQueue(item)
    IF count = SIZE THEN
        OUTPUT "Queue full"
    ELSE
        q[rear] ← item
        rear  ← (rear + 1) MOD SIZE
        count ← count + 1
    ENDIF
ENDPROCEDURE

FUNCTION deQueue() RETURNS Integer
    IF count = 0 THEN
        OUTPUT "Queue empty" ; RETURN -1
    ELSE
        item  ← q[front]
        front ← (front + 1) MOD SIZE
        count ← count - 1
        RETURN item
    ENDIF
ENDFUNCTION

Starting with front = 0, rear = 0, count = 0:

Linked-List Queue

An alternative implementation uses nodes on the heap, each with a data field and a next pointer. The queue object stores just two pointers: head (front) and tail (rear).

TYPE Node = RECORD
    data : Integer
    next : Pointer TO Node
ENDRECORD

head ← NULL
tail ← NULL

PROCEDURE enQueue(x)
    newNode ← NEW Node
    newNode.data ← x
    newNode.next ← NULL
    IF tail = NULL THEN
        head ← newNode
    ELSE
        tail.next ← newNode
    ENDIF
    tail ← newNode
ENDPROCEDURE

Priority Queue

A priority queue is a queue in which each element has an associated priority. deQueue() returns the element with the highest priority, not necessarily the one added earliest. Ties are usually broken by insertion order (FIFO).

Implementations include: a sorted linked list, an unsorted array scanned linearly for the highest priority, or a binary heap (the most efficient, giving O(log n) insert and remove). Priority queues underlie CPU scheduling, Dijkstra's shortest-path algorithm, and print-job queues.

When a 3-mark question says "describe a queue", write all three of: (1) FIFO discipline, (2) enQueue at rear and deQueue at front, (3) a real-world example (print queue, keyboard buffer, OS process queue). One mark each.

4.2.6 Stacks

A stack is a Last-In-First-Out (LIFO) abstract data type. Items are pushed onto the top and popped from the top. Only the top item is ever accessible. Think of a pile of plates: you add and remove from the top; you never pull one out of the middle.

Standard Operations

  • push(item) — place item on top of the stack. Fails if isFull().
  • pop() — remove and return the top item. Fails if isEmpty().
  • peek() / top() — return the top item without removing it.
  • isEmpty(), isFull() — as for queues.

Array Implementation with a Top Pointer

CONSTANT SIZE = 100
DECLARE stack : ARRAY[0..SIZE-1] OF Integer
DECLARE top   : Integer ← -1      // -1 means empty

PROCEDURE push(item)
    IF top = SIZE - 1 THEN
        OUTPUT "Stack overflow"
    ELSE
        top ← top + 1
        stack[top] ← item
    ENDIF
ENDPROCEDURE

FUNCTION pop() RETURNS Integer
    IF top = -1 THEN
        OUTPUT "Stack underflow" ; RETURN -1
    ELSE
        item ← stack[top]
        top  ← top - 1
        RETURN item
    ENDIF
ENDFUNCTION

FUNCTION peek() RETURNS Integer
    IF top = -1 THEN RETURN -1 ELSE RETURN stack[top] ENDIF
ENDFUNCTION

Applications of Stacks

  • Expression evaluation — in particular Reverse Polish (postfix) notation.
  • Function call / recursion support — each call pushes a stack frame; RETURN pops one.
  • Undo / redo in editors — each edit pushes a reversible action; undo pops.
  • Back button in web browsers — URLs pushed on navigation.
  • Depth-first search on graphs / trees — explicit stack replaces recursion.
  • Syntax parsing — matching brackets, compiler symbol tables.

Worked Example — Reverse Polish Evaluation

Evaluate the postfix expression 5 1 2 + 4 * + 3 −. Read left to right: if it is a number, push it; if it is an operator, pop two operands, apply, push the result.

AQA's favourite stack question is a Reverse Polish trace. Present your answer as a three-column table (Token, Action, Stack after). Always show the stack contents explicitly after every token — the marks are awarded per row, not per final answer.

4.2.7 Graphs

A graph is a non-linear data structure consisting of a set of vertices (nodes) and a set of edges (connections) between pairs of vertices. Formally, G = (V, E).

Graph Terminology

  • Directed graph (digraph) — each edge has a direction, e.g. one-way streets; edge (A, B) ≠ edge (B, A).
  • Undirected graph — edges are unordered pairs; connection works both ways.
  • Weighted graph — each edge carries a numerical weight (distance, cost, capacity).
  • Connected graph — a path exists between every pair of vertices.
  • Cycle — a path that starts and ends at the same vertex via distinct edges.
  • Degree of a vertex — the number of edges incident to it (in-degree and out-degree for digraphs).
  • Tree — a connected, acyclic, undirected graph. A tree on n vertices has exactly n − 1 edges.

Adjacency Matrix Representation

A V × V matrix where M[i][j] is 1 (or the edge weight) if there is an edge from vertex i to vertex j, and 0 otherwise.

Undirected graph: A—B, A—C, B—D, C—D

         A  B  C  D
      A [0, 1, 1, 0]
      B [1, 0, 0, 1]
      C [1, 0, 0, 1]
      D [0, 1, 1, 0]

For undirected graphs the matrix is symmetric along the leading diagonal. Storage is O(V2) regardless of edge count.

Adjacency List Representation

For each vertex, store a list of the vertices it is connected to. Total storage is O(V + E).

A → [B, C]
B → [A, D]
C → [A, D]
D → [B, C]

When to Choose Which

When AQA asks "which representation would you use for a road network of a city?" say adjacency list because road networks are sparse (each junction has only a few roads leaving it, so E is close to V not V2). For a completely-connected tournament graph, say adjacency matrix.

4.2.8 Trees

A tree is a connected, acyclic graph with a designated root vertex. Every non-root vertex has exactly one parent. A tree with n vertices has exactly n − 1 edges. Nodes with no children are called leaves.

Tree Terminology

  • Root — the unique topmost node; it has no parent.
  • Parent / child — immediate predecessor / successor.
  • Sibling — nodes sharing the same parent.
  • Ancestor / descendant — any node on the path from root / to leaves.
  • Leaf (external node) — has no children.
  • Depth of a node — number of edges from the root to that node.
  • Height of a tree — maximum depth of any leaf (height of single-node tree is 0).
  • Subtree — a node together with all its descendants.

Binary Trees

A binary tree is a tree in which every node has at most two children, conventionally called left and right. Special cases:

  • Full binary tree — every internal node has exactly two children.
  • Complete binary tree — every level is filled left-to-right, with only the last level possibly short.
  • Balanced — for every node, the heights of its two subtrees differ by no more than 1.

Binary Search Tree (BST)

A binary search tree (BST) is a binary tree that maintains the BST property: for every node N, every key in N's left subtree is strictly less than N's key, and every key in its right subtree is strictly greater. This property enables O(log n) search, insert and delete on a balanced tree.

Building a BST — Worked Example

Insert the sequence 50, 30, 70, 20, 40, 60, 80 into an initially empty BST.

  1. 50 becomes the root.
  2. 30 < 50 → left child of 50.
  3. 70 > 50 → right child of 50.
  4. 20 < 50 → go left to 30; 20 < 30 → left child of 30.
  5. 40 < 50 → go left to 30; 40 > 30 → right child of 30.
  6. 60 > 50 → go right to 70; 60 < 70 → left child of 70.
  7. 80 > 50 → go right to 70; 80 > 70 → right child of 70.
               50
             /    \
           30      70
          /  \    /  \
        20   40  60   80

BST Insertion (Recursive Pseudocode)

TYPE TreeNode = RECORD
    key   : Integer
    left  : Pointer TO TreeNode
    right : Pointer TO TreeNode
ENDRECORD

FUNCTION insert(node, key) RETURNS Pointer TO TreeNode
    IF node = NULL THEN
        newNode ← NEW TreeNode
        newNode.key ← key
        newNode.left ← NULL
        newNode.right ← NULL
        RETURN newNode
    ENDIF
    IF key < node.key THEN
        node.left ← insert(node.left, key)
    ELSE IF key > node.key THEN
        node.right ← insert(node.right, key)
    ENDIF
    RETURN node
ENDFUNCTION

Searching a BST

FUNCTION search(node, target) RETURNS Boolean
    IF node = NULL THEN RETURN False ENDIF
    IF node.key = target THEN RETURN True ENDIF
    IF target < node.key THEN
        RETURN search(node.left, target)
    ELSE
        RETURN search(node.right, target)
    ENDIF
ENDFUNCTION

Worst-case complexity is O(h) where h is the tree's height. For a balanced tree h = ⌈log2n⌉, giving O(log n). A pathologically unbalanced tree (keys inserted in sorted order) degenerates to h = n, so search becomes O(n).

Tree Traversals

A traversal visits every node exactly once in a defined order. The three canonical depth-first orders differ only in where the current node is visited relative to its subtrees.

TraversalOrder (for each node)Result on example BSTTypical use
Pre-orderNode, Left, Right50, 30, 20, 40, 70, 60, 80Copy a tree; prefix notation.
In-orderLeft, Node, Right20, 30, 40, 50, 60, 70, 80Visit BST in sorted order.
Post-orderLeft, Right, Node20, 40, 30, 60, 80, 70, 50Delete / free a tree; postfix notation.

A breadth-first (level-order) traversal visits nodes level by level and requires a queue (not a stack). Result: 50, 30, 70, 20, 40, 60, 80.

PROCEDURE inOrder(node)
    IF node ≠ NULL THEN
        inOrder(node.left)
        OUTPUT node.key
        inOrder(node.right)
    ENDIF
ENDPROCEDURE

    IF node ≠ NULL THEN
        OUTPUT node.key
        preOrder(node.left)
        preOrder(node.right)
    ENDIF
ENDPROCEDURE

    IF node ≠ NULL THEN
        postOrder(node.left)
        postOrder(node.right)
        OUTPUT node.key
    ENDIF
ENDPROCEDURE
For BST-insertion questions, always write out each insertion step comparing the new key against every node on the path. Examiners award a mark per correct comparison as well as for the final tree diagram.

General Rooted Trees

A general tree allows any number of children per node. It is often represented using first-child / next-sibling pointers so each node still needs only two pointer fields. File-system directories, XML/HTML document trees, and organisation charts are all general rooted trees.

4.2.9 Hash Tables

A hash table is a data structure that maps keys to values by passing the key through a hash function to obtain an array index. If the hash function spreads keys evenly, insert, look-up and delete run in O(1) average time.

Properties of a Good Hash Function

  • Deterministic — the same key always hashes to the same index.
  • Uniform distribution — output values are spread evenly across the array's indices to minimise collisions.
  • Fast to compute — hashing should add negligible overhead.
  • Use the full key — every bit of the key should influence the output; otherwise similar keys collide.
// A simple hash function for an integer key
FUNCTION hash(key, tableSize) RETURNS Integer
    RETURN key MOD tableSize
ENDFUNCTION

// A simple hash function for a string key
FUNCTION hashString(s, tableSize) RETURNS Integer
    total ← 0
    FOR i ← 1 TO LENGTH(s)
        total ← (total * 31 + ASC(s[i])) MOD tableSize
    ENDFOR
    RETURN total
ENDFUNCTION

Collisions

A collision occurs when two distinct keys hash to the same index. The pigeonhole principle guarantees collisions are unavoidable in any table with more possible keys than slots.

Collision Resolution — Chaining

Each slot in the array holds a linked list (or other small container) of all keys that hashed there. Insertion adds to the front of the list; look-up scans the list.

index 0: → (key="tom", val=25)
index 1: → (key="ann", val=18) → (key="ian", val=42)   // collision
index 2:
index 3: → (key="eve", val=31)

Collision Resolution — Open Addressing with Linear Probing

All entries live in the array itself. On collision, try the next slot, then the next, wrapping with modulo, until an empty slot is found.

FUNCTION insertLinear(key, value)
    i ← hash(key, SIZE)
    WHILE table[i] ≠ NULL AND table[i].key ≠ key DO
        i ← (i + 1) MOD SIZE
    ENDWHILE
    table[i] ← (key, value)
ENDFUNCTION

Linear probing is cache-friendly but suffers from primary clustering: long runs of occupied cells cause subsequent insertions to pile up. Quadratic probing (step by 1, 4, 9, 16, ...) and double hashing (step by a second hash of the key) reduce clustering.

Load Factor and Rehashing

The load factor λ of a hash table is the ratio n / size where n is the number of stored entries and size is the array length. As λ rises, the probability of collision rises, and average operation time degrades.

Rehashing keeps performance near O(1): when λ exceeds a threshold (typically 0.7 for open addressing, 1.0 for chaining) the table is replaced with one roughly double the size, and every existing key is re-inserted using the new size. Amortised cost of insertion remains O(1).

For a 4-mark "explain how a hash table works" question, hit: (1) keys hashed via a hash function, (2) resulting index gives O(1) access, (3) collisions resolved by chaining or open addressing, (4) load factor triggers rehashing when the table fills.

4.2.10 Dictionaries

A dictionary (also called an associative array, map, or symbol table) is an ADT storing unique key → value pairs with the operations put(key, value), get(key), remove(key) and contains(key). Keys must support equality testing; values can be of any type.

Typical Implementations

  • Hash table — average O(1) put / get; no natural ordering of keys. This is how Python dict, Java HashMap and C# Dictionary work internally.
  • Balanced BST (red-black or AVL) — guarantees O(log n) worst case and keeps keys in sorted order, supporting range queries. Java TreeMap, C++ std::map.
  • Sorted array + binary search — O(log n) look-up but O(n) insertion; viable when the dictionary is built once and then queried many times.
// AQA pseudocode dictionary
DECLARE ages : Dictionary
ages["Ada"]      ← 36
ages["Babbage"]  ← 79
ages["Turing"]   ← 41

OUTPUT ages["Turing"]            // 41
IF "Ada" IN ages THEN OUTPUT "known" ENDIF
REMOVE ages["Babbage"]

4.2.11 Vectors

A vector is an ADT representing a dynamic, index-addressable sequence. It supports get(i), set(i, v), length() and append(v), and can grow arbitrarily at run time. The standard implementation is a resizing array with geometric growth, giving amortised O(1) append.

Imperative (Resizing-Array) View

The vector holds a capacity cap (the array length) and a count n (number of items used). append writes to data[n] and increments n. When n = cap, the array is doubled: a new array of length 2 × cap is allocated, all items copied across, and the old array discarded.

PROCEDURE append(v, item)
    IF v.n = v.cap THEN
        newCap ← IF v.cap = 0 THEN 1 ELSE v.cap * 2 ENDIF
        newArr ← NEW ARRAY[newCap]
        FOR i ← 0 TO v.n - 1
            newArr[i] ← v.data[i]
        ENDFOR
        v.data ← newArr
        v.cap  ← newCap
    ENDIF
    v.data[v.n] ← item
    v.n ← v.n + 1
ENDPROCEDURE

Even though a single resize costs O(n), the cost is amortised across the n cheap O(1) appends that preceded it, so the average append is still O(1). Linear growth (e.g. add a fixed 10 each time) would give O(n) amortised append — much worse.

Functional (Immutable) View

In functional programming languages (Haskell, Scala, Clojure, F#) a vector is often an immutable persistent data structure. Operations such as set do not modify the original vector; they return a new vector that shares most of its internal structure with the old one. This is typically implemented as a shallow tree (often a 32-way trie) giving O(log32n) ≈ O(1) practical access while preserving immutability.

Advantages of immutable vectors: thread-safety without locks, referential transparency (an expression always evaluates to the same value), easier reasoning, free undo (keep a reference to the previous version).

AQA distinguishes vectors from plain arrays by emphasising dynamic resizing. If asked "why is append O(1) amortised rather than O(1) worst-case?" explain that most appends are cheap but the occasional resize copies all n items; the doubling strategy ensures resizes are rare enough for the average to remain constant.
