Topic 4.3 builds directly on the data structures of 4.2: algorithms are the procedures that operate on those structures. Paper 1 examiners will ask you to trace an algorithm step-by-step, compare two algorithms for the same task, state the Big-O time and space complexity, and justify which algorithm best fits a given scenario. This note works through every named algorithm in the AQA specification — graph and tree traversals, Reverse Polish, searching, sorting, Dijkstra — with full worked traces and pseudocode suitable for exam answers.

4.3.1 Graph-Traversal Algorithms

A graph traversal visits every reachable vertex starting from a given source. The two standard strategies differ only in which auxiliary data structure holds the frontier of vertices to process next.

Depth-First Search (DFS) explores as far as possible along one branch before backtracking. The frontier is held in a stack (explicit or via recursion). DFS discovers a deep path quickly but may miss shorter routes.
Breadth-First Search (BFS) explores all neighbours of a vertex before moving on to their neighbours. The frontier is held in a queue. BFS discovers vertices in non-decreasing order of number of edges from the source, giving shortest paths in unweighted graphs.

DFS Pseudocode (Recursive)

PROCEDURE DFS(graph, v, visited)
    ADD v TO visited
    OUTPUT v
    FOR EACH neighbour IN graph[v]
        IF neighbour NOT IN visited THEN
            DFS(graph, neighbour, visited)
        ENDIF
    ENDFOR
ENDPROCEDURE

DFS Pseudocode (Iterative, Explicit Stack)

PROCEDURE DFS_iterative(graph, start)
    DECLARE stack : Stack
    DECLARE visited : Set
    stack.push(start)
    WHILE NOT stack.isEmpty() DO
        v ← stack.pop()
        IF v NOT IN visited THEN
            ADD v TO visited
            OUTPUT v
            FOR EACH neighbour IN REVERSE(graph[v])
                IF neighbour NOT IN visited THEN
                    stack.push(neighbour)
                ENDIF
            ENDFOR
        ENDIF
    ENDWHILE
ENDPROCEDURE

Reversing the neighbour list before pushing is a common trick: because the stack is LIFO, pushing in reverse makes the iterative version visit vertices in the same order as the recursive version.

BFS Pseudocode

PROCEDURE BFS(graph, start)
    DECLARE queue : Queue
    DECLARE visited : Set
    queue.enQueue(start)
    ADD start TO visited
    WHILE NOT queue.isEmpty() DO
        v ← queue.deQueue()
        OUTPUT v
        FOR EACH neighbour IN graph[v]
            IF neighbour NOT IN visited THEN
                ADD neighbour TO visited
                queue.enQueue(neighbour)
            ENDIF
        ENDFOR
    ENDWHILE
ENDPROCEDURE

Examiners distinguish DFS from BFS by the data structure: stack = DFS, queue = BFS. Whenever you are asked to trace a traversal, always label which structure you are using and show its contents at each step.

Applications

  • DFS: topological sort of a DAG (schedule tasks with prerequisites), detecting cycles, finding strongly-connected components (Tarjan / Kosaraju), solving mazes, parsing language grammars.
  • BFS: shortest-path in unweighted graphs, level-order traversal of trees, finding the minimum number of moves in puzzles (15-puzzle, Rubik's), web crawlers limited by depth, network-broadcast simulation, peer-to-peer search.

4.3.2 Tree-Traversal Algorithms

For a binary tree the three depth-first orders differ in when the current node is visited relative to its left and right subtrees. All three have the same asymptotic cost — O(n) time, O(h) auxiliary stack space — but produce different orderings.

Pre-order: visit Node, then Left subtree, then Right subtree (NLR).
In-order: visit Left subtree, then Node, then Right subtree (LNR).
Post-order: visit Left subtree, then Right subtree, then Node (LRN).

Example BST

Insert 50, 30, 70, 20, 40, 60, 80:

                 50
               /    \
             30      70
            /  \    /  \
          20   40  60   80

Traversal Pseudocode

PROCEDURE preOrder(node)
    IF node ≠ NULL THEN
        OUTPUT node.key
        preOrder(node.left)
        preOrder(node.right)
    ENDIF
ENDPROCEDURE

PROCEDURE inOrder(node)
    IF node ≠ NULL THEN
        inOrder(node.left)
        OUTPUT node.key
        inOrder(node.right)
    ENDIF
ENDPROCEDURE

PROCEDURE postOrder(node)
    IF node ≠ NULL THEN
        postOrder(node.left)
        postOrder(node.right)
        OUTPUT node.key
    ENDIF
ENDPROCEDURE

Connection to Expression Trees

An expression tree stores operators at internal nodes and operands at the leaves. Take the infix expression (3 + 4) * (5 − 2):

A classic AQA question gives you a tree diagram and asks for one specific traversal. Circle each node in the correct order as you sweep through it; never try to read the answer off the tree diagram visually — always apply the recursive rule node-by-node.

4.3.3 Reverse Polish (Postfix) Notation

Reverse Polish notation (RPN), or postfix, writes each operator after its two operands: 3 4 + means 3 + 4. RPN requires no parentheses and no operator-precedence rules — the structure is implicit in the order.

Advantages over infix: (1) no parentheses required, (2) no precedence rules required at evaluation time, (3) evaluation is a simple stack loop, (4) the ordering maps directly onto post-order traversal of an expression tree. RPN is used internally by compilers and by calculators such as the HP 48.

Evaluating RPN with a Stack

FUNCTION evaluateRPN(tokens) RETURNS Real
    DECLARE s : Stack
    FOR EACH t IN tokens
        IF isNumber(t) THEN
            s.push(t)
        ELSE                         // operator
            b ← s.pop()
            a ← s.pop()
            s.push(apply(t, a, b))
        ENDIF
    ENDFOR
    RETURN s.pop()
ENDFUNCTION

Infix → Postfix: the Shunting-Yard Algorithm

Edsger Dijkstra's shunting-yard algorithm converts infix to postfix in one pass. It uses a stack of operators and an output queue.

FOR EACH token t IN infixExpression
    IF t is a number THEN
        output.enQueue(t)
    ELSE IF t is an operator o1 THEN
        WHILE opStack is not empty AND
              top(opStack) is an operator o2 AND
              (precedence(o2) > precedence(o1)
               OR (precedence(o2) = precedence(o1) AND o1 is left-assoc)) DO
            output.enQueue(opStack.pop())
        ENDWHILE
        opStack.push(o1)
    ELSE IF t is '(' THEN
        opStack.push(t)
    ELSE IF t is ')' THEN
        WHILE top(opStack) ≠ '(' DO
            output.enQueue(opStack.pop())
        ENDWHILE
        opStack.pop()          // discard the '('
    ENDIF
ENDFOR
WHILE opStack is not empty
    output.enQueue(opStack.pop())
ENDWHILE

Two pitfalls AQA loves to test in RPN conversion: (1) the minus sign in 1 − 5 gives a negative intermediate, so 8 / −4 = −2 — watch your signs; (2) operators of equal precedence are left-associative in AQA, so the earlier one pops off first. Always state your assumed precedence and associativity.

4.3.4 Searching Algorithms

Linear Search

Scan the list from the first element to the last, comparing each to the target. Works on any list — sorted or unsorted, array or linked list.

FUNCTION linearSearch(list, target) RETURNS Integer
    FOR i ← 0 TO LENGTH(list) - 1
        IF list[i] = target THEN RETURN i ENDIF
    ENDFOR
    RETURN -1                 // not found
ENDFUNCTION

Time complexity: O(n) worst and average case; O(1) best case (target at index 0). Space: O(1). Use linear search whenever the list is small, unsorted, or changes constantly so sorting is not worthwhile.

Binary Search

Binary search finds a target in a sorted list by repeatedly comparing it with the middle element and discarding the half of the list that cannot contain the target. Each iteration halves the search space, giving O(log n) time. 
FUNCTION binarySearch(list, target) RETURNS Integer
    low  ← 0
    high ← LENGTH(list) - 1
    WHILE low ≤ high DO
        mid ← (low + high) DIV 2
        IF list[mid] = target THEN
            RETURN mid
        ELSE IF list[mid] < target THEN
            low  ← mid + 1
        ELSE
            high ← mid - 1
        ENDIF
    ENDWHILE
    RETURN -1
ENDFUNCTION

Binary-Tree Search

Searching a BST follows the BST property: at each node compare the target to the key, go left if smaller, right if larger. See 4.2.8 for full pseudocode. Complexity is O(h): O(log n) for a balanced tree, O(n) if the tree is pathologically skewed.

FUNCTION search(node, target) RETURNS Boolean
    IF node = NULL THEN RETURN False ENDIF
    IF node.key = target THEN RETURN True ENDIF
    IF target < node.key THEN
        RETURN search(node.left, target)
    ELSE
        RETURN search(node.right, target)
    ENDIF
ENDFUNCTION

4.3.5 Sorting Algorithms

Bubble Sort

Compare each adjacent pair from left to right; swap if the left is bigger than the right. Each full pass floats the largest remaining element to the end (like a bubble rising). Repeat until a complete pass performs no swaps.

PROCEDURE bubbleSort(a)
    n ← LENGTH(a)
    REPEAT
        swapped ← False
        FOR i ← 0 TO n - 2
            IF a[i] > a[i + 1] THEN
                SWAP a[i], a[i + 1]
                swapped ← True
            ENDIF
        ENDFOR
        n ← n - 1           // last element is now in place
    UNTIL swapped = False
ENDPROCEDURE

Merge Sort

Merge sort is a divide-and-conquer algorithm: recursively divide the list in half until each sub-list has length 1, then merge adjacent sorted sub-lists back together. Runs in O(n log n) time with O(n) auxiliary memory in all cases, and is stable. 
FUNCTION mergeSort(a) RETURNS List
    IF LENGTH(a) ≤ 1 THEN RETURN a ENDIF
    mid   ← LENGTH(a) DIV 2
    left  ← mergeSort(a[0 .. mid - 1])
    right ← mergeSort(a[mid .. LENGTH(a) - 1])
    RETURN merge(left, right)
ENDFUNCTION

FUNCTION merge(left, right) RETURNS List
    DECLARE result : List
    i ← 0 ; j ← 0
    WHILE i < LENGTH(left) AND j < LENGTH(right) DO
        IF left[i] ≤ right[j] THEN
            result.append(left[i]) ; i ← i + 1
        ELSE
            result.append(right[j]) ; j ← j + 1
        ENDIF
    ENDWHILE
    WHILE i < LENGTH(left)  DO result.append(left[i])  ; i ← i + 1 ENDWHILE
    WHILE j < LENGTH(right) DO result.append(right[j]) ; j ← j + 1 ENDWHILE
    RETURN result
ENDFUNCTION

Why Merge Sort is Stable

A sort is stable if equal elements retain their original relative order. In the merge function the test left[i] ≤ right[j] uses ≤ (not <): when two keys are equal, the one from the left half is emitted first, preserving original order. A tiny change to < would break stability.

4.3.6 Optimisation Algorithms — Dijkstra's Shortest Path

Dijkstra's algorithm finds the shortest path from a single source vertex to every other vertex in a weighted graph with non-negative edge weights. It is a greedy algorithm: at each step it commits to the unvisited vertex with the currently smallest known distance.

Pseudocode (with a Priority Queue)

FUNCTION dijkstra(graph, source) RETURNS Dictionary
    FOR EACH v IN graph.vertices
        dist[v] ← ∞
        prev[v] ← NULL
    ENDFOR
    dist[source] ← 0
    DECLARE pq : PriorityQueue          // ordered by dist
    pq.insert(source, 0)

    WHILE NOT pq.isEmpty() DO
        u ← pq.extractMin()
        FOR EACH (v, w) IN graph.adj[u]   // edge u→v with weight w
            alt ← dist[u] + w
            IF alt < dist[v] THEN
                dist[v] ← alt
                prev[v] ← u
                pq.insertOrDecrease(v, alt)
            ENDIF
        ENDFOR
    ENDWHILE
    RETURN dist, prev
ENDFUNCTION

Why Dijkstra Fails on Negative Weights

Dijkstra's correctness relies on the invariant that once a vertex is extracted from the priority queue, its shortest distance is final — because any future path through yet-unvisited vertices can only get longer (edges add non-negative weight). A negative edge could later shorten an already-finalised distance, violating the invariant. Counter-example:

A --1--> B --1--> C
  \______ −5 ______/

Dijkstra from A finalises dist[B] = 1, then dist[C] = 2, and misses the A → B path through the negative edge (which would give dist[C] = 1 + (−5) + 1 = −3).

When edges may be negative (but there are no negative cycles), use the Bellman–Ford algorithm. It relaxes every edge V − 1 times and runs in O(VE); it also detects a negative cycle by checking if any edge can still be relaxed after V − 1 rounds.

A* — Dijkstra with a Heuristic

An admissible heuristic h(n) is a function that estimates the cost from node n to the goal and never overestimates the true remaining cost. Admissibility guarantees that A* returns an optimal path.

A* search extends Dijkstra by ordering the priority queue by f(n) = g(n) + h(n) rather than g(n) alone, where g(n) is the known cost from source to n and h(n) is the admissible heuristic estimate of the remaining cost to the goal. With h ≡ 0, A* collapses back to Dijkstra. A typical admissible heuristic for geographic graphs is straight-line (Euclidean) distance — never an over-estimate because the true path along roads is at least as long.

To trace Dijkstra in an exam: draw a table with one row per step, always extract the vertex with the smallest distance, and update distances with strict inequality (alt < dist[v]). Mark vertices as "settled" once extracted — examiners award marks for showing the settled set, not just the final distances.

Analysis and Design — Big-O Notation

Big-O notation describes an upper bound on the growth rate of a function. Formally f(n) = O(g(n)) means there exist constants c > 0 and n0 such that 0 ≤ f(n) ≤ c · g(n) for all n ≥ n0. In practice we use it to describe worst-case time or space complexity of an algorithm as a function of input size n.

Worst vs Average vs Best Case

  • Worst case — the maximum number of basic operations over all inputs of size n. This is the guarantee we can offer. Most AQA questions want worst case.
  • Average case — the expected number of operations over inputs drawn from some probability distribution (usually uniform). Harder to derive; sometimes matches the worst case (merge sort), sometimes much better (quicksort: O(n log n) average but O(n2) worst).
  • Best case — the minimum over all inputs. Rarely informative on its own but useful to report (e.g. bubble sort is O(n) best because an already-sorted array needs one clean pass).

From fastest-growing to slowest-growing (i.e. best-to-worst as n → ∞):

Find an item in an unsorted list of 100 elements.Linear searchSorting first would cost O(n log n) just to enable a single O(log n) look-up.
Find an item repeatedly in a sorted list of 106 elements.Binary searchO(log n) each = ~20 comparisons vs ~500 000 for linear.

Space Complexity

Space complexity counts auxiliary memory beyond the input. Key numbers to remember:

  • Bubble sort, linear search, iterative binary search: O(1) — in-place / no extra structure.
  • Recursive binary search, BST search, recursive tree traversal: O(h) stack space where h is depth.
  • Merge sort: O(n) auxiliary array (cannot do better with this algorithm).
  • BFS: O(V) for the queue and visited set.
  • DFS (recursive): O(V) worst case for the recursion stack.
  • Dijkstra: O(V) for dist, prev, and priority queue.
When AQA asks for "time complexity of X", quote Big-O worst case unless the question asks for best or average. Always justify briefly: "O(log n) because each iteration halves the remaining search space" earns more marks than the symbol alone.

  • Quote O(V + E) time for both traversals and state the defining data-structure difference (stack vs queue).
  • Give pre-order, in-order and post-order outputs for a given binary tree, connecting them to prefix / infix / postfix expression notation.
  • Convert an infix expression to Reverse Polish using the shunting-yard algorithm; evaluate RPN with a stack.
  • Implement and trace linear search (O(n)) and binary search (O(log n)); state binary search's preconditions (sorted, random-access).
  • Describe binary-tree search and its dependence on tree height.
  • Trace bubble sort pass-by-pass and merge sort as a recursion tree; state complexity classes and stability.
  • Trace Dijkstra's algorithm on a small weighted graph showing the priority queue and distance table at every step.
  • Explain why Dijkstra fails on negative edge weights and name Bellman–Ford as the replacement.
  • Define an admissible heuristic and describe how A* extends Dijkstra.
  • State the formal definition of Big-O and list complexity classes from O(1) to O(n!).
