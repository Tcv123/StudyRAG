Entities, Attributes, Relationships

Entity: A category of object about which data is stored. Becomes a table. (Student, Course, Order)
Attribute: A property of an entity. Becomes a column. (StudentName, Price, Email)
Relationship: An association between entities. (Student enrols on Course)
Record/Instance: A specific occurrence. Becomes a row.

4.10.2 Keys

Primary Key (PK): A field (or combination of fields) that uniquely identifies each record. Must be unique, not null, and ideally immutable.
Foreign Key (FK): A field in one table that references the primary key of another table, creating a link (relationship) between them.
Composite Key: A primary key made up of two or more fields combined (e.g., StudentID + ModuleID in Enrolment).
Candidate Key: Any field (or combination) that could serve as the primary key. The chosen one becomes the PK; the rest are alternate keys.
Secondary Key: A field indexed for efficient searching but not the primary key (e.g., Surname for searching customers).

Referential Integrity

Referential integrity ensures that every foreign key value matches an existing primary key value in the referenced table (or is NULL). This prevents "orphan records" — e.g., an Order referencing a CustomerID that does not exist in the Customer table.

-- Example: referential integrity constraint
-- If CustomerID 5 does not exist in Customer table,
-- this INSERT will be REJECTED:
INSERT INTO Orders (OrderID, CustomerID, Total)
VALUES (101, 5, 49.99);  -- ERROR: FK violation

4.10.3 Normalisation

Normalisation: The process of organising data in a database to reduce redundancy and dependency, ensuring data integrity. It involves decomposing tables into smaller, well-structured tables according to a series of normal forms.

Benefits: reduces data redundancy, prevents update/insert/delete anomalies, improves data integrity, makes the database easier to maintain.

First Normal Form (1NF)

A table is in 1NF if: (1) all fields contain atomic (indivisible) values, (2) there are no repeating groups, and (3) each row is unique (has a primary key). 
NOT in 1NF (repeating group "Modules"):
StudentID | Name  | Modules
S001      | Alice | Maths, Physics, CS

IN 1NF (each module on its own row):
StudentID | Name  | Module
S001      | Alice | Maths
S001      | Alice | Physics
S001      | Alice | CS

Second Normal Form (2NF)

A table is in 2NF if: (1) it is in 1NF, and (2) all non-key attributes depend on the entire primary key (no partial dependencies). Only relevant when the PK is composite. 
NOT in 2NF (partial dependency: Name depends only on StudentID, not full key):
StudentID | Module  | Name  | Grade
S001      | Maths   | Alice | A
S001      | Physics | Alice | B    <-- "Alice" repeated (redundant)

IN 2NF (split into two tables):
Student:    StudentID PK | Name
            S001         | Alice

Enrolment:  StudentID FK | Module | Grade
            S001         | Maths  | A
            S001         | Physics| B

Third Normal Form (3NF)

A table is in 3NF if: (1) it is in 2NF, and (2) all non-key attributes depend only on the primary key and nothing else (no transitive dependencies — non-key field depending on another non-key field). 
NOT in 3NF (transitive dependency: TutorRoom depends on TutorID, not StudentID):
StudentID PK | Name  | TutorID | TutorRoom
S001         | Alice | T01     | Room 5
S002         | Bob   | T01     | Room 5   <-- Room 5 repeated

IN 3NF (separate Tutor table):
Student: StudentID PK | Name  | TutorID FK
Tutor:   TutorID PK   | TutorRoom

Summary

Normal FormRuleRemoves
1NFAtomic values, no repeating groups, has PKRepeating groups
2NF1NF + no partial dependenciesPartial dependencies on composite key
3NF2NF + no transitive dependenciesNon-key fields depending on other non-key fields

Mnemonic: "The key, the whole key, and nothing but the key" — 1NF (depends on a key), 2NF (the whole key), 3NF (nothing but the key).

Normalisation questions often give you an unnormalised table and ask you to convert to 3NF step by step. Always: (1) identify the PK, (2) remove repeating groups (1NF), (3) remove partial dependencies (2NF), (4) remove transitive dependencies (3NF). Show the resulting tables with PKs and FKs clearly marked.

4.10.4 SQL (Structured Query Language)

SQL is the standard language for managing and querying relational databases. It has two main sublanguages:

DDL (Data Definition Language) — Defines Structure

-- CREATE a table
CREATE TABLE Student (
    StudentID   INTEGER     PRIMARY KEY,
    Name        VARCHAR(50) NOT NULL,
    Email       VARCHAR(100),
    DateOfBirth DATE,
    TutorID     INTEGER     FOREIGN KEY REFERENCES Tutor(TutorID)
);

-- ALTER a table (add a column)
ALTER TABLE Student ADD Phone VARCHAR(15);

-- DROP a table (permanently delete)
DROP TABLE Student;

DML (Data Manipulation Language) — Manipulates Data

SELECT (retrieve data)

-- Basic query
SELECT Name, Email FROM Student WHERE TutorID = 1;

-- All columns
SELECT * FROM Student;

-- With sorting
SELECT Name, Grade FROM Enrolment
ORDER BY Grade DESC;

-- With aggregation
SELECT TutorID, COUNT(*) AS NumStudents
FROM Student
GROUP BY TutorID
HAVING COUNT(*) > 5;

-- LIKE for pattern matching
SELECT * FROM Student WHERE Name LIKE 'A%';   -- starts with A
SELECT * FROM Student WHERE Email LIKE '%@gmail.com';

-- BETWEEN
SELECT * FROM Student WHERE DateOfBirth BETWEEN '2005-01-01' AND '2006-12-31';

JOIN (combine tables)

-- INNER JOIN: only matching records from both tables
SELECT Student.Name, Enrolment.Module, Enrolment.Grade
FROM Student
INNER JOIN Enrolment ON Student.StudentID = Enrolment.StudentID;

-- LEFT JOIN: all records from left table + matching from right
SELECT Student.Name, Enrolment.Module
FROM Student
LEFT JOIN Enrolment ON Student.StudentID = Enrolment.StudentID;
-- Students with no enrolments will show NULL for Module

-- Multiple joins
SELECT Student.Name, Module.Title, Enrolment.Grade
FROM Student
INNER JOIN Enrolment ON Student.StudentID = Enrolment.StudentID
INNER JOIN Module ON Enrolment.ModuleID = Module.ModuleID;

INSERT, UPDATE, DELETE

-- INSERT a new record
INSERT INTO Student (StudentID, Name, Email, TutorID)
VALUES (1, 'Alice Smith', 'alice@school.uk', 1);

-- UPDATE existing records
UPDATE Student
SET Email = 'alice.new@school.uk'
WHERE StudentID = 1;

-- DELETE records
DELETE FROM Student WHERE StudentID = 1;

-- DELETE with condition
DELETE FROM Enrolment WHERE Grade = 'U';

Aggregate Functions

FunctionPurposeExample
COUNT(*)Number of rowsSELECT COUNT(*) FROM Student
SUM(field)Total of numeric fieldSELECT SUM(Credits) FROM Module
AVG(field)Average of numeric fieldSELECT AVG(Grade) FROM Enrolment
MAX(field)Maximum valueSELECT MAX(Grade) FROM Enrolment
MIN(field)Minimum valueSELECT MIN(DateOfBirth) FROM Student
SQL is case-insensitive for keywords but case-sensitive for string comparisons in many systems. Always use WHERE before GROUP BY, and HAVING after GROUP BY. WHERE filters rows before grouping; HAVING filters groups after aggregation. For JOINs, always specify the join condition with ON.

4.10.5 Client-Server Databases

In a client-server database, the DBMS runs on a central server. Clients send SQL queries over a network; the server processes them and returns results. Multiple clients can access the database simultaneously.

Concurrent Access Issues

When multiple users access the same data simultaneously, problems can arise:

  • Lost updates — two users read the same record, modify it, and save; the second save overwrites the first
  • Uncommitted dependency — reading data that another transaction has modified but not yet committed (dirty read)
  • Inconsistent analysis — reading data while another transaction is modifying it, leading to inconsistent totals

Solutions

  • Record locking — when a user accesses a record, the DBMS locks it so no one else can modify it until the lock is released. Can cause deadlock (two transactions each waiting for the other's lock).
  • Serialisation — transactions are executed one at a time, as if serial, even though they may overlap in reality
  • Timestamp ordering — each transaction is given a timestamp; conflicts are resolved by priority (earlier timestamp wins)
  • Commitment ordering — transactions commit in an order that maintains consistency

ACID Properties

Transactions should satisfy ACID:
Atomicity: Transaction completes fully or not at all (all-or-nothing).
Consistency: Database moves from one valid state to another; constraints always satisfied.
Isolation: Concurrent transactions do not interfere with each other.
Durability: Once committed, changes are permanent (survive crashes).
For concurrent access questions, explain the specific problem (e.g., lost update), then the solution (e.g., record locking), then the trade-off (e.g., deadlock risk). Know ACID properties and be able to explain each with an example. Deadlock = two transactions each holding a lock the other needs; resolved by aborting one transaction.

4.10.6 Full Worked Normalisation Example (Unnormalised → 3NF)

A school stores exam results in a single spreadsheet. Here is a sample of the raw data:

Step 0: Unnormalised Form (UNF)

ExamRecord:
StudentID | StudentName | StudentDOB | TutorID | TutorName | SubjectCode | SubjectName | Grade | ExamDate  | ExamBoard
S001      | Alice Smith | 2007-03-12 | T01     | Mr Jones  | CS101       | Computing   | A     | 2026-05-14| AQA
S001      | Alice Smith | 2007-03-12 | T01     | Mr Jones  | MA201       | Mathematics | B     | 2026-05-20| Edexcel
S002      | Bob Lee     | 2006-11-05 | T02     | Ms Patel  | CS101       | Computing   | C     | 2026-05-14| AQA
S002      | Bob Lee     | 2006-11-05 | T02     | Ms Patel  | EN301       | English Lit | A     | 2026-05-22| AQA

Problems:
- Repeating groups (each student has multiple exam rows)
- Redundancy: StudentName, StudentDOB, TutorID, TutorName repeated for each exam
- Update anomaly: change TutorName in one row, miss others
- Delete anomaly: delete all of Bob's results and lose that he exists
- Insert anomaly: cannot add a student who has not yet sat exams

Step 1: First Normal Form (1NF)

Ensure all values are atomic, no repeating groups, and a primary key exists.

The table above already has atomic values (one value per cell).
We need a composite PK to uniquely identify each row: (StudentID, SubjectCode)

ExamRecord (StudentID, SubjectCode, StudentName, StudentDOB, TutorID,
            TutorName, SubjectName, Grade, ExamDate, ExamBoard)
PK: (StudentID, SubjectCode)

Is 1NF? YES - atomic values, no repeating groups, PK exists.
Still has partial and transitive dependencies → not yet 2NF.

Step 2: Second Normal Form (2NF)

Remove attributes that depend on only part of the composite primary key.

PK is (StudentID, SubjectCode).

Partial dependencies:
  StudentName, StudentDOB, TutorID, TutorName → depend only on StudentID (not SubjectCode)
  SubjectName, ExamBoard          → depend only on SubjectCode (not StudentID)
  Grade, ExamDate                 → depend on BOTH StudentID AND SubjectCode (keep here)

Solution: split into three tables:

Student (StudentID PK, StudentName, StudentDOB, TutorID, TutorName)
Subject (SubjectCode PK, SubjectName, ExamBoard)
Result  (StudentID FK, SubjectCode FK, Grade, ExamDate)
         -- Composite PK: (StudentID, SubjectCode)

Now in 2NF. But Student table still has transitive dependency → not yet 3NF.

Step 3: Third Normal Form (3NF)

Remove non-key attributes that depend on other non-key attributes (transitive dependencies).

In the Student table:
  TutorName depends on TutorID (not StudentID directly).
  This is a TRANSITIVE DEPENDENCY: StudentID → TutorID → TutorName

Solution: extract Tutor data to its own table:

Student (StudentID PK, StudentName, StudentDOB, TutorID FK)
Tutor   (TutorID PK, TutorName)
Subject (SubjectCode PK, SubjectName, ExamBoard)
Result  (StudentID FK, SubjectCode FK, Grade, ExamDate)
         -- PK: (StudentID, SubjectCode)

Final schema is now in 3NF:
- No repeating groups (1NF)
- No partial dependencies (2NF)
- No transitive dependencies (3NF)

Final Schema Summary

TableFieldsPKFK
StudentStudentID, StudentName, StudentDOB, TutorIDStudentIDTutorID → Tutor
TutorTutorID, TutorNameTutorID—
SubjectSubjectCode, SubjectName, ExamBoardSubjectCode—
ResultStudentID, SubjectCode, Grade, ExamDate(StudentID, SubjectCode)StudentID → Student; SubjectCode → Subject
Normalisation step-by-step is one of the most common database exam questions. Always: (1) identify the current primary key clearly, (2) name the type of dependency being removed at each step (partial or transitive), (3) show the resulting tables with PKs and FKs. Use arrows and labels. Do not skip steps — go UNF → 1NF → 2NF → 3NF.

4.10.7 SQL — Advanced Worked Examples

Using the 3NF school database: Student, Tutor, Subject, Result

Example 1: All A-grade results with student names

SELECT Student.StudentName, Subject.SubjectName, Result.Grade
FROM Result
INNER JOIN Student ON Result.StudentID = Student.StudentID
INNER JOIN Subject ON Result.SubjectCode = Subject.SubjectCode
WHERE Result.Grade = 'A'
ORDER BY Student.StudentName ASC;

Example 2: Count how many students each tutor has

SELECT Tutor.TutorName, COUNT(Student.StudentID) AS NumberOfStudents
FROM Tutor
INNER JOIN Student ON Tutor.TutorID = Student.TutorID
GROUP BY Tutor.TutorName
ORDER BY NumberOfStudents DESC;

Example 3: Find students who have NOT sat any exams (LEFT JOIN)

SELECT Student.StudentName
FROM Student
LEFT JOIN Result ON Student.StudentID = Result.StudentID
WHERE Result.StudentID IS NULL;

Example 4: Average grade per subject (requires numeric grades)

-- Assume Grade stored as integer (1=A*, 2=A, 3=B, etc.)
SELECT Subject.SubjectName, AVG(Result.GradeNum) AS AverageGrade,
       COUNT(*) AS Entries
FROM Result
INNER JOIN Subject ON Result.SubjectCode = Subject.SubjectCode
GROUP BY Subject.SubjectName
HAVING COUNT(*) >= 5
ORDER BY AverageGrade ASC;

Example 5: Insert, Update, and Delete

-- Add a new student
INSERT INTO Student (StudentID, StudentName, StudentDOB, TutorID)
VALUES ('S003', 'Clara Diaz', '2007-06-22', 'T01');

-- Record a new exam result
INSERT INTO Result (StudentID, SubjectCode, Grade, ExamDate)
VALUES ('S003', 'CS101', 'B', '2026-05-14');

-- Correct a grade entry
UPDATE Result
SET Grade = 'A'
WHERE StudentID = 'S003' AND SubjectCode = 'CS101';

-- Remove a student's results (before removing the student)
DELETE FROM Result WHERE StudentID = 'S003';
DELETE FROM Student WHERE StudentID = 'S003';

Example 6: CREATE TABLE with constraints

CREATE TABLE Result (
    StudentID   VARCHAR(10)  NOT NULL,
    SubjectCode VARCHAR(10)  NOT NULL,
    Grade       CHAR(2)      NOT NULL,
    ExamDate    DATE,
    PRIMARY KEY (StudentID, SubjectCode),
    FOREIGN KEY (StudentID)   REFERENCES Student(StudentID)
        ON DELETE CASCADE,
    FOREIGN KEY (SubjectCode) REFERENCES Subject(SubjectCode)
        ON DELETE RESTRICT
);

-- ON DELETE CASCADE: if a Student is deleted, their Results are also deleted.
-- ON DELETE RESTRICT: cannot delete a Subject that still has Results.

SQL Query Structure — Order of Clauses

ClausePurposeRequired?
SELECTSpecify columns to retrieveYes
FROMSpecify the table(s)Yes
INNER/LEFT JOIN ... ONCombine tablesIf using multiple tables
WHEREFilter rows (before grouping)No
GROUP BYGroup rows for aggregationWith aggregate functions
HAVINGFilter groups (after aggregation)With GROUP BY
ORDER BYSort results (ASC or DESC)No
LIMITRestrict number of resultsNo
In SQL questions: (1) read the schema carefully and identify table names and column names exactly, (2) use table.column notation for joins to avoid ambiguity, (3) remember WHERE filters before aggregation, HAVING filters after, (4) LEFT JOIN returns all rows from the left table including those with no match (showing NULL). Never confuse DELETE (removes rows) with DROP (removes the whole table structure).

4.10.8 Database Management Systems (DBMS)

A DBMS (Database Management System) is software that manages creation, access, updating, and administration of a database. Examples: MySQL, PostgreSQL, Microsoft SQL Server, Oracle, SQLite. It provides an interface between users/applications and the physical data storage.

Functions of a DBMS

FunctionDetail
Data storage managementOrganises physical storage; manages file structure and indexing
Data retrievalSQL query processing; query optimisation for efficiency
Data integrityEnforces constraints (PK, FK, NOT NULL); validates data types
Concurrent access controlManages simultaneous users via locking and transaction control
Security and authorisationUser accounts; permissions (SELECT, INSERT, UPDATE, DELETE, GRANT)
Backup and recoveryTransaction logs enable rollback; backup schedules for disaster recovery
Data independencePhysical storage can change without affecting application layer (logical view)

Transaction Processing

-- A transaction bundles multiple SQL statements into an atomic unit.

BEGIN TRANSACTION;
    UPDATE Account SET Balance = Balance - 500 WHERE AccountID = 'A001';
    UPDATE Account SET Balance = Balance + 500 WHERE AccountID = 'A002';
COMMIT;  -- Both changes saved permanently

-- If anything goes wrong between the UPDATEs:
ROLLBACK;  -- Both changes are undone (atomicity preserved)

-- This is essential for banking: money must not disappear or appear from nowhere.

This demonstrates Atomicity: either both transfers happen or neither does. Without transactions, a system crash between the two UPDATE statements would result in money being deducted but never credited.

Transaction questions nearly always link to ACID. For Atomicity: "all or nothing". For Durability: "committed changes survive a crash (written to non-volatile storage)". The banking transfer is the classic ACID example — learn it thoroughly. Know COMMIT (makes permanent) vs ROLLBACK (undoes).
