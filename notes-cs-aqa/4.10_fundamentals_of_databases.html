4.10.1 Conceptual Data Modelling

Flat File Databases

A flat file database stores all data in a single table (a single file, such as a CSV or spreadsheet). Every record occupies one row; every field occupies one column.

  • Simple to set up and suitable for very small datasets.
  • Prone to data redundancy — the same data is repeated in multiple rows.
  • Prone to data inconsistency — if redundant data is updated in one place but not another, the database becomes inconsistent.
  • No mechanism for enforcing relationships or integrity constraints.

Relational Databases

A relational database organises data into multiple related tables (also called relations). Each table stores data about one entity. Relationships between tables are maintained through foreign keys.

  • Reduces data redundancy by splitting data across tables.
  • Maintains data consistency through normalisation and referential integrity.
  • Supports complex queries via SQL.
  • Based on Codd's relational model (1970).

Entities, Attributes and Relationships

Entity — a category of object, person, event or concept about which data is stored (e.g., Student, Course, Order).
Attribute — a property or characteristic of an entity (e.g., StudentName, DateOfBirth).
Relationship — an association between two entities (e.g., a Student enrols on a Course).

In a relational database, each entity becomes a table, each attribute becomes a column (field), and each instance becomes a row (record).

Entity-Relationship (ER) Diagrams

ER diagrams visually represent entities (rectangles), attributes (ovals), and relationships (diamonds or lines with labels).

Degree of relationships:

TypeNotationExample
One-to-One (1:1)1 —— 1Employee — Company Car (each employee has at most one car)
One-to-Many (1:M)1 —— *Customer — Order (one customer places many orders)
Many-to-Many (M:M)* —— *Student — Module (students take many modules; modules have many students)

Resolving Many-to-Many Relationships

A relational database cannot directly implement a many-to-many relationship. Instead, a junction table (also called a link table or associative entity) is introduced. The junction table has two foreign keys, one referencing each of the original entities, and the M:M relationship is replaced by two 1:M relationships.

Student (StudentID PK, Name, Email)
Module  (ModuleID PK, Title, Credits)
Enrolment (StudentID FK, ModuleID FK, EnrolmentDate)
  -- composite primary key: (StudentID, ModuleID)
When drawing ER diagrams, always resolve M:M relationships into two 1:M relationships via a junction table. Label both sides of every relationship line.

4.10.2 Keys

Key TypeDefinition
Primary Key (PK)A unique identifier for each record in a table. Cannot be NULL. Each table has exactly one PK.
Foreign Key (FK)An attribute in one table that references the primary key of another table, creating a link between the two tables.
Composite KeyA primary key made up of two or more attributes together (e.g., StudentID + ModuleID in a junction table).
Candidate KeyAny attribute (or minimal set of attributes) that could serve as a primary key — it uniquely identifies every record. The chosen candidate key becomes the PK; the rest are alternate keys.
Referential Integrity — a constraint ensuring that every foreign key value in one table matches an existing primary key value in the referenced table (or is NULL, if allowed). This prevents orphan records — rows that reference non-existent data.

For example, if an Order table has a foreign key CustomerID referencing the Customer table, you cannot insert an order with a CustomerID that does not exist in Customer, and you cannot delete a Customer who still has orders unless you cascade the delete or set the FK to NULL.

4.10.3 Normalisation

Normalisation is the process of organising a relational database to reduce data redundancy and improve data integrity. It involves progressively decomposing tables through a series of normal forms.

Why normalise?

  • Eliminate data redundancy (repeated data).
  • Eliminate update anomalies (changing data in one place but not another).
  • Eliminate insertion anomalies (unable to add data without other unrelated data).
  • Eliminate deletion anomalies (deleting a record accidentally removes other needed data).

Unnormalised Form (UNF)

Data as it comes in, potentially with repeating groups and no primary key. May have multiple values in a single cell.

OrderID | Customer | Items
1001    | Smith    | Pen, Notebook, Ruler
1002    | Jones    | Pen

First Normal Form (1NF)

A table is in 1NF if:
  • All values are atomic (no repeating groups or multi-valued attributes).
  • Each row is uniquely identifiable (a primary key exists).

Remove repeating groups by creating separate rows:

OrderID | Customer | Item
1001    | Smith    | Pen
1001    | Smith    | Notebook
1001    | Smith    | Ruler
1002    | Jones    | Pen
-- Composite PK: (OrderID, Item)

Second Normal Form (2NF)

A table is in 2NF if:
  • It is in 1NF.
  • Every non-key attribute is fully functionally dependent on the whole primary key (no partial dependencies).

Partial dependency: an attribute depends on only part of a composite key. In the table above, Customer depends only on OrderID, not on (OrderID, Item).

Fix: decompose into separate tables.

Order (OrderID PK, Customer)
OrderItem (OrderID FK, Item)  -- composite PK: (OrderID, Item)
2NF only applies when the primary key is composite. If a table has a single-column PK, it is automatically in 2NF (assuming 1NF).

Third Normal Form (3NF)

A table is in 3NF if:
  • It is in 2NF.
  • There are no transitive dependencies — no non-key attribute depends on another non-key attribute.

Transitive dependency example:

Student (StudentID PK, Name, TutorID, TutorName)
-- TutorName depends on TutorID, not on StudentID directly.

Fix: move the transitively dependent attribute to its own table.

Student (StudentID PK, Name, TutorID FK)
Tutor   (TutorID PK, TutorName)

Full Worked Example

UNF — Library Loans:

MemberID | MemberName | Books Borrowed
M01      | Alice      | {(B1, "Databases", 2020), (B2, "Networks", 2019)}
M02      | Bob        | {(B1, "Databases", 2020)}

1NF — remove repeating group:

MemberID | MemberName | BookID | BookTitle  | Year
M01      | Alice      | B1     | Databases  | 2020
M01      | Alice      | B2     | Networks   | 2019
M02      | Bob        | B1     | Databases  | 2020
-- PK: (MemberID, BookID)

2NF — remove partial dependencies:

Member (MemberID PK, MemberName)
   -- MemberName depends only on MemberID

Book (BookID PK, BookTitle, Year)
   -- BookTitle and Year depend only on BookID

Loan (MemberID FK, BookID FK)
   -- composite PK: (MemberID, BookID)

3NF — already achieved (no transitive dependencies in any table).

4.10.4 SQL (Structured Query Language)

SQL is the standard language for querying and manipulating relational databases. It is declarative — you describe what data you want, not how to get it.

Data Definition Language (DDL)

-- Create a table
CREATE TABLE Student (
    StudentID   INTEGER     PRIMARY KEY,
    FirstName   VARCHAR(50) NOT NULL,
    LastName    VARCHAR(50) NOT NULL,
    DOB         DATE,
    Email       VARCHAR(100),
    TutorID     INTEGER,
    FOREIGN KEY (TutorID) REFERENCES Tutor(TutorID)
);

-- Modify a table structure
ALTER TABLE Student ADD Phone VARCHAR(20);
ALTER TABLE Student DROP COLUMN Phone;

-- Delete a table entirely
DROP TABLE Student;

Data Manipulation Language (DML)

INSERT — add a new record:

INSERT INTO Student (StudentID, FirstName, LastName, DOB, Email, TutorID)
VALUES (1, 'Alice', 'Smith', '2006-09-15', 'alice@school.uk', 101);

UPDATE — modify existing records:

UPDATE Student
SET Email = 'alice.smith@school.uk'
WHERE StudentID = 1;

DELETE — remove records:

DELETE FROM Student
WHERE StudentID = 1;

SELECT Queries

-- Basic select
SELECT FirstName, LastName FROM Student;

-- Select all columns
SELECT * FROM Student;

-- With condition
SELECT * FROM Student WHERE LastName = 'Smith';

-- Multiple conditions
SELECT * FROM Student
WHERE LastName = 'Smith' AND DOB > '2005-01-01';

SELECT * FROM Student
WHERE LastName = 'Smith' OR LastName = 'Jones';

SELECT * FROM Student
WHERE NOT LastName = 'Smith';

Ordering and Grouping

-- Sort results
SELECT * FROM Student ORDER BY LastName ASC;
SELECT * FROM Student ORDER BY DOB DESC;

-- Group and aggregate
SELECT TutorID, COUNT(*) AS NumStudents
FROM Student
GROUP BY TutorID;

-- Filter groups (HAVING filters after grouping; WHERE filters before)
SELECT TutorID, COUNT(*) AS NumStudents
FROM Student
GROUP BY TutorID
HAVING COUNT(*) > 5;
WHERE filters individual rows before grouping. HAVING filters groups after aggregation. A common exam question is asking you to use both in the same query.

JOIN Operations

Joins combine rows from two or more tables based on a related column.

INNER JOIN — returns only rows that have matching values in both tables:

SELECT Student.FirstName, Tutor.TutorName
FROM Student
INNER JOIN Tutor ON Student.TutorID = Tutor.TutorID;

LEFT JOIN (LEFT OUTER JOIN) — returns all rows from the left table and matching rows from the right table. Non-matching right-side columns are NULL:

SELECT Student.FirstName, Tutor.TutorName
FROM Student
LEFT JOIN Tutor ON Student.TutorID = Tutor.TutorID;
-- Includes students even if they have no tutor assigned

RIGHT JOIN (RIGHT OUTER JOIN) — returns all rows from the right table and matching rows from the left table:

SELECT Student.FirstName, Tutor.TutorName
FROM Student
RIGHT JOIN Tutor ON Student.TutorID = Tutor.TutorID;
-- Includes tutors even if they have no students assigned
Join TypeReturns
INNER JOINOnly matching rows from both tables
LEFT JOINAll rows from left table + matching from right (NULLs for no match)
RIGHT JOINAll rows from right table + matching from left (NULLs for no match)
In the exam, you may need to write multi-table queries combining JOINs, WHERE, GROUP BY, HAVING and ORDER BY. Practise writing these from scratch without a reference.

4.10.5 Transaction Processing

What is a Transaction?

A transaction is a sequence of one or more database operations (reads/writes) that are treated as a single logical unit of work. Either all operations succeed, or none of them are applied.

ACID Properties

ACID is the set of four properties that guarantee reliable transaction processing:
PropertyMeaning
AtomicityA transaction is "all or nothing." If any part fails, the entire transaction is rolled back to its previous state. No partial updates are applied.
ConsistencyA transaction moves the database from one valid state to another. All integrity constraints (e.g., referential integrity, data types) are maintained.
IsolationConcurrent transactions do not interfere with each other. The result is the same as if they had executed sequentially.
DurabilityOnce a transaction is committed, its changes are permanent and survive system failures (e.g., power loss), typically by writing to disk or a transaction log.

Record Locking

When multiple users access the same database concurrently, record locking prevents conflicts. When a transaction reads or writes a record, it acquires a lock, preventing other transactions from modifying that record until the lock is released.

  • Shared lock (read lock) — multiple transactions can read the same record simultaneously, but none can write.
  • Exclusive lock (write lock) — only one transaction can access the record (no other reads or writes).

Deadlock

Deadlock occurs when two or more transactions are each waiting for the other to release a lock, resulting in a circular dependency where none can proceed.

Example:

  • Transaction A locks Record 1 and requests Record 2.
  • Transaction B locks Record 2 and requests Record 1.
  • Neither can proceed — deadlock.

Solutions to deadlock:

  • Timeout — if a transaction waits too long for a lock, it is rolled back and restarted.
  • Deadlock detection — the DBMS periodically checks for circular dependencies and rolls back one transaction (the "victim").
  • Lock ordering — require all transactions to acquire locks in the same predefined order.
  • Serialisation — force transactions to execute one at a time (reduces concurrency).

4.10.6 Data Redundancy and Inconsistency

Data redundancy — unnecessary duplication of data across the database (e.g., a customer's address stored in every order row).

Data inconsistency — contradictory copies of the same data (e.g., a customer's address updated in one row but not in others).

Redundancy leads to:

  • Wasted storage space.
  • Increased risk of inconsistency.
  • Update anomalies, insertion anomalies and deletion anomalies.

These problems are minimised through normalisation and the use of foreign keys to reference shared data rather than duplicating it.

4.10.7 Client-Server Databases

In a client-server database architecture, the database resides on a central server, and multiple clients send requests (queries) to the server over a network. The server processes the queries and returns results.

AspectDetail
Central storageOne authoritative copy of the database on the server avoids synchronisation issues.
Concurrent accessMultiple users can access the database simultaneously; the DBMS handles locking and transactions.
SecurityAccess control is managed centrally on the server (user accounts, permissions).
BackupOnly the server needs to be backed up, simplifying disaster recovery.
ProcessingQuery processing happens on the server, reducing load on client machines. Only results are sent across the network.
Network dependencyClients cannot access the database without a network connection to the server.

Key components:

  • DBMS (Database Management System) — software on the server that manages the database (e.g., MySQL, PostgreSQL, Oracle). Handles storage, querying, security, concurrency and backup.
  • Query processor — parses and optimises SQL queries.
  • Transaction manager — ensures ACID properties are maintained.
  • Client application — the front-end interface used to submit queries (e.g., a web application, desktop app, or command-line tool).
Be ready to compare client-server databases with flat file approaches and to explain how the DBMS manages concurrent access using record locking and transactions. Questions often combine these concepts together.
