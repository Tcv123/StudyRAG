4.10.1 Conceptual Data Modelling

Flat File Databases

A flat file database stores all data in a single table (a single file, such as a CSV or spreadsheet). Every record occupies one row; every field occupies one column.

  • Simple to set up and suitable for very small datasets with few entities.
  • Prone to data redundancy — the same data is repeated in multiple rows (e.g., a customer's address appears in every order row).
  • Prone to data inconsistency — if redundant data is updated in one place but not another, different rows contradict each other.
  • No mechanism for enforcing relationships, referential integrity, or complex constraints.
  • Difficult to query complex questions spanning multiple entity types.

Relational Databases

A relational database organises data into multiple related tables (also called relations). Each table stores data about one entity. Relationships between tables are maintained through foreign keys.

  • Reduces data redundancy by splitting data across normalised tables.
  • Maintains data consistency through normalisation and referential integrity constraints.
  • Supports complex queries via SQL (Structured Query Language).
  • Based on Codd's relational model (1970) — Edgar F. Codd defined the mathematical foundations.
  • Managed by a DBMS (Database Management System) such as MySQL, PostgreSQL, Oracle, or SQL Server.
FeatureFlat FileRelational
StructureSingle tableMultiple linked tables
RedundancyHighLow (after normalisation)

Entities, Attributes and Relationships

Entity — a category of object, person, event, or concept about which data is stored. Each entity becomes a table. Examples: Student, Course, Order, Product, Employee.
Attribute — a property or characteristic of an entity. Each attribute becomes a column (field). Examples: StudentName, DateOfBirth, Price, Email.
Relationship — an association between two entities. Examples: a Student enrols on a Course; a Customer places an Order.
Instance/Record — a specific occurrence of an entity. Each instance becomes a row (record). Example: "Alice Smith, DOB 15/09/2006" is one Student record.

Entity-Relationship (ER) Diagrams

ER diagrams visually represent entities (rectangles), attributes (ovals, or listed within the rectangle), and relationships (lines connecting entities, labelled with the relationship name and cardinality).

Degree of relationships:

TypeNotationExampleImplementation
One-to-One (1:1)1 —— 1Employee — Company Car (each employee has at most one car; each car is assigned to at most one employee)FK in either table, or merge into one table
One-to-Many (1:M)1 —— *Customer — Order (one customer places many orders; each order belongs to one customer)FK in the "many" side table (Order has CustomerID FK)
Many-to-Many (M:M)* —— *Student — Module (students take many modules; modules have many students)Cannot be implemented directly — must use a junction table

Resolving Many-to-Many Relationships

A relational database cannot directly implement a many-to-many relationship. Instead, a junction table (also called a link table, bridge table, or associative entity) is introduced. The junction table has two foreign keys (one referencing each of the original entities), and the M:M relationship is replaced by two 1:M relationships.

-- Original M:M relationship: Student <--> Module
-- Resolution using a junction table:

Student (StudentID PK, Name, Email)
Module  (ModuleID PK, Title, Credits)
Enrolment (StudentID FK, ModuleID FK, EnrolmentDate, Grade)
  -- Composite primary key: (StudentID, ModuleID)

-- Now: Student 1---* Enrolment *---1 Module

The junction table can also hold attributes that belong to the relationship itself (e.g., EnrolmentDate, Grade — these are properties of the enrolment, not of the student or module individually).

When drawing ER diagrams, always resolve M:M relationships into two 1:M relationships via a junction table. Label both sides of every relationship line with the cardinality (1 or *). Name the junction table with a meaningful name that describes the relationship (e.g., "Enrolment", "OrderItem", "Booking").

4.10.2 Keys

Key TypeDefinitionExample
Primary Key (PK)A unique identifier for each record in a table. Cannot be NULL. Each table has exactly one PK. Must be unique for every row.StudentID in the Student table; OrderID in the Order table. Often an auto-incrementing integer or UUID.
Foreign Key (FK)An attribute in one table that references the primary key of another table, creating a link between them. Must match an existing PK value (or be NULL if allowed).CustomerID in the Order table referencing CustomerID PK in the Customer table.
Composite KeyA primary key made up of two or more attributes together. No single attribute is unique, but the combination is.(StudentID, ModuleID) in an Enrolment junction table — neither alone is unique, but the pair is.
Candidate KeyAny attribute (or minimal set of attributes) that could serve as a primary key — it uniquely identifies every record. There may be multiple candidate keys.In a Person table, both NationalInsuranceNo and PassportNo could uniquely identify a person.
Alternate KeyA candidate key that was not chosen as the primary key.If NationalInsuranceNo is chosen as PK, then PassportNo is an alternate key.
Referential Integrity — a constraint ensuring that every foreign key value in one table matches an existing primary key value in the referenced table (or is NULL, if allowed). This prevents orphan records — rows that reference non-existent data.

Consequences: You cannot insert an Order with a CustomerID that does not exist in the Customer table. You cannot delete a Customer who still has associated Orders (unless you cascade the delete or set the FK to NULL).

Cascade actions define what happens when a referenced PK is deleted or updated:

  • ON DELETE CASCADE — automatically delete all related records (e.g., deleting a Customer deletes all their Orders)
  • ON DELETE SET NULL — set the FK to NULL (the Order remains but has no Customer)
  • ON DELETE RESTRICT — prevent deletion if related records exist (default behaviour)

4.10.3 Normalisation

Normalisation is the systematic process of organising a relational database to reduce data redundancy and improve data integrity. It involves progressively decomposing tables through a series of normal forms.

Why Normalise?

Normalisation eliminates anomalies — problems that arise from poorly structured data:

  • Update anomaly: Changing data in one place but not in all redundant copies (e.g., a customer changes address but it is only updated in 3 of 5 order rows).
  • Insertion anomaly: Unable to add data without other unrelated data (e.g., cannot add a new module to the database until at least one student enrols on it).
  • Deletion anomaly: Deleting a record accidentally removes other needed data (e.g., deleting the last student on a module removes all information about that module).

Unnormalised Form (UNF)

Data as it comes in — potentially with repeating groups, multi-valued attributes, and no primary key.

OrderID | Customer | Items
1001    | Smith    | Pen, Notebook, Ruler
1002    | Jones    | Pen
-- Items column contains multiple values (repeating group)

First Normal Form (1NF)

A table is in 1NF if:
  • All values are atomic (each cell contains a single value — no repeating groups, no lists, no nested tables).
  • Each row is uniquely identifiable (a primary key exists).
  • Each column contains values of a single data type.

Fix: remove repeating groups by creating separate rows for each value:

OrderID | Customer | Item
1001    | Smith    | Pen
1001    | Smith    | Notebook
1001    | Smith    | Ruler
1002    | Jones    | Pen
-- Composite PK: (OrderID, Item)
-- Now every cell contains exactly one value

Second Normal Form (2NF)

A table is in 2NF if:
  • It is in 1NF.
  • Every non-key attribute is fully functionally dependent on the whole primary key (no partial dependencies).

Partial dependency: An attribute depends on only part of a composite key. In the table above, Customer depends only on OrderID (not on Item), so it is partially dependent on the composite key (OrderID, Item).

Fix: decompose into separate tables, moving partially dependent attributes to their own table:

Order     (OrderID PK, Customer)
OrderItem (OrderID FK, Item)   -- Composite PK: (OrderID, Item)

-- Customer now stored once per order, not repeated per item
2NF only applies when the primary key is composite (made of multiple columns). If a table has a single-column PK, it is automatically in 2NF (assuming it is already in 1NF), because partial dependency on part of the key is impossible when the key is only one column.

Third Normal Form (3NF)

A table is in 3NF if:
  • It is in 2NF.
  • There are no transitive dependencies — no non-key attribute depends on another non-key attribute. Every non-key attribute must depend directly on the primary key and nothing else.

Transitive dependency example:

Student (StudentID PK, Name, TutorID, TutorName, TutorEmail)
-- TutorName and TutorEmail depend on TutorID, NOT directly on StudentID.
-- StudentID → TutorID → TutorName  (transitive chain)

Fix: move the transitively dependent attributes to their own table:

Student (StudentID PK, Name, TutorID FK)
Tutor   (TutorID PK, TutorName, TutorEmail)

-- Now TutorName depends directly on TutorID (its own PK)
-- No transitive dependencies remain

Informal rule for 3NF: "Every non-key attribute must depend on the key, the whole key, and nothing but the key" (attributed to Bill Kent).

Full Worked Example

UNF — Library Loans:

MemberID | MemberName | Phone     | Books Borrowed
M01      | Alice      | 07700123  | {(B1, "Databases", "Connolly", 2020), (B2, "Networks", "Kurose", 2019)}
M02      | Bob        | 07700456  | {(B1, "Databases", "Connolly", 2020)}

1NF — remove repeating group, make all values atomic:

MemberID | MemberName | Phone    | BookID | BookTitle  | Author   | Year
M01      | Alice      | 07700123 | B1     | Databases  | Connolly | 2020
M01      | Alice      | 07700123 | B2     | Networks   | Kurose   | 2019
M02      | Bob        | 07700456 | B1     | Databases  | Connolly | 2020
-- PK: (MemberID, BookID)

2NF — remove partial dependencies (attributes depending on only part of the composite key):

Member (MemberID PK, MemberName, Phone)
   -- MemberName and Phone depend only on MemberID (partial dep. removed)

Book (BookID PK, BookTitle, Author, Year)
   -- BookTitle, Author, Year depend only on BookID (partial dep. removed)

Loan (MemberID FK, BookID FK, LoanDate)
   -- Composite PK: (MemberID, BookID)
   -- LoanDate depends on the whole key (which member borrowed which book)

3NF — check for transitive dependencies. None remain in any table (all non-key attributes depend directly on their table's PK). Already in 3NF.

4.10.4 SQL (Structured Query Language)

SQL is the standard language for querying and manipulating relational databases. It is declarative — you describe what data you want, not how to get it. The DBMS optimises the execution. SQL is divided into DDL (defining structure) and DML (manipulating data).

Data Definition Language (DDL)

-- Create a table with constraints
CREATE TABLE Student (
    StudentID   INTEGER     PRIMARY KEY,
    FirstName   VARCHAR(50) NOT NULL,
    LastName    VARCHAR(50) NOT NULL,
    DOB         DATE,
    Email       VARCHAR(100) UNIQUE,
    TutorID     INTEGER,
    FOREIGN KEY (TutorID) REFERENCES Tutor(TutorID)
);

-- Common data types:
-- INTEGER, REAL/FLOAT, VARCHAR(n), CHAR(n), DATE, BOOLEAN, TEXT

-- Modify a table structure
ALTER TABLE Student ADD Phone VARCHAR(20);
ALTER TABLE Student DROP COLUMN Phone;

-- Delete a table entirely (irreversible!)
DROP TABLE Student;

Data Manipulation Language (DML)

-- INSERT: add a new record
INSERT INTO Student (StudentID, FirstName, LastName, DOB, Email, TutorID)
VALUES (1, 'Alice', 'Smith', '2006-09-15', 'alice@school.uk', 101);

-- UPDATE: modify existing records (always use WHERE to avoid updating ALL rows!)
UPDATE Student
SET Email = 'alice.smith@school.uk'
WHERE StudentID = 1;

-- DELETE: remove records (always use WHERE!)
DELETE FROM Student
WHERE StudentID = 1;

SELECT Queries

-- Select specific columns
SELECT FirstName, LastName FROM Student;

-- Select all columns
SELECT * FROM Student;

-- With a condition (WHERE clause)
SELECT * FROM Student WHERE LastName = 'Smith';

-- Multiple conditions with AND / OR / NOT
SELECT * FROM Student
WHERE LastName = 'Smith' AND DOB > '2005-01-01';

SELECT * FROM Student
WHERE LastName = 'Smith' OR LastName = 'Jones';

SELECT * FROM Student
WHERE NOT LastName = 'Smith';

-- Wildcard matching with LIKE
SELECT * FROM Student WHERE Email LIKE '%@school.uk';  -- ends with @school.uk
SELECT * FROM Student WHERE FirstName LIKE 'A%';       -- starts with A
SELECT * FROM Student WHERE LastName LIKE '_ones';      -- 5 chars ending in 'ones'

-- IN: match against a list
SELECT * FROM Student WHERE LastName IN ('Smith', 'Jones', 'Williams');

-- BETWEEN: inclusive range
SELECT * FROM Student WHERE DOB BETWEEN '2005-01-01' AND '2006-12-31';

Ordering, Grouping and Aggregation

-- Sort results
SELECT * FROM Student ORDER BY LastName ASC;        -- A-Z (ascending)
SELECT * FROM Student ORDER BY DOB DESC;            -- newest first (descending)
SELECT * FROM Student ORDER BY LastName ASC, FirstName ASC;  -- multi-column sort

-- Aggregate functions: COUNT, SUM, AVG, MIN, MAX
SELECT COUNT(*) FROM Student;                       -- total number of students
SELECT AVG(Mark) FROM Result WHERE ModuleID = 'CS1';
SELECT MAX(Mark) AS TopMark FROM Result;            -- AS creates a column alias
SELECT MIN(DOB) AS OldestStudent FROM Student;

-- GROUP BY: group rows by a column, then aggregate within each group
SELECT TutorID, COUNT(*) AS NumStudents
FROM Student
GROUP BY TutorID;

-- HAVING: filter groups AFTER aggregation (WHERE filters BEFORE grouping)
SELECT TutorID, COUNT(*) AS NumStudents
FROM Student
GROUP BY TutorID
HAVING COUNT(*) > 5;

-- Combined example: average mark per module, only modules with avg > 60, sorted
SELECT ModuleID, AVG(Mark) AS AvgMark, COUNT(*) AS NumStudents
FROM Result
WHERE Mark IS NOT NULL
GROUP BY ModuleID
HAVING AVG(Mark) > 60
ORDER BY AvgMark DESC;
WHERE filters individual rows before grouping. HAVING filters groups after aggregation. A common exam question requires both in the same query. The execution order is: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY.

JOIN Operations

Joins combine rows from two or more tables based on a related column (typically a foreign key matching a primary key).

-- INNER JOIN: returns only rows that have matching values in BOTH tables
SELECT Student.FirstName, Student.LastName, Tutor.TutorName
FROM Student
INNER JOIN Tutor ON Student.TutorID = Tutor.TutorID;
-- Students without a tutor are EXCLUDED; tutors without students are EXCLUDED

-- LEFT JOIN: ALL rows from left table + matching rows from right (NULLs where no match)
SELECT Student.FirstName, Tutor.TutorName
FROM Student
LEFT JOIN Tutor ON Student.TutorID = Tutor.TutorID;
-- ALL students are included, even those without a tutor (TutorName = NULL)

-- RIGHT JOIN: ALL rows from right table + matching rows from left
SELECT Student.FirstName, Tutor.TutorName
FROM Student
RIGHT JOIN Tutor ON Student.TutorID = Tutor.TutorID;
-- ALL tutors included, even those with no students (FirstName = NULL)

-- Multi-table join example: student names with their module titles
SELECT s.FirstName, s.LastName, m.Title
FROM Student s
INNER JOIN Enrolment e ON s.StudentID = e.StudentID
INNER JOIN Module m ON e.ModuleID = m.ModuleID
WHERE m.Credits >= 20
ORDER BY s.LastName;
Join TypeReturnsUse When
INNER JOINOnly matching rows from both tablesYou only want records that have related data in both tables
LEFT JOINAll rows from left + matching from right (NULLs for no match)You want all records from the left table regardless of whether they have a match
RIGHT JOINAll rows from right + matching from left (NULLs for no match)You want all records from the right table regardless
In the exam, you may need to write multi-table queries combining JOINs, WHERE, GROUP BY, HAVING, and ORDER BY. Practice writing these from scratch. Common mistakes: forgetting the ON clause in a JOIN; using WHERE instead of HAVING for aggregate conditions; omitting the table name prefix when column names are ambiguous.

4.10.5 Transaction Processing

What is a Transaction?

A transaction is a sequence of one or more database operations (reads and writes) that are treated as a single logical unit of work. Either all operations succeed (commit), or none of them are applied (rollback). The classic example is a bank transfer: debit account A AND credit account B must both succeed or both fail.

-- Example transaction: transfer £100 from account A to account B
BEGIN TRANSACTION;
    UPDATE Account SET Balance = Balance - 100 WHERE AccountID = 'A';
    UPDATE Account SET Balance = Balance + 100 WHERE AccountID = 'B';
COMMIT;
-- If any statement fails, ROLLBACK undoes both updates

ACID Properties

ACID is the set of four properties that guarantee reliable transaction processing:
PropertyMeaningExample
Atomicity"All or nothing." If any part of the transaction fails, the entire transaction is rolled back to its previous state. No partial updates are applied.In a bank transfer, if the credit fails after the debit succeeds, the debit is also reversed.
ConsistencyA transaction moves the database from one valid state to another. All integrity constraints (referential integrity, data types, check constraints) are maintained throughout.After a transfer, the total balance across all accounts remains the same.
IsolationConcurrent transactions do not interfere with each other. The result is the same as if they had executed sequentially (one at a time).Two simultaneous transfers from the same account do not result in an incorrect balance.
DurabilityOnce a transaction is committed, its changes are permanent and survive system failures (power loss, crashes). Typically achieved by writing to a transaction log on disk before acknowledging the commit.If the server crashes after a committed transfer, the transfer is not lost when the server restarts.

Record Locking

When multiple users access the same database concurrently, record locking prevents conflicts by controlling access to shared data:

  • Shared lock (read lock) — multiple transactions can read the same record simultaneously, but none can write to it until all shared locks are released.
  • Exclusive lock (write lock) — only one transaction can access the record. No other reads or writes are allowed until the lock is released.

Locks are acquired automatically by the DBMS when a transaction reads or writes data, and released when the transaction commits or rolls back.

Deadlock

Deadlock occurs when two or more transactions are each waiting for the other to release a lock, creating a circular dependency where none can proceed. 
-- Deadlock scenario:
-- Transaction A locks Record 1, then requests Record 2
-- Transaction B locks Record 2, then requests Record 1
-- Both are blocked waiting for the other — deadlock!

-- Timeline:
-- T1: Transaction A: LOCK Record 1  (success)
-- T2: Transaction B: LOCK Record 2  (success)
-- T3: Transaction A: LOCK Record 2  (BLOCKED — B holds it)
-- T4: Transaction B: LOCK Record 1  (BLOCKED — A holds it)
-- Neither can proceed → DEADLOCK

Solutions to deadlock:

  • Timeout — if a transaction waits too long for a lock, it is automatically rolled back and restarted. Simple but may cause unnecessary rollbacks.
  • Deadlock detection — the DBMS periodically checks for circular dependencies in the wait graph. If found, one transaction is chosen as the "victim", rolled back, and restarted.
  • Lock ordering — require all transactions to acquire locks in the same predefined order. If both A and B always lock Record 1 before Record 2, circular waits cannot occur.
  • Timestamp ordering — transactions are ordered by timestamp; older transactions get priority.

4.10.6 Data Redundancy and Inconsistency

Data redundancy — unnecessary duplication of data across the database (e.g., a customer's address stored in every order row instead of once in a Customer table).

Data inconsistency — contradictory copies of the same data (e.g., a customer's address updated in one order row but not in others, leading to different versions of the "truth").

Redundancy leads to: wasted storage space; increased risk of inconsistency; update, insertion, and deletion anomalies. These problems are minimised through normalisation (to at least 3NF) and the use of foreign keys to reference shared data rather than duplicating it.

4.10.7 Client-Server Databases

In a client-server database architecture, the database resides on a central server, and multiple clients send SQL queries to the server over a network. The server processes the queries using its DBMS and returns only the results.

AspectDetail
Central storageOne authoritative copy of the database on the server — avoids synchronisation issues and conflicting copies.
Concurrent accessMultiple users can query and update the database simultaneously. The DBMS handles locking, transactions, and ACID compliance.
SecurityAccess control managed centrally: user accounts, role-based permissions (GRANT/REVOKE), encryption at rest and in transit.
BackupOnly the server needs to be backed up, simplifying disaster recovery. Automated backup schedules, point-in-time recovery.
ProcessingQuery processing and optimisation happen on the server (which has powerful hardware). Only results are sent across the network, minimising traffic.
Network dependencyClients cannot access the database without a network connection to the server.

Key components:

  • DBMS (Database Management System) — software on the server that manages all aspects of the database: storage engine, query processor, transaction manager, security, backup/recovery. Examples: MySQL, PostgreSQL, Oracle Database, Microsoft SQL Server, SQLite (embedded).
  • Query processor / optimiser — parses SQL queries, checks syntax and semantics, generates an optimal execution plan (choosing the best indexes, join strategies, etc.).
  • Transaction manager — ensures ACID properties are maintained for all transactions.
  • Client application — the front-end interface: a web application (PHP/Python connecting to MySQL), a desktop app, a mobile app, or a command-line SQL client.
Be ready to compare client-server databases with flat file approaches, explain how the DBMS manages concurrent access (using record locking and transactions), and describe the ACID properties. A common exam scenario: "Two users try to book the same seat on a flight simultaneously. Explain how the DBMS prevents double-booking." Answer: use exclusive locking on the seat record, wrapped in a transaction with atomicity.
