Entities, Attributes, Relationships

Entity: A category of object about which data is stored. Becomes a table. (Student, Course, Order)
Attribute: A property of an entity. Becomes a column. (StudentName, Price, Email)
Relationship: An association between entities. (Student enrols on Course)
Record/Instance: A specific occurrence. Becomes a row.

4.10.2 Keys

Primary Key (PK): A field (or combination of fields) that uniquely identifies each record. Must be unique, not null, and ideally immutable.
Foreign Key (FK): A field in one table that references the primary key of another table, creating a link (relationship) between them.
Composite Key: A primary key made up of two or more fields combined (e.g., StudentID + ModuleID in Enrolment).
Candidate Key: Any field (or combination) that could serve as the primary key. The chosen one becomes the PK; the rest are alternate keys.
Secondary Key: A field indexed for efficient searching but not the primary key (e.g., Surname for searching customers).

Referential Integrity

Referential integrity ensures that every foreign key value matches an existing primary key value in the referenced table (or is NULL). This prevents "orphan records" — e.g., an Order referencing a CustomerID that does not exist in the Customer table.

-- Example: referential integrity constraint
-- If CustomerID 5 does not exist in Customer table,
-- this INSERT will be REJECTED:
INSERT INTO Orders (OrderID, CustomerID, Total)
VALUES (101, 5, 49.99);  -- ERROR: FK violation

4.10.3 Normalisation

Normalisation: The process of organising data in a database to reduce redundancy and dependency, ensuring data integrity. It involves decomposing tables into smaller, well-structured tables according to a series of normal forms.

Benefits: reduces data redundancy, prevents update/insert/delete anomalies, improves data integrity, makes the database easier to maintain.

First Normal Form (1NF)

A table is in 1NF if: (1) all fields contain atomic (indivisible) values, (2) there are no repeating groups, and (3) each row is unique (has a primary key). 
NOT in 1NF (repeating group "Modules"):
StudentID | Name  | Modules
S001      | Alice | Maths, Physics, CS

IN 1NF (each module on its own row):
StudentID | Name  | Module
S001      | Alice | Maths
S001      | Alice | Physics
S001      | Alice | CS

Second Normal Form (2NF)

A table is in 2NF if: (1) it is in 1NF, and (2) all non-key attributes depend on the entire primary key (no partial dependencies). Only relevant when the PK is composite. 
NOT in 2NF (partial dependency: Name depends only on StudentID, not full key):
StudentID | Module  | Name  | Grade
S001      | Maths   | Alice | A
S001      | Physics | Alice | B    <-- "Alice" repeated (redundant)

IN 2NF (split into two tables):
Student:    StudentID PK | Name
            S001         | Alice

Enrolment:  StudentID FK | Module | Grade
            S001         | Maths  | A
            S001         | Physics| B

Third Normal Form (3NF)

A table is in 3NF if: (1) it is in 2NF, and (2) all non-key attributes depend only on the primary key and nothing else (no transitive dependencies — non-key field depending on another non-key field). 
NOT in 3NF (transitive dependency: TutorRoom depends on TutorID, not StudentID):
StudentID PK | Name  | TutorID | TutorRoom
S001         | Alice | T01     | Room 5
S002         | Bob   | T01     | Room 5   <-- Room 5 repeated

IN 3NF (separate Tutor table):
Student: StudentID PK | Name  | TutorID FK
Tutor:   TutorID PK   | TutorRoom

Summary

Normal FormRuleRemoves
1NFAtomic values, no repeating groups, has PKRepeating groups
2NF1NF + no partial dependenciesPartial dependencies on composite key
3NF2NF + no transitive dependenciesNon-key fields depending on other non-key fields

Mnemonic: "The key, the whole key, and nothing but the key" — 1NF (depends on a key), 2NF (the whole key), 3NF (nothing but the key).

Normalisation questions often give you an unnormalised table and ask you to convert to 3NF step by step. Always: (1) identify the PK, (2) remove repeating groups (1NF), (3) remove partial dependencies (2NF), (4) remove transitive dependencies (3NF). Show the resulting tables with PKs and FKs clearly marked.

4.10.4 SQL (Structured Query Language)

SQL is the standard language for managing and querying relational databases. It has two main sublanguages:

DDL (Data Definition Language) — Defines Structure

-- CREATE a table
CREATE TABLE Student (
    StudentID   INTEGER     PRIMARY KEY,
    Name        VARCHAR(50) NOT NULL,
    Email       VARCHAR(100),
    DateOfBirth DATE,
    TutorID     INTEGER     FOREIGN KEY REFERENCES Tutor(TutorID)
);

-- ALTER a table (add a column)
ALTER TABLE Student ADD Phone VARCHAR(15);

-- DROP a table (permanently delete)
DROP TABLE Student;

DML (Data Manipulation Language) — Manipulates Data

SELECT (retrieve data)

-- Basic query
SELECT Name, Email FROM Student WHERE TutorID = 1;

-- All columns
SELECT * FROM Student;

-- With sorting
SELECT Name, Grade FROM Enrolment
ORDER BY Grade DESC;

-- With aggregation
SELECT TutorID, COUNT(*) AS NumStudents
FROM Student
GROUP BY TutorID
HAVING COUNT(*) > 5;

-- LIKE for pattern matching
SELECT * FROM Student WHERE Name LIKE 'A%';   -- starts with A
SELECT * FROM Student WHERE Email LIKE '%@gmail.com';

-- BETWEEN
SELECT * FROM Student WHERE DateOfBirth BETWEEN '2005-01-01' AND '2006-12-31';

JOIN (combine tables)

-- INNER JOIN: only matching records from both tables
SELECT Student.Name, Enrolment.Module, Enrolment.Grade
FROM Student
INNER JOIN Enrolment ON Student.StudentID = Enrolment.StudentID;

-- LEFT JOIN: all records from left table + matching from right
SELECT Student.Name, Enrolment.Module
FROM Student
LEFT JOIN Enrolment ON Student.StudentID = Enrolment.StudentID;
-- Students with no enrolments will show NULL for Module

-- Multiple joins
SELECT Student.Name, Module.Title, Enrolment.Grade
FROM Student
INNER JOIN Enrolment ON Student.StudentID = Enrolment.StudentID
INNER JOIN Module ON Enrolment.ModuleID = Module.ModuleID;

INSERT, UPDATE, DELETE

-- INSERT a new record
INSERT INTO Student (StudentID, Name, Email, TutorID)
VALUES (1, 'Alice Smith', 'alice@school.uk', 1);

-- UPDATE existing records
UPDATE Student
SET Email = 'alice.new@school.uk'
WHERE StudentID = 1;

-- DELETE records
DELETE FROM Student WHERE StudentID = 1;

-- DELETE with condition
DELETE FROM Enrolment WHERE Grade = 'U';

Aggregate Functions

FunctionPurposeExample
COUNT(*)Number of rowsSELECT COUNT(*) FROM Student
SUM(field)Total of numeric fieldSELECT SUM(Credits) FROM Module

SQL is case-insensitive for keywords but case-sensitive for string comparisons in many systems. Always use WHERE before GROUP BY, and HAVING after GROUP BY. WHERE filters rows before grouping; HAVING filters groups after aggregation. For JOINs, always specify the join condition with ON.

4.10.5 Client-Server Databases

In a client-server database, the DBMS runs on a central server. Clients send SQL queries over a network; the server processes them and returns results. Multiple clients can access the database simultaneously.

Concurrent Access Issues

When multiple users access the same data simultaneously, problems can arise:

  • Lost updates — two users read the same record, modify it, and save; the second save overwrites the first
  • Uncommitted dependency — reading data that another transaction has modified but not yet committed (dirty read)
  • Inconsistent analysis — reading data while another transaction is modifying it, leading to inconsistent totals

Solutions

  • Record locking — when a user accesses a record, the DBMS locks it so no one else can modify it until the lock is released. Can cause deadlock (two transactions each waiting for the other's lock).
  • Serialisation — transactions are executed one at a time, as if serial, even though they may overlap in reality
  • Timestamp ordering — each transaction is given a timestamp; conflicts are resolved by priority (earlier timestamp wins)
  • Commitment ordering — transactions commit in an order that maintains consistency

ACID Properties

Transactions should satisfy ACID:
Atomicity: Transaction completes fully or not at all (all-or-nothing).
Consistency: Database moves from one valid state to another; constraints always satisfied.
Isolation: Concurrent transactions do not interfere with each other.
Durability: Once committed, changes are permanent (survive crashes).
For concurrent access questions, explain the specific problem (e.g., lost update), then the solution (e.g., record locking), then the trade-off (e.g., deadlock risk). Know ACID properties and be able to explain each with an example. Deadlock = two transactions each holding a lock the other needs; resolved by aborting one transaction.
