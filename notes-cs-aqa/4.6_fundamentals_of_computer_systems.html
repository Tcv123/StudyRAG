4.6.1 Hardware and Software

Hardware is the physical components of a computer system (CPU, RAM, hard drive, keyboard, monitor, GPU). Software is the set of instructions (programs) that tell the hardware what to do.
  • Hardware is useless without software to control it; software cannot run without hardware to execute it
  • Software provides the interface between the user and the hardware
  • The same hardware can run different software for different tasks — the key insight of the stored-program concept
  • Hardware determines the capability ceiling; software determines what the hardware actually does

Libraries

A library is a collection of pre-compiled routines that programs can reuse, saving development time and reducing errors.
  • Static libraries — code copied into executable at compile time. Larger executable, but self-contained, no external dependencies.
  • Dynamic libraries (DLLs / shared objects) — loaded at runtime, shared between programs. Saves memory and disk; updates benefit all users. Risk of version conflicts ("DLL hell").

Assembler — Two-Pass Assembly

Assembly language uses mnemonics (human-readable operation names like MOV, ADD, JMP) that map one-to-one to machine code instructions. It also supports labels (symbolic names for memory addresses, used for jump targets and variable locations).

A simple assembler cannot resolve labels in a single scan because a jump instruction may reference a label defined later in the code ("forward reference"). The solution is two-pass assembly:

  • First pass: scan the entire source; assign an address to every instruction and data item; build a symbol table mapping each label name to its address. No machine code is produced yet.
  • Second pass: translate each mnemonic to its machine code opcode; replace all label references with the actual addresses from the symbol table.
Assembly source              Symbol table (after pass 1)
LOOP:  MOV R0, #0           LOOP  → address 100
       ADD R0, R0, #1       END   → address 108
       CMP R0, #10
       BNE LOOP             Pass 2: BNE LOOP becomes BNE 100
END:   HALT

Interpreter — Additional Detail

An interpreter reads, translates, and immediately executes one statement at a time. No standalone executable is produced. Key characteristics:

  • Errors are reported at runtime when the faulty line is reached (not before)
  • Easier to debug interactively (Python's REPL: Read-Evaluate-Print Loop)
  • Source code must be distributed (unless obfuscated), reducing IP protection
  • Slower than compiled code because translation overhead occurs on every execution
  • Good for scripting, automation, rapid prototyping, and educational environments

Examples of interpreted languages: Python, Ruby, PHP (traditional), JavaScript (originally), Bash.

JIT (Just-In-Time) Compilation

JIT compilation combines portability of interpreted code with speed approaching compiled code. Source code is first compiled to portable intermediate bytecode. At runtime, a JIT compiler translates bytecode to native machine code on the fly, caching compiled sections for reuse.
  • Java: Java source → Java bytecode (.class) → JVM JIT compiler → native machine code. The same bytecode runs on any platform with a JVM installed ("write once, run anywhere").
  • C# / .NET: C# source → CIL (Common Intermediate Language) → CLR JIT → native machine code
  • Python: Python source → CPython bytecode (.pyc) → CPython interpreter (traditionally interpreted, but PyPy uses JIT)
  • JavaScript: Modern engines (V8, SpiderMonkey) JIT-compile hot code paths at runtime

JIT is slower to start than compiled code (compilation overhead at launch) but faster than pure interpretation (compiled sections run at near-native speed).

4.6.3b Linkers and Loaders

Linker

A linker takes one or more object files produced by a compiler or assembler, resolves cross-references between them, and combines them with required library code to produce a single executable file.

When a large program is split into multiple source files, each file is compiled independently into its own object file. The linker's job is to:

  1. Combine all object files into one
  2. Resolve external references — when file A calls a function defined in file B, the linker fills in the correct memory address
  3. Include required library code (either copying it in, or recording which external library to use at runtime)
  4. Produce the final executable or shared library

Loader

The loader is an OS component that takes an executable file from disk, loads it into RAM, and prepares it for execution.

The loader performs the following steps:

  1. Reads the executable from disk (the file header describes program structure)
  2. Allocates memory — creates a process address space with segments for code, data, stack, and heap
  3. Copies code and data from the file into allocated RAM
  4. Relocation — adjusts absolute addresses in the code if the program was not loaded at its expected address (modern systems use ASLR — Address Space Layout Randomisation, a security feature)
  5. Loads dynamic libraries if the program uses dynamic linking
  6. Sets up the stack and heap
  7. Transfers control to the program's entry point (e.g., main())

Library

A library is a collection of pre-written, pre-compiled routines (functions, classes, modules) that programmers can use without writing the code themselves.
  • Standard library — provided with the language/compiler (Python's math, os, sys; C's stdio.h)
  • Third-party library — written by others, installed separately (NumPy, Requests, OpenSSL)
  • Static library — file extension .lib (Windows) or .a (Linux); merged into executable at link time
  • Dynamic library — file extension .dll (Windows) or .so (Linux); loaded at runtime
Know the pipeline: compiler produces object files; linker combines object files and libraries into an executable; loader puts the executable into RAM for execution. Static linking = library code inside the executable (self-contained, larger). Dynamic linking = library loaded at runtime (smaller executable, shared, but must be present). AQA may ask why programs use libraries: saves development time, pre-tested code, reduces errors, shared code reduces memory use.

4.6.4 Role of the Operating System

Five major management functions, each operating continuously:

1. Memory Management

  • Allocation/deallocation of RAM to processes
  • Virtual memory — uses hard drive as overflow when RAM is full (swap/page file). Allows programs larger than physical RAM but much slower.
  • Paging — fixed-size blocks (pages, typically 4 KB). Page table maps virtual to physical addresses. Non-contiguous physical memory appears contiguous to each process.
  • Segmentation — variable-size segments based on logical program divisions (code, data, stack, heap).
  • Memory protection — prevents processes from accessing each other's memory.
  • Thrashing — excessive page swapping; system spends more time swapping than doing useful work.

2. Process Scheduling

Scheduling decides which process gets the CPU and for how long. In multitasking, the OS rapidly switches between processes to give the illusion of simultaneous execution.
AlgorithmDescriptionProsCons
FCFSProcesses run in arrival orderSimple, fair (FIFO)Convoy effect; poor for interactive
SJFShortest estimated time runs nextMinimises average waitRequires time estimates; starvation of long jobs
Round RobinFixed time slice (quantum), then back of queueFair, no starvation, good responseContext switch overhead; quantum size critical
PriorityHighest priority runs firstUrgent tasks prioritisedStarvation of low priority (fix: aging)
Multilevel FeedbackMultiple queues; processes move between themAdaptive, balances responsivenessComplex to implement

Key concepts:

  • Pre-emptive — OS can interrupt a running process (RR, Priority). Essential for responsive multitasking.
  • Non-pre-emptive — process runs until it finishes or yields (FCFS, SJF). A misbehaving process can monopolise the CPU.
  • Context switching — saving current process state (registers, PC, stack pointer) and loading the next. Has time overhead.
  • Starvation — low-priority process never runs because higher-priority processes are always available.

3. I/O Management

  • Device drivers — translate generic OS commands into device-specific instructions
  • Buffering — temporary data storage in RAM to compensate for speed differences between devices
  • Spooling — jobs queued on disk for slow devices (e.g., print spooler). CPU does not wait.
  • Interrupts — devices signal the CPU via control bus when needing attention, avoiding wasteful polling

4. File Management

  • File system — organises data into files and directories (FAT, NTFS, ext4, APFS)
  • Access control — permissions determine who can read, write, or execute files
  • Storage allocation — tracks which disk blocks belong to which files

5. Security Management

  • User authentication — passwords, biometrics, multi-factor
  • Access control — principle of least privilege
  • Encryption — data at rest and in transit
  • Firewall management — monitoring network traffic
  • Audit logging — recording events for security review
Structure OS role answers around these five areas: memory, scheduling, I/O, files, security. For scheduling, compare algorithms and explain which suits different scenarios (Round Robin for desktops; Priority for real-time systems; SJF minimises average waiting time).

4.6.7 Karnaugh Maps (K-Maps)

A Karnaugh map simplifies Boolean expressions visually. Truth table values are arranged in a grid where adjacent cells differ by one variable (Gray code ordering), making it easy to identify groups of 1s to combine.

K-Map Layouts

3-variable (Gray code order: 00, 01, 11, 10):

BC=00BC=01BC=11BC=10
A=0m0m1m3m2
A=1m4m5m7m6

4-variable:

CD=00CD=01CD=11CD=10
AB=00m0m1m3m2
AB=01m4m5m7m6
AB=11m12m13m15m14
AB=10m8m9m11m10

Grouping Rules

  1. Groups contain 1s only
  2. Group sizes must be powers of 2 (1, 2, 4, 8, 16)
  3. Groups must be rectangular
  4. Groups should be as large as possible
  5. Groups can wrap around edges (the map is a torus)
  6. Every 1 must be in at least one group
  7. Overlapping groups are allowed (and often necessary)
  8. Use fewest groups possible

Worked Example

Boolean expression: Q = ¬A.¬B.C + ¬A.B.C + A.¬B.C + A.B.C

3-variable K-map:
        BC=00  BC=01  BC=11  BC=10
A=0  [  0      1      1      0   ]
A=1  [  0      1      1      0   ]

Group all four 1s (2x2 block in columns 01 and 11):
This eliminates A and B (they both change within the group).
Result: Q = C

Simplified from 4 terms with 3 variables to just C!

Another Example

4-variable K-map:
        CD=00  CD=01  CD=11  CD=10
AB=00 [  1      0      0      1   ]
AB=01 [  1      0      0      1   ]
AB=11 [  0      0      0      0   ]
AB=10 [  1      0      0      1   ]

Group the six 1s:
- All cells in CD=00 column (except AB=11): wrap rows 00,01,10 = group of 4 if we
  consider corners wrapping. Actually: column CD=00 rows 00,01,10 = 3 cells (not power of 2).
  Better: pair (00,00)+(01,00) = 2, pair (00,00)+(10,00) wrap = 2.
- Column CD=10 same pattern.
- Larger group: all 1s form an L-shape; cover with two groups of 4 wrapping edges.
  Group 1: CD=00 column rows 00,01,10 + CD=10 column row 00 (wrap) = 4 cells → ¬B.¬D
  Group 2: CD=10 column rows 00,01,10 + CD=00 column row 00 (wrap) = ¬B.¬C...

  Simpler: two groups of 4:
  Corners wrap: (00,00),(01,00),(00,10),(01,10) = ¬A.¬D (4 cells)
  And: (00,00),(10,00),(00,10),(10,10) = ¬B.¬D (4 cells)
  Result: Q = ¬A.¬D + ¬B.¬D = ¬D.(¬A + ¬B)
In K-map questions, remember Gray code ordering (00, 01, 11, 10 — NOT 00, 01, 10, 11). Groups can wrap around edges. A group of 2 eliminates 1 variable; a group of 4 eliminates 2 variables; a group of 8 eliminates 3. Always make groups as large as possible to maximise simplification.

A full adder can be built from two half adders and an OR gate. Multiple full adders are chained together (the carry out of one feeds the carry in of the next) to create an n-bit ripple-carry adder.

D-type Flip-Flop

A D-type flip-flop is a sequential logic circuit that stores one bit of data. It has a data input (D) and a clock input. On the rising edge of the clock signal, the output Q takes the value of D and holds it until the next clock edge. This is the fundamental building block of registers and memory.

The D-type flip-flop stores 1 bit on the clock edge — it is the basis of all registers and memory. Be able to draw and explain the truth tables for all three circuits.

4.6.9 SR Flip-Flop

The SR (Set-Reset) flip-flop is the most basic sequential logic circuit — a 1-bit memory element built from two cross-coupled NOR gates (or NAND gates). It has two inputs: S (Set) and R (Reset), and two complementary outputs: Q and Q̄ (Q-bar / NOT Q).

SR Flip-Flop Truth Table (NOR-based)

Circuit Diagram (NOR-based SR Flip-Flop)

The cross-coupling (output of each gate feeds back as an input to the other) creates bistable behaviour — the circuit can be in two stable states and remains in one state until explicitly changed. This bistability is the fundamental mechanism of all digital memory.

The forbidden state (S=1, R=1) for an SR flip-flop causes both outputs to try to become 0 simultaneously, violating Q̄ = NOT Q. When S and R return to 0, the output is unpredictable. AQA exam questions may ask you to complete a timing diagram for an SR flip-flop, or to explain the difference between SR and D-type. Key: SR has a forbidden state; D-type eliminates this by having a single data input.

4.6.11 Virtual Machines

A virtual machine (VM) is software that emulates a complete computer system, providing a virtualised hardware environment in which another operating system and its applications can run, completely isolated from the host system.

How Virtual Machines Work

The hypervisor (also called the Virtual Machine Monitor, VMM) is the software layer that creates and manages VMs. It runs on the host machine and provides each VM with its own virtualised CPU, RAM, storage, and network interfaces. The OS running inside the VM is called the guest OS.

FeatureType 1 Hypervisor (bare-metal)Type 2 Hypervisor (hosted)
Runs onDirectly on hardware (no host OS)On top of a host operating system
PerformanceHigher (direct hardware access)Lower (host OS adds overhead)
ExamplesVMware ESXi, Microsoft Hyper-V, XenVMware Workstation, VirtualBox, Parallels
Use caseData centres, cloud serversDeveloper workstations, personal use

Uses of Virtual Machines

  • Running multiple operating systems on one physical machine — e.g., running Windows and Linux simultaneously on the same hardware. Each VM believes it has dedicated hardware.
  • Software testing — test software in different OS environments without needing separate machines. Snapshots allow the VM to be reset to a clean state after testing.
  • Sandboxing — running untrusted or potentially malicious software in an isolated environment. If the guest is compromised, the host machine is protected because the VM's access is restricted by the hypervisor.
  • Malware analysis — security researchers run malware inside a VM to observe its behaviour safely. The VM can be discarded afterwards.
  • Development and deployment — developers can create a VM matching the production server environment, eliminating "works on my machine" problems.
  • Cloud computing — cloud providers (AWS, Azure, GCP) run thousands of VMs on large physical servers. Customers rent VMs without managing the underlying hardware.
  • Legacy software — old software that only runs on Windows XP can be run in a Windows XP VM on modern hardware.

Java Virtual Machine (JVM) — A Different Kind of VM

The JVM is a process virtual machine rather than a system virtual machine. It does not emulate a whole computer — instead, it provides an abstract execution environment for Java bytecode.

  • Java source code is compiled to platform-independent bytecode (.class files)
  • The JVM interprets/JIT-compiles that bytecode on any platform that has a JVM installed
  • This achieves portability: "write once, run anywhere"
  • The JVM is not a hypervisor; it cannot run a different OS

AQA exam questions on virtual machines focus on: (1) what a VM is (software that emulates hardware, allowing a guest OS to run on a host); (2) uses — especially sandboxing (isolating untrusted code), testing (clean snapshots), and running multiple OSes; (3) advantages/disadvantages. For sandboxing specifically: the key benefit is that malicious code is confined to the VM and cannot access the host OS or other VMs. Common 4-mark question: "Explain two uses of a virtual machine" — each answer needs what it is + why it is useful.

4.6.12 BIOS and Firmware

The BIOS (Basic Input/Output System) is firmware stored in non-volatile ROM on the motherboard. It is the first software the CPU executes when the computer powers on, initialising hardware and loading the operating system.

BIOS / UEFI roles:

  1. POST (Power-On Self-Test) — tests and initialises CPU, RAM, storage, keyboard. Reports faults via beep codes.
  2. Hardware initialisation — configures memory controller, chipset, and basic I/O to a working state.
  3. Locates and loads the bootloader — reads the boot sector (MBR or UEFI boot partition) from the selected boot device into RAM and transfers control to it.
  4. Provides basic I/O services — in legacy systems, provided a software interface for disk and keyboard access during early boot. (Modern OSes bypass this entirely.)

Why BIOS is in ROM: ROM is non-volatile — its contents survive when the computer is turned off. If BIOS were in RAM, it would be lost on power-off and the computer could never start. ROM is also read-only under normal conditions, preventing accidental corruption.

Modern systems use UEFI (Unified Extensible Firmware Interface), which provides a graphical pre-boot environment, supports disks larger than 2 TB, and adds Secure Boot (verification that the bootloader has not been tampered with by malware). Both BIOS and UEFI serve the same fundamental role in the boot sequence.

If asked "why is the BIOS stored in ROM rather than RAM?": RAM is volatile (loses contents when power is off), so if BIOS were in RAM it would be gone when the computer powers on, meaning the computer could never start. ROM retains contents without power. If asked what BIOS does: initialises hardware (POST), then loads the OS bootloader from secondary storage into RAM.
