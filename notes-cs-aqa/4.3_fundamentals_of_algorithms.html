4.3.1 Graph Traversal

Graph traversal means systematically visiting every vertex in a graph exactly once. The two fundamental approaches differ in which vertices they explore first: breadth-first explores all neighbours before going deeper, while depth-first explores as far as possible along each branch before backtracking.

Breadth-First Search (BFS)

BFS: Explores all neighbours of the current vertex before moving to the next level of neighbours. It visits vertices in order of their distance (number of edges) from the starting vertex. BFS uses a queue (FIFO) to track which vertex to visit next.

Algorithm:

  1. Add the starting vertex to a queue and mark it as visited
  2. While the queue is not empty:
    • Dequeue a vertex from the front of the queue
    • Process (output) that vertex
    • For each unvisited neighbour of that vertex, mark it as visited and enqueue it
def bfs(graph, start):
    visited = set()           # track which vertices have been visited
    queue = [start]           # initialise queue with start vertex
    visited.add(start)
    result = []

    while queue:
        vertex = queue.pop(0)       # dequeue from front (FIFO)
        result.append(vertex)       # process this vertex

        for neighbour in sorted(graph[vertex]):   # alphabetical order
            if neighbour not in visited:
                visited.add(neighbour)
                queue.append(neighbour)   # enqueue at rear

    return result

Worked example:

Graph (adjacency list):
  A: [B, C, D]
  B: [A, E]
  C: [A, F]
  D: [A, E]
  E: [B, D, F]
  F: [C, E]

Applications of BFS:

  • Finding the shortest path in an unweighted graph (guaranteed because BFS explores by distance)
  • Level-order traversal of a tree
  • Finding all vertices within a connected component
  • Web crawlers — discover pages level by level from a starting URL
  • Social network analysis — finding degrees of separation between people
  • GPS navigation — finding the fewest turns to reach a destination

Depth-First Search (DFS)

DFS: Explores as far as possible along each branch before backtracking. It goes deep before going wide. DFS uses a stack (LIFO), or equivalently, recursion (which implicitly uses the call stack).

Algorithm (using explicit stack):

  1. Push the starting vertex onto a stack
  2. While the stack is not empty:
    • Pop a vertex from the top of the stack
    • If it has not been visited: mark it as visited, process it, and push all its unvisited neighbours
def dfs_stack(graph, start):
    visited = set()
    stack = [start]
    result = []

    while stack:
        vertex = stack.pop()        # pop from top (LIFO)
        if vertex not in visited:
            visited.add(vertex)
            result.append(vertex)
            # Push neighbours in reverse order for alphabetical processing
            for neighbour in sorted(graph[vertex], reverse=True):
                if neighbour not in visited:
                    stack.append(neighbour)
    return result

# Recursive version (cleaner, uses the call stack implicitly)
def dfs_recursive(graph, vertex, visited=None):
    if visited is None:
        visited = set()
    visited.add(vertex)
    print(vertex, end=" ")          # process vertex
    for neighbour in sorted(graph[vertex]):
        if neighbour not in visited:
            dfs_recursive(graph, neighbour, visited)

Call Stack Trace:

    dfs(B) → visit B, recurse into E (A already visited)
      dfs(E) → visit E, recurse into D (B already visited)
        dfs(D) → visit D, no unvisited neighbours (A,E visited) → backtrack
      back to E: recurse into F

  back to A: C,D already visited → done

Notice: goes deep along A->B->E->D before backtracking

Applications of DFS:

  • Topological sorting — ordering tasks with dependencies (e.g., course prerequisites)
  • Detecting cycles in a graph
  • Solving puzzles with backtracking — mazes, Sudoku, N-Queens
  • Finding connected components
  • Generating expression trees and evaluating them
  • Path finding (finds a path, though not necessarily the shortest)

When tracing BFS/DFS in an exam, always show the state of the queue/stack and the visited set at each step. When multiple unvisited neighbours exist, state your ordering convention (e.g., "processing alphabetically") and follow it consistently throughout. A common mistake is forgetting to mark vertices as visited when they are added to the queue (BFS) rather than when they are processed — this can lead to duplicates in the queue.

4.3.2 Tree Traversal

Tree traversals visit every node in a binary tree exactly once. There are three depth-first traversals, each differing in when the root of a subtree is processed relative to its children. All three are naturally recursive.

Example tree:
         20
        /  \
      10    30
     / \      \
    5   15    35
   /
  2
TraversalOrderResultCommon Use
Pre-orderRoot → Left → Right20, 10, 5, 2, 15, 30, 35Copying/serialising a tree; prefix expressions
In-orderLeft → Root → Right2, 5, 10, 15, 20, 30, 35BST sorted output; infix expressions
Post-orderLeft → Right → Root2, 5, 15, 10, 35, 30, 20Deleting a tree; evaluating expression trees; postfix
# The three traversals differ only in where "process" appears:
def pre_order(node):
    if node is not None:
        process(node.data)        # ROOT first
        pre_order(node.left)      # then left
        pre_order(node.right)     # then right

def in_order(node):
    if node is not None:
        in_order(node.left)       # left first
        process(node.data)        # ROOT in middle
        in_order(node.right)      # then right

def post_order(node):
    if node is not None:
        post_order(node.left)     # left first
        post_order(node.right)    # then right
        process(node.data)        # ROOT last

Key fact: In-order traversal of a binary search tree always produces values in ascending sorted order. This is because the BST property ensures all left-subtree values are smaller and all right-subtree values are larger.

To trace traversals quickly, use the "wall-hugging" method: draw a continuous line around the outside of the tree starting from the left of the root. For pre-order, note a node when the line passes its left side. For in-order, when it passes underneath. For post-order, when it passes its right side. This visual method is faster and more reliable than tracing the recursion.

4.3.3 Dijkstra's Shortest Path Algorithm

Dijkstra's algorithm: Finds the shortest path from a single source vertex to all other vertices in a weighted graph with non-negative edge weights. It is a greedy algorithm that always processes the unvisited vertex with the smallest known distance.

Algorithm:

  1. Set the distance to the source vertex as 0 and all other vertices as infinity (∞)
  2. Mark all vertices as unvisited. Set the source as the current vertex.
  3. For the current vertex, examine each unvisited neighbour:
    • Calculate: tentative distance = current vertex's distance + edge weight to neighbour
    • If this tentative distance is less than the neighbour's current recorded distance, update it
  4. Mark the current vertex as visited (it will never be reconsidered)
  5. Select the unvisited vertex with the smallest distance as the new current vertex
  6. Repeat from step 3 until all vertices are visited or the destination is reached

Worked Example

Graph:
  A --5-- B --2-- D
  |       |       |
  2       3       4
  |       |       |
  C --7-- E --1-- F

Find shortest paths from A to all vertices.

Step  Current   A    B    C    D    E    F    Visited     Reasoning
Init  -         0   ∞    ∞    ∞    ∞    ∞    {}
 1    A         0    5    2    ∞    ∞    ∞    {A}        A->B=5, A->C=2
 2    C         0    5    2    ∞    9    ∞    {A,C}      C->E=2+7=9
 3    B         0    5    2    7    8    ∞    {A,C,B}    B->D=5+2=7, B->E=5+3=8<9
 4    D         0    5    2    7    8    11   {A,C,B,D}  D->F=7+4=11
 5    E         0    5    2    7    8    9    {A,C,B,D,E} E->F=8+1=9<11
 6    F         0    5    2    7    8    9    {all}       Done

Shortest distances from A: B=5, C=2, D=7, E=8, F=9

Reconstructing paths (using "previous vertex" tracking):
  A to C: A -> C (cost 2)
  A to B: A -> B (cost 5)
  A to D: A -> B -> D (cost 7)
  A to E: A -> B -> E (cost 8)
  A to F: A -> B -> E -> F (cost 9)

Pseudocode Implementation

FUNCTION dijkstra(graph, source)
    distances ← {v: ∞ for all vertices v}
    distances[source] ← 0
    previous ← {v: None for all vertices v}
    visited ← {}

    WHILE there are unvisited vertices with distance < ∞
        current ← unvisited vertex with smallest distance
        ADD current TO visited

        FOR EACH neighbour OF current NOT IN visited
            tentative ← distances[current] + weight(current, neighbour)
            IF tentative < distances[neighbour] THEN
                distances[neighbour] ← tentative
                previous[neighbour] ← current
            ENDIF
        ENDFOR
    ENDWHILE

    RETURN distances, previous
ENDFUNCTION

Properties

  • Time complexity: O(V²) with a simple array; O((V + E) log V) with a min-heap priority queue
  • Does NOT work with negative edge weights (use Bellman-Ford algorithm instead)
  • It is a greedy algorithm — at each step it makes the locally optimal choice (smallest unvisited distance), and this guarantees the globally optimal solution for non-negative weights
  • Produces a shortest-path tree rooted at the source
Dijkstra's algorithm is used in GPS navigation (Google Maps, Waze) to find the fastest route, in network routing protocols (OSPF uses a variant), and in telecommunications to route calls through the network with minimum cost.
In the exam, show a clear table with columns for each vertex and rows for each step. At each step: (1) identify the current vertex, (2) update distances to all unvisited neighbours, (3) mark the current vertex as visited. Circle or highlight the vertex selected at each step. You may also be asked to reconstruct the actual path (not just the distance) by tracking which vertex each distance update came from.

4.3.4 Searching Algorithms

Linear Search

Linear search: Examines each element in a list sequentially from the beginning until the target is found or the end of the list is reached. Works on any list (sorted or unsorted). Also called sequential search. 
FUNCTION linearSearch(array, target)
    FOR i ← 0 TO LENGTH(array) - 1
        IF array[i] = target THEN
            RETURN i             # found at index i
        ENDIF
    ENDFOR
    RETURN -1                    # not found (sentinel value)
ENDFUNCTION

# Python implementation
def linear_search(arr, target):
    for i in range(len(arr)):
        if arr[i] == target:
            return i
    return -1

# i=2: 23 == 23, RETURN 2 (found at index 2, after 3 comparisons)

Binary Search

Binary search: Repeatedly divides a sorted list in half, comparing the target to the middle element. Each comparison eliminates half the remaining search space. Precondition: the data must be sorted. Requires random access (works on arrays, not linked lists). 
FUNCTION binarySearch(array, target)
    low ← 0
    high ← LENGTH(array) - 1

    WHILE low <= high
        mid ← (low + high) DIV 2

        IF array[mid] = target THEN
            RETURN mid               # found!
        ELSE IF array[mid] < target THEN
            low ← mid + 1          # target is in RIGHT half
        ELSE
            high ← mid - 1         # target is in LEFT half
        ENDIF
    ENDWHILE

    RETURN -1                         # not found
ENDFUNCTION

But if you need to search the same data many times, sort once then use binary search for each query. A common exam error is applying binary search to unsorted data. Also remember: binary search does NOT work on linked lists because they don't support O(1) random access.

4.3.5 Sorting Algorithms

Bubble Sort

Bubble sort: Repeatedly steps through the list, compares adjacent elements, and swaps them if they are in the wrong order. Each pass "bubbles" the largest unsorted element to its correct position at the end of the unsorted portion. 
FUNCTION bubbleSort(array)
    n ← LENGTH(array)
    FOR i ← 0 TO n - 2                    # n-1 passes needed
        swapped ← False
        FOR j ← 0 TO n - 2 - i            # -i because i elements already sorted
            IF array[j] > array[j+1] THEN
                SWAP array[j] AND array[j+1]
                swapped ← True
            ENDIF
        ENDFOR
        IF NOT swapped THEN
            BREAK                              # early exit: already sorted
        ENDIF
    ENDFOR
    RETURN array
ENDFUNCTION

Step-by-step trace: Sort [5, 3, 8, 1, 2]

Pass 1 (compare indices 0-3):
  [5,3,8,1,2] → 5>3? YES, swap → [3,5,8,1,2]
  [3,5,8,1,2] → 5>8? NO         → [3,5,8,1,2]
  [3,5,8,1,2] → 8>1? YES, swap → [3,5,1,8,2]
  [3,5,1,8,2] → 8>2? YES, swap → [3,5,1,2,8]  (8 is now in final position)

Pass 2 (compare indices 0-2):
  [3,5,1,2,8] → 3>5? NO         → [3,5,1,2,8]
  [3,5,1,2,8] → 5>1? YES, swap → [3,1,5,2,8]
  [3,1,5,2,8] → 5>2? YES, swap → [3,1,2,5,8]  (5 in final position)

Pass 3 (compare indices 0-1):

Merge Sort

Merge sort: A divide-and-conquer algorithm that (1) divides the list into two halves, (2) recursively sorts each half, then (3) merges the two sorted halves into one sorted list. The key insight: merging two already-sorted lists into one sorted list is O(n). 
FUNCTION mergeSort(array)
    IF LENGTH(array) <= 1 THEN
        RETURN array                     # base case: 0 or 1 elements already sorted
    ENDIF

    mid ← LENGTH(array) DIV 2
    left ← mergeSort(array[0..mid-1])    # recursively sort left half
    right ← mergeSort(array[mid..end])   # recursively sort right half

    RETURN merge(left, right)             # merge the two sorted halves
ENDFUNCTION

FUNCTION merge(left, right)
    result ← []
    i ← 0                               # pointer into left
    j ← 0                               # pointer into right

    WHILE i < LENGTH(left) AND j < LENGTH(right)
        IF left[i] <= right[j] THEN      # take smaller element
            APPEND left[i] TO result
            i ← i + 1
        ELSE
            APPEND right[j] TO result
            j ← j + 1
        ENDIF
    ENDWHILE

    # Append any remaining elements
    WHILE i < LENGTH(left)
        APPEND left[i] TO result
        i ← i + 1
    ENDWHILE
    WHILE j < LENGTH(right)
        APPEND right[j] TO result
        j ← j + 1
    ENDWHILE

    RETURN result
ENDFUNCTION

Quick Sort

Quick sort: A divide-and-conquer algorithm that (1) selects a pivot element, (2) partitions the list so elements less than pivot go left and elements greater go right, then (3) recursively sorts each partition. The pivot ends up in its final sorted position after partitioning. 
FUNCTION quickSort(array, low, high)
    IF low < high THEN
        pivotIndex ← partition(array, low, high)
        quickSort(array, low, pivotIndex - 1)     # sort left of pivot
        quickSort(array, pivotIndex + 1, high)    # sort right of pivot
    ENDIF
ENDFUNCTION

FUNCTION partition(array, low, high)
    pivot ← array[high]             # choose last element as pivot
    i ← low - 1                     # boundary of "less than pivot" section

    FOR j ← low TO high - 1
        IF array[j] <= pivot THEN
            i ← i + 1
            SWAP array[i] AND array[j]
        ENDIF
    ENDFOR

    SWAP array[i+1] AND array[high]   # place pivot in its final position
    RETURN i + 1                       # return pivot's index
ENDFUNCTION

Stable sort: A sorting algorithm that preserves the relative order of elements with equal keys. If two records have the same sorting key but appear in a certain order before sorting, a stable sort guarantees they remain in that order after sorting. Important when sorting by multiple criteria (e.g., sort by name, then by age — stability preserves the name ordering within each age group).
You must be able to trace each sorting algorithm step by step, showing the state of the array after each pass or recursive call. Know the best/average/worst time complexities and be ready to explain when each is most suitable: bubble sort for nearly-sorted small data; merge sort when guaranteed O(n log n) is needed (e.g., external sorting of large files); quick sort for general-purpose in-memory sorting (fastest in practice despite O(n²) worst case). A common exam mistake is forgetting that quick sort's worst case is O(n²).

4.3.6 Reverse Polish Notation (RPN)

Reverse Polish Notation (postfix notation): A way of writing mathematical expressions where operators come after their operands, eliminating the need for parentheses and operator precedence rules. Named after Polish logician Jan Lukasiewicz.

Converting Infix to RPN (Shunting Yard Algorithm)

Dijkstra's Shunting Yard algorithm uses an output queue and an operator stack:

  1. Read tokens left to right
  2. If the token is a number, add it directly to the output queue
  3. If the token is an operator:
    • While there is an operator on top of the stack with greater or equal precedence, pop it to the output
    • Push the current operator onto the stack
  4. If the token is (, push it onto the stack
  5. If the token is ), pop operators to output until ( is found, then discard (
  6. After all tokens are read, pop remaining operators to the output
Convert: 3 + 4 * 2

Token   Output Queue      Operator Stack   Action
 3      [3]               []               Number → output
 +      [3]               [+]              Push operator
 4      [3, 4]            [+]              Number → output
 *      [3, 4]            [+, *]           * has higher precedence than +, push
 2      [3, 4, 2]         [+, *]           Number → output
 END    [3, 4, 2, *, +]   []               Pop remaining: *, then +

Result: 3 4 2 * +

Evaluating RPN Using a Stack

Algorithm:

  1. Read tokens left to right
  2. If the token is a number, push it onto the stack
  3. If the token is an operator: pop the top two values, apply the operator (second-popped OP first-popped), push the result
  4. The final value remaining on the stack is the answer

Worked example: Evaluate 5 3 - 8 2 + *

Token   Action                           Stack
 5      push 5                           [5]
 3      push 3                           [5, 3]
 -      pop 3 (right), pop 5 (left)      [2]         5 - 3 = 2
 8      push 8                           [2, 8]
 2      push 2                           [2, 8, 2]
 +      pop 2 (right), pop 8 (left)      [2, 10]     8 + 2 = 10
 *      pop 10 (right), pop 2 (left)     [20]        2 * 10 = 20

Result: 20
Verify: (5 - 3) * (8 + 2) = 2 * 10 = 20  Correct!

Another example: Evaluate 10 3 2 - /

Token   Action                           Stack
 10     push 10                          [10]
 3      push 3                           [10, 3]
 2      push 2                           [10, 3, 2]
 -      pop 2, pop 3: 3 - 2 = 1         [10, 1]
 /      pop 1, pop 10: 10 / 1 = 10      [10]

Result: 10
Verify: 10 / (3 - 2) = 10 / 1 = 10  Correct!

Critical rule on operand order: When popping two values for an operator, the first popped is the right operand and the second popped is the left operand. This matters for non-commutative operations: 5 3 - means 5 - 3 = 2 (NOT 3 - 5 = -2).

Why Use RPN?

  • No need for brackets or precedence rules — unambiguous
  • Easy for computers to evaluate using a simple stack algorithm (very efficient)
  • Used in stack-based calculators (HP), programming languages (Forth, PostScript), and internally by compilers
  • Compilers convert infix expressions to postfix as an intermediate step during code generation
The most common exam mistake with RPN evaluation: getting the operand order wrong for subtraction and division. For 5 3 -, you pop 3 first (right operand) then 5 (left operand), giving 5 - 3 = 2, NOT 3 - 5. Always remember: second-popped OPERATOR first-popped.

4.3.7 Time Complexity and Big-O Notation

Big-O notation: Describes the upper bound (worst case) of an algorithm's growth rate as a function of input size n. It expresses how execution time (or space) scales as n increases, ignoring constants and lower-order terms. Big-O answers: "As the input gets arbitrarily large, how does the running time grow?"

Determining Big-O from Code

# O(1) - constant: no loops dependent on input size
def get_first(array):
    return array[0]                    # always 1 operation

# O(n) - linear: single loop through n items
def find_max(array):
    max_val = array[0]
    for item in array:                 # n iterations
        if item > max_val:
            max_val = item
    return max_val

# O(n^2) - quadratic: nested loops, each up to n
def has_duplicates(array):
    for i in range(len(array)):        # n
        for j in range(i+1, len(array)):   # up to n
            if array[i] == array[j]:
                return True
    return False

# O(log n) - logarithmic: input halved each step
def binary_search(array, target):
    low, high = 0, len(array) - 1
    while low <= high:                # log n iterations
        mid = (low + high) // 2
        if array[mid] == target:
            return mid
        elif array[mid] < target:
            low = mid + 1              # eliminate left half
        else:
            high = mid - 1             # eliminate right half
    return -1

# O(n log n) - typically recursive divide-and-conquer
# merge_sort: splits in half (log n levels) and merges n items per level

Rules for Determining Big-O

  • Drop constants: O(3n) = O(n), O(100n) = O(n), O(n/2) = O(n)
  • Drop lower-order terms: O(n² + n) = O(n²), O(n³ + n² + n) = O(n³)
  • Sequential statements: Add complexities, keep dominant. O(n) + O(n²) = O(n²)
  • Nested loops: Multiply complexities. Loop of n inside loop of n = O(n × n) = O(n²)
  • Conditional: Take the worst case of the branches
  • Recursive halving: O(log n) per recursion × O(n) work per level = O(n log n)
When asked to state the time complexity of code, identify the loops. A single loop over n items: O(n). Nested loops (loop within a loop) each up to n: O(n²). An algorithm that halves the data each step: O(log n). If asked "which is more efficient?", always compare the Big-O classes, not specific numbers. Remember: O(n log n) beats O(n²) for all sufficiently large n, even though for small n the constants may matter.

Time-Space Trade-offs

A fundamental principle in computer science: you can often make an algorithm faster by using more memory, or save memory at the cost of speed.

  • Hash tables use O(n) space but provide O(1) average lookup (vs. O(n) for linear search with O(1) space)
  • Memoisation / Dynamic Programming: Store previously computed results to avoid re-computation. Fibonacci: O(n) time + O(n) space with memoisation, vs. O(2n) time without
  • Merge sort vs Bubble sort: Merge sort uses O(n) extra space for guaranteed O(n log n) time; bubble sort uses O(1) space but is O(n²) time
  • Lookup tables: Pre-compute and store results for all inputs; O(n) space but O(1) lookup time
Caching is a universal time-space trade-off. Web browsers cache pages (use disk space to avoid re-downloading). CPUs use cache memory (expensive fast memory to avoid slow RAM access). Database indexes use disk space to speed up queries. In all cases: more memory = faster access.

4.3.9 Algorithm Comparison and Optimisation

Comparing Algorithms

When choosing between algorithms, consider:

  • Time complexity — how execution time scales with input size (the dominant factor for large inputs)
  • Space complexity — how memory usage scales (important for memory-constrained environments)
  • Best, average, and worst case — which scenario is most likely for your data?
  • Constants and overhead — Big-O hides constants; O(n) with a large constant can be slower than O(n log n) for small n. Quick sort often beats merge sort in practice despite having a worse worst case.
  • Input characteristics — Is the data already partially sorted? How large is n? Are there many duplicates?
  • Stability — does the algorithm preserve relative order of equal elements?
  • Ease of implementation — simpler algorithms have fewer bugs

Optimisation Techniques

  • Choose the right algorithm: For n < 50, a simple O(n²) sort may outperform O(n log n) due to lower overhead. For n > 10,000, algorithmic efficiency dominates.
  • Early termination: Bubble sort's "swapped" flag, binary search returning immediately when found
  • Reducing unnecessary work: Bubble sort's decreasing inner loop; using a sentinel value in linear search
  • Caching / memoisation: Store results of expensive function calls (e.g., recursive Fibonacci)
  • Using appropriate data structures: Hash table for O(1) lookup; balanced BST for O(log n) sorted access; priority queue for efficient min/max extraction
  • Divide and conquer: Breaking problems into smaller subproblems (merge sort, quick sort, binary search)

Tractable vs Intractable Problems

CategoryGrowth RateDescriptionExample
TractableO(nk) for some constant kSolvable in polynomial time; practical even for large inputsSorting, searching, shortest path
IntractableO(2n), O(n!), etc.No known polynomial-time solution; impractical for large inputsTravelling salesman (exact), satisfiability

For intractable problems, heuristic approaches find approximate (good enough) solutions in reasonable time without guaranteeing the optimal answer. Example: the nearest-neighbour heuristic for TSP runs in O(n²) and typically finds a route within 25% of optimal.

When comparing algorithms in an exam, always reference Big-O notation and apply it to the specific problem. For example: "Merge sort would be more appropriate than bubble sort for sorting 1 million student records because merge sort guarantees O(n log n) = approximately 20 million operations, while bubble sort is O(n²) = 1 trillion operations in the worst case, which would take several hours versus a few seconds."
