4.3.1 Graph Traversal

Graph traversal means visiting every vertex in a graph in a systematic way. The two fundamental approaches are breadth-first search and depth-first search.

Breadth-First Search (BFS)

BFS: Explores all neighbours of the current vertex before moving to the next level. It visits vertices in order of their distance from the starting vertex. Uses a queue.

Algorithm:

  1. Add the starting vertex to a queue and mark it as visited
  2. While the queue is not empty:
    • Dequeue a vertex from the front
    • Process (output) that vertex
    • For each unvisited neighbour, mark it as visited and enqueue it
def bfs(graph, start):
    visited = set()
    queue = [start]
    visited.add(start)
    result = []

    while queue:
        vertex = queue.pop(0)       # dequeue from front
        result.append(vertex)

        for neighbour in graph[vertex]:
            if neighbour not in visited:
                visited.add(neighbour)
                queue.append(neighbour)  # enqueue at rear

    return result

Worked example:

Graph (adjacency list):
  A: [B, C]
  B: [A, D, E]
  C: [A, F]
  D: [B]
  E: [B, F]
  F: [C, E]

BFS starting from A:

Step  Queue (front...rear)   Visited         Output
 1    [A]                    {A}
 2    [B, C]                 {A,B,C}         A
 3    [C, D, E]              {A,B,C,D,E}     B
 4    [D, E, F]              {A,B,C,D,E,F}   C
 5    [E, F]                 {A,B,C,D,E,F}   D
 6    [F]                    {A,B,C,D,E,F}   E
 7    []                     {A,B,C,D,E,F}   F

BFS order: A, B, C, D, E, F

Applications of BFS:

  • Finding the shortest path in an unweighted graph
  • Level-order traversal of a tree
  • Finding all vertices within a connected component
  • Web crawlers, social network analysis (degrees of separation)

Depth-First Search (DFS)

DFS: Explores as far as possible along each branch before backtracking. It goes deep before going wide. Uses a stack (or recursion, which implicitly uses the call stack).

Algorithm:

  1. Push the starting vertex onto a stack and mark it as visited
  2. While the stack is not empty:
    • Pop a vertex from the top
    • Process (output) that vertex
    • For each unvisited neighbour, mark it as visited and push it onto the stack
def dfs(graph, start):
    visited = set()
    stack = [start]
    result = []

    while stack:
        vertex = stack.pop()        # pop from top
        if vertex not in visited:
            visited.add(vertex)
            result.append(vertex)

            # Push neighbours (reverse for alphabetical order)
            for neighbour in reversed(graph[vertex]):
                if neighbour not in visited:
                    stack.append(neighbour)

    return result

# Recursive version
def dfs_recursive(graph, vertex, visited=None):
    if visited is None:
        visited = set()
    visited.add(vertex)
    print(vertex)
    for neighbour in graph[vertex]:
        if neighbour not in visited:
            dfs_recursive(graph, neighbour, visited)

Worked example (same graph, starting from A):

DFS starting from A (choosing alphabetically first neighbour):

Step  Stack (top at right)   Visited         Output
 1    [A]                    {}
 2    [C, B]                 {A}             A
 3    [C, E, D]              {A,B}           B
 4    [C, E]                 {A,B,D}         D
 5    [C, F]                 {A,B,D,E}       E
 6    [C]                    {A,B,D,E,F}     F
 7    []                     {A,B,C,D,E,F}   C

DFS order: A, B, D, E, F, C

Applications of DFS:

  • Topological sorting
  • Detecting cycles in a graph
  • Solving puzzles (mazes, Sudoku) with backtracking
  • Finding connected components

4.3.2 Tree Traversal

Tree traversals visit every node in a binary tree exactly once. There are three depth-first traversals, each differing in when the root of a subtree is processed relative to its children.

Example tree:
         20
        /  \
      10    30
     / \      \
    5   15    35
TraversalOrderResult for example treeCommon use
Pre-orderRoot → Left → Right20, 10, 5, 15, 30, 35Copying/serialising a tree
In-orderLeft → Root → Right5, 10, 15, 20, 30, 35BST sorted output
Post-orderLeft → Right → Root5, 15, 10, 35, 30, 20Deleting a tree, evaluating expressions
def pre_order(node):
    if node is not None:
        process(node.data)
        pre_order(node.left)
        pre_order(node.right)

def in_order(node):
    if node is not None:
        in_order(node.left)
        process(node.data)
        in_order(node.right)

def post_order(node):
    if node is not None:
        post_order(node.left)
        post_order(node.right)
        process(node.data)

Key fact: In-order traversal of a binary search tree always produces values in ascending sorted order. This is because smaller values are in the left subtree and larger values are in the right subtree.

To trace traversals quickly, draw the tree and use the "wall-hugging" method: trace a line around the outside of the tree starting from the left of the root. For pre-order, note a node when the line passes its left side. For in-order, when it passes underneath. For post-order, when it passes its right side.

4.3.3 Dijkstra's Shortest Path Algorithm

Dijkstra's algorithm: Finds the shortest path from a single source vertex to all other vertices in a weighted graph with non-negative edge weights. It is a greedy algorithm.

Algorithm:

  1. Set the distance to the source vertex as 0 and all other vertices as infinity (∞)
  2. Mark all vertices as unvisited. Set the source as the current vertex
  3. For the current vertex, examine each unvisited neighbour:
    • Calculate the tentative distance = current vertex's distance + edge weight to neighbour
    • If this tentative distance is less than the neighbour's current recorded distance, update it
  4. Mark the current vertex as visited (it will not be checked again)
  5. Select the unvisited vertex with the smallest distance as the new current vertex
  6. Repeat from step 3 until all vertices are visited or the destination is reached

Worked Example

Graph:
  A --5-- B --2-- D
  |       |       |
  2       3       4
  |       |       |
  C --7-- E --1-- F

Adjacency list (weighted):
  A: [(B,5), (C,2)]
  B: [(A,5), (D,2), (E,3)]
  C: [(A,2), (E,7)]
  D: [(B,2), (F,4)]
  E: [(B,3), (C,7), (F,1)]
  F: [(D,4), (E,1)]

Find shortest path from A to all vertices:

Step  Current   A    B    C    D    E    F    Visited
Init           0   ∞    ∞    ∞    ∞    ∞    {}
 1     A       0    5    2    ∞    ∞    ∞    {A}
 2     C       0    5    2    ∞    9    ∞    {A,C}
 3     B       0    5    2    7    8    ∞    {A,C,B}
 4     D       0    5    2    7    8    11   {A,C,B,D}
 5     E       0    5    2    7    8    9    {A,C,B,D,E}
 6     F       0    5    2    7    8    9    {A,C,B,D,E,F}

Step 1: Current = A (distance 0).
  Neighbours: B = 0+5=5 (update), C = 0+2=2 (update)

Step 2: Current = C (smallest unvisited = 2).
  Neighbours: E = 2+7=9 (update)

Step 3: Current = B (smallest unvisited = 5).
  Neighbours: D = 5+2=7 (update), E = 5+3=8 < 9 (update)

Step 4: Current = D (smallest unvisited = 7).
  Neighbours: F = 7+4=11 (update)

Step 5: Current = E (smallest unvisited = 8).
  Neighbours: F = 8+1=9 < 11 (update)

Shortest distances from A: A=0, B=5, C=2, D=7, E=8, F=9

To reconstruct the actual path, maintain a previous vertex table. For example, the shortest path from A to F is: A → B → E → F (cost 9), traced back through the previous pointers: F came from E, E came from B, B came from A.

Properties

  • Time complexity: O(V²) with a simple array, O((V + E) log V) with a priority queue (min-heap)
  • Does not work with negative edge weights (use Bellman-Ford instead)
  • It is a greedy algorithm — always picks the locally optimal choice (smallest unvisited distance)
In the exam, show a clear table with columns for each vertex and rows for each step. Update distances when a shorter path is found. Circle or highlight the vertex selected at each step. You may also be asked to state the shortest path itself, not just the distance.

4.3.4 Searching Algorithms

Linear Search

Linear search: Examines each element in a list sequentially from the beginning until the target is found or the end is reached. Works on any list (sorted or unsorted). 
FUNCTION linearSearch(array, target)
    FOR i ← 0 TO LENGTH(array) - 1
        IF array[i] = target THEN
            RETURN i             # found at index i
        ENDIF
    ENDFOR
    RETURN -1                    # not found
ENDFUNCTION
CaseTime ComplexityExplanation
Best caseO(1)Target is the first element
Average caseO(n)Target is somewhere in the middle
Worst caseO(n)Target is last element or not present

Binary Search

Binary search: Repeatedly divides a sorted list in half, comparing the target to the middle element. Each comparison eliminates half the remaining elements. Requires the data to be sorted. 
FUNCTION binarySearch(array, target)
    low ← 0
    high ← LENGTH(array) - 1

    WHILE low <= high
        mid ← (low + high) DIV 2
        IF array[mid] = target THEN
            RETURN mid
        ELSE IF array[mid] < target THEN
            low ← mid + 1       # search right half
        ELSE
            high ← mid - 1      # search left half
        ENDIF
    ENDWHILE

    RETURN -1                     # not found
ENDFUNCTION

Step-by-step example:

Array: [3, 7, 11, 15, 19, 23, 27, 31]
Target: 19

Pass 1: low=0, high=7, mid=3 → array[3]=15 < 19 → low=4
Pass 2: low=4, high=7, mid=5 → array[5]=23 > 19 → high=4
Pass 3: low=4, high=4, mid=4 → array[4]=19 = target → FOUND at index 4
CaseTime ComplexityExplanation
Best caseO(1)Target is at the midpoint
Average caseO(log n)Halving the search space each step
Worst caseO(log n)Target not present or found last

4.3.5 Sorting Algorithms

Bubble Sort

Bubble sort: Repeatedly steps through the list, compares adjacent elements, and swaps them if they are in the wrong order. Each pass "bubbles" the largest unsorted element to its correct position. 
FUNCTION bubbleSort(array)
    n ← LENGTH(array)
    FOR i ← 0 TO n - 2
        swapped ← False
        FOR j ← 0 TO n - 2 - i
            IF array[j] > array[j+1] THEN
                SWAP array[j] AND array[j+1]
                swapped ← True
            ENDIF
        ENDFOR
        IF NOT swapped THEN
            BREAK                  # already sorted
        ENDIF
    ENDFOR
    RETURN array
ENDFUNCTION

Step-by-step example: Sort [5, 3, 8, 1, 2]

Pass 1: [5,3,8,1,2] → [3,5,8,1,2] → [3,5,8,1,2] → [3,5,1,8,2] → [3,5,1,2,8]
  (8 is now in its final position)

Pass 2: [3,5,1,2,8] → [3,5,1,2,8] → [3,1,5,2,8] → [3,1,2,5,8]
  (5 is now in its final position)

Pass 3: [3,1,2,5,8] → [1,3,2,5,8] → [1,2,3,5,8]
  (3 is now in its final position)

Pass 4: [1,2,3,5,8] → no swaps needed, algorithm terminates early

Result: [1, 2, 3, 5, 8]
CaseTimeWhen
BestO(n)Already sorted (with early exit optimisation)
AverageO(n²)Random order
WorstO(n²)Reverse sorted

Space complexity: O(1) — sorts in place.

Merge Sort

Merge sort: A divide-and-conquer algorithm that repeatedly splits the list in half until each sublist has one element, then merges the sublists back together in sorted order. 
FUNCTION mergeSort(array)
    IF LENGTH(array) <= 1 THEN
        RETURN array
    ENDIF

    mid ← LENGTH(array) DIV 2
    left ← mergeSort(array[0..mid-1])
    right ← mergeSort(array[mid..end])

    RETURN merge(left, right)
ENDFUNCTION

FUNCTION merge(left, right)
    result ← []
    i ← 0
    j ← 0
    WHILE i < LENGTH(left) AND j < LENGTH(right)
        IF left[i] <= right[j] THEN
            APPEND left[i] TO result
            i ← i + 1
        ELSE
            APPEND right[j] TO result
            j ← j + 1
        ENDIF
    ENDWHILE
    APPEND remaining elements from left or right to result
    RETURN result
ENDFUNCTION

Step-by-step example: Sort [6, 3, 8, 1, 5, 2]

Split phase:
  [6, 3, 8, 1, 5, 2]
  [6, 3, 8]         [1, 5, 2]
  [6] [3, 8]        [1] [5, 2]
  [6] [3] [8]       [1] [5] [2]

Merge phase:
  [6] [3] [8]       [1] [5] [2]
  [6] [3, 8]        [1] [2, 5]
  [3, 6, 8]         [1, 2, 5]
  [1, 2, 3, 5, 6, 8]
CaseTime
BestO(n log n)
AverageO(n log n)
WorstO(n log n)

Space complexity: O(n) — requires additional space for the temporary merged arrays.

Why O(n log n)? The list is divided in half log n times (split phase). Each level of merging processes all n elements once. So total work = n × log n.

Quick Sort

Quick sort: A divide-and-conquer algorithm that selects a pivot element, partitions the list so that elements less than the pivot are on the left and elements greater are on the right, then recursively sorts each partition. 
FUNCTION quickSort(array, low, high)
    IF low < high THEN
        pivotIndex ← partition(array, low, high)
        quickSort(array, low, pivotIndex - 1)
        quickSort(array, pivotIndex + 1, high)
    ENDIF
ENDFUNCTION

FUNCTION partition(array, low, high)
    pivot ← array[high]         # choose last element as pivot
    i ← low - 1

    FOR j ← low TO high - 1
        IF array[j] <= pivot THEN
            i ← i + 1
            SWAP array[i] AND array[j]
        ENDIF
    ENDFOR
    SWAP array[i+1] AND array[high]
    RETURN i + 1                  # pivot's final position
ENDFUNCTION

Step-by-step example: Sort [7, 2, 1, 6, 8, 5, 3, 4]

Pivot = 4 (last element)
Partition: elements <= 4 go left, elements > 4 go right
  [2, 1, 3] [4] [8, 5, 7, 6]

Recursively sort left: [2, 1, 3], pivot = 3
  [2, 1] [3] []
  Recursively sort [2, 1], pivot = 1
    [] [1] [2]
  Left sorted: [1, 2, 3]

Recursively sort right: [8, 5, 7, 6], pivot = 6
  [5] [6] [8, 7]
  Recursively sort [8, 7], pivot = 7
    [] [7] [8]
  Right sorted: [5, 6, 7, 8]

Final result: [1, 2, 3, 4, 5, 6, 7, 8]
CaseTimeWhen
BestO(n log n)Pivot always divides list roughly in half
AverageO(n log n)Random data
WorstO(n²)Pivot is always the smallest or largest element (e.g., already sorted data with first/last element as pivot)

Space complexity: O(log n) on average for the recursive call stack; O(n) in the worst case.

Stable sort: A sorting algorithm that preserves the relative order of equal elements. If two elements have the same key, their original order is maintained after sorting.
You must be able to trace each sorting algorithm step by step, showing the state of the array after each pass or recursive call. Know the best/average/worst time complexities and be ready to explain when each algorithm is most suitable. Merge sort is preferred for guaranteed O(n log n); quick sort is fast in practice but can degrade to O(n²).

4.3.6 Reverse Polish Notation (RPN)

Reverse Polish Notation (postfix notation): A way of writing expressions where operators come after their operands, eliminating the need for parentheses and operator precedence rules.

Infix vs Postfix

Infix (normal)Postfix (RPN)
3 + 43 4 +
(3 + 4) * 23 4 + 2 *
3 + 4 * 23 4 2 * +
(5 - 3) * (8 + 2)5 3 - 8 2 + *
A * B + C * DA B * C D * +

Converting Infix to RPN (using an expression tree)

One method: draw the expression as a tree (operands as leaves, operators as internal nodes), then perform a post-order traversal to get the RPN.

Expression: (3 + 4) * 2

Tree:        *
           /   \
          +     2
         / \
        3   4

Post-order traversal: 3, 4, +, 2, * → RPN: 3 4 + 2 *

Converting Infix to RPN (Shunting Yard Algorithm)

Dijkstra's Shunting Yard algorithm uses an output queue and an operator stack:

  1. Read tokens left to right
  2. If the token is a number, add it to the output queue
  3. If the token is an operator:
    • While there is an operator on top of the stack with greater or equal precedence, pop it to the output queue
    • Push the current operator onto the stack
  4. If the token is (, push it onto the stack
  5. If the token is ), pop operators to the output queue until ( is found, then discard (
  6. After all tokens are read, pop remaining operators to the output queue

Evaluating RPN Using a Stack

Algorithm:

  1. Read tokens left to right
  2. If the token is a number, push it onto the stack
  3. If the token is an operator, pop the top two values, apply the operator, and push the result
  4. The final value on the stack is the answer

Worked example: Evaluate 5 3 - 8 2 + *

Token   Action                    Stack
 5      push 5                    [5]
 3      push 3                    [5, 3]
 -      pop 3,5 → 5-3=2          [2]
 8      push 8                    [2, 8]
 2      push 2                    [2, 8, 2]
 +      pop 2,8 → 8+2=10         [2, 10]
 *      pop 10,2 → 2*10=20       [20]

Result: 20

Check: (5 - 3) * (8 + 2) = 2 * 10 = 20  ✓

Important note on operand order: When popping two values for an operator, the first popped value is the right operand and the second popped is the left operand. This matters for non-commutative operations like subtraction and division.

Why Use RPN?

  • No need for brackets or precedence rules — unambiguous
  • Easy for computers to evaluate using a simple stack algorithm
  • Used in stack-based calculators (HP calculators) and some programming languages (Forth, PostScript)
  • Compilers convert infix expressions to postfix for efficient evaluation
When evaluating RPN, always pop the right operand first, then the left. For 5 3 -, you pop 3 then 5, giving 5 - 3 = 2 (not 3 - 5). This is the most common mistake in exam answers.

4.3.7 Time Complexity and Big-O Notation

Big-O notation: Describes the upper bound of an algorithm's time (or space) complexity as a function of input size n. It describes the worst-case growth rate, ignoring constants and lower-order terms.

Common Time Complexities

Big-ONameDescriptionExample
O(1)ConstantExecution time does not depend on input sizeAccessing an array element by index, push/pop on a stack
O(log n)LogarithmicHalves the problem each stepBinary search
O(n)LinearTime grows proportionally to input sizeLinear search, traversing an array
O(n log n)LinearithmicSlightly worse than linear; common for efficient sortsMerge sort, quick sort (average)

O(2n)ExponentialDoubles with each additional input elementBrute-force subset generation, naive Fibonacci

  O(log n)   = 3.3

  O(1)       = 1

Determining Big-O from Code

# O(1) - constant
def get_first(array):
    return array[0]

# O(n) - single loop through n items
def linear_sum(array):
    total = 0
    for item in array:       # n iterations
        total += item
    return total

# O(n^2) - nested loops, each iterating n times
def bubble_sort(array):
    for i in range(len(array)):           # n
        for j in range(len(array) - 1):   # n
            if array[j] > array[j+1]:
                array[j], array[j+1] = array[j+1], array[j]

# O(log n) - halving the input each step
def binary_search(array, target):
    low, high = 0, len(array) - 1
    while low <= high:                    # log n iterations
        mid = (low + high) // 2
        if array[mid] == target:
            return mid
        elif array[mid] < target:
            low = mid + 1
        else:
            high = mid - 1
    return -1

Rules for Determining Big-O

  • Drop constants: O(3n) = O(n), O(n/2) = O(n)
  • Drop lower-order terms: O(n² + n) = O(n²), O(n + log n) = O(n)
  • Sequential statements: Add complexities, keep the dominant term. O(n) + O(n²) = O(n²)
  • Nested loops: Multiply complexities. A loop of n inside a loop of n = O(n²)
  • Conditional: Take the worst case of the branches
When asked to state the time complexity of an algorithm, look at the loops. A single loop over n items is O(n). Nested loops (loop within a loop) over n items is O(n²). An algorithm that halves the data each step is O(log n). Always state the worst-case complexity unless asked otherwise.

4.3.8 Space Complexity

Space complexity: The amount of memory an algorithm requires as a function of the input size n. Includes memory for inputs, outputs, and any auxiliary (extra) storage used during execution.

When analysing space complexity, we typically focus on auxiliary space — the extra memory used beyond the input itself.

Time-Space Trade-offs

Often you can make an algorithm faster by using more memory, or save memory at the cost of speed. Examples:

  • Hash tables use O(n) space but provide O(1) average lookup (vs. O(n) for linear search with O(1) space)
  • Memoisation stores previously computed results in memory to avoid re-computation (e.g., dynamic programming for Fibonacci: O(n) space and time, vs. naive recursive O(2n) time with O(n) stack space)
  • Merge sort uses O(n) extra space for guaranteed O(n log n) time; bubble sort uses O(1) space but is O(n²) time

4.3.9 Algorithm Comparison and Optimisation

Comparing Algorithms

When comparing algorithms, consider:

  • Time complexity — how execution time scales with input size
  • Space complexity — how memory usage scales with input size
  • Best, average, and worst case performance
  • Constants and overhead — Big-O hides constants; O(n) with a large constant can be slower than O(n log n) for small n
  • Problem characteristics — is the data already partially sorted? How large is n?
  • Stability — does the algorithm preserve the relative order of equal elements?

Optimisation Techniques

  • Choose the right algorithm: For small data sets, a simple O(n²) algorithm may outperform a complex O(n log n) one due to lower overhead. For large data sets, algorithmic efficiency dominates.
  • Early termination: Bubble sort's "swapped" flag allows it to stop early if the list is already sorted, improving the best case to O(n).
  • Reducing unnecessary work: In bubble sort, the inner loop range decreases each pass because the largest elements are already in place.
  • Caching / memoisation: Store results of expensive function calls to avoid redundant computation.
  • Using appropriate data structures: A hash table gives O(1) lookup versus O(n) for an unsorted list. A balanced BST gives O(log n) for search, insert, and delete.

Tractable vs Intractable Problems

CategoryDescriptionExample
TractableCan be solved in polynomial time: O(n), O(n²), O(nk)Sorting, searching, shortest path
IntractableNo known polynomial-time solution; grows exponentially or worseTravelling salesman (brute force), knapsack problem

For intractable problems, heuristic approaches are used to find approximate solutions in reasonable time, though they do not guarantee the optimal answer.

When comparing algorithms in an exam, always reference Big-O notation and explain in the context of the problem. For example: "Merge sort would be more appropriate than bubble sort for this data set of 1 million records because merge sort is O(n log n) in all cases, whereas bubble sort is O(n²) in the average and worst case, making it impractical for large data sets."
