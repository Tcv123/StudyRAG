4.3.1 Graph Traversal

Graph traversal means visiting every vertex in a graph exactly once, starting from a given source vertex. The two fundamental strategies differ in which vertex to visit next.

Example graph (used throughout this section):
    A --- B --- E
    |  \      / |
    |   D ---   |
    |           |
    C --------- F

Adjacency list:
A: [B, C, D]   B: [A, E]   C: [A, F]
D: [A, E]      E: [B, D, F] F: [C, E]

Breadth-First Search (BFS)

BFS: Explores all vertices at the current depth level before moving to the next. It spreads outward like ripples on a pond. BFS uses a queue (FIFO) to track which vertex to visit next.

Algorithm:

  1. Enqueue the starting vertex and mark it as visited
  2. While the queue is not empty:
    • Dequeue a vertex from the front
    • Process it (e.g., output it)
    • Enqueue all its unvisited neighbours and mark them as visited
# Pseudocode
PROCEDURE bfs(graph, start)
    queue ← empty queue
    visited ← empty set
    ENQUEUE start INTO queue
    ADD start TO visited

    WHILE queue IS NOT empty
        current ← DEQUEUE from queue
        OUTPUT current

        FOR EACH neighbour OF graph[current]
            IF neighbour NOT IN visited THEN
                ADD neighbour TO visited
                ENQUEUE neighbour INTO queue
            ENDIF
        ENDFOR
    ENDWHILE
ENDPROCEDURE
# Python
from collections import deque

def bfs(graph, start):
    visited = set()
    queue = deque([start])
    visited.add(start)
    result = []

    while queue:
        vertex = queue.popleft()      # dequeue from front (FIFO)
        result.append(vertex)
        for neighbour in sorted(graph[vertex]):
            if neighbour not in visited:
                visited.add(neighbour)
                queue.append(neighbour)
    return result

Applications of BFS:

  • Finding the shortest path in an unweighted graph (guaranteed because BFS explores by distance)
  • Level-order traversal of a tree
  • Finding connected components
  • Web crawlers — discover pages level by level
  • Social network analysis — degrees of separation

Depth-First Search (DFS)

DFS: Explores as far as possible along each branch before backtracking. It goes deep before going wide. DFS uses a stack (LIFO), or equivalently, recursion (which implicitly uses the call stack). 
# Pseudocode (recursive)
PROCEDURE dfs(graph, vertex, visited)
    ADD vertex TO visited
    OUTPUT vertex

    FOR EACH neighbour OF graph[vertex]
        IF neighbour NOT IN visited THEN
            dfs(graph, neighbour, visited)
        ENDIF
    ENDFOR
ENDPROCEDURE

# Pseudocode (iterative with explicit stack)
PROCEDURE dfsIterative(graph, start)
    stack ← empty stack
    visited ← empty set
    PUSH start ONTO stack

    WHILE stack IS NOT empty
        current ← POP from stack
        IF current NOT IN visited THEN
            ADD current TO visited
            OUTPUT current
            FOR EACH neighbour OF graph[current] IN REVERSE ORDER
                IF neighbour NOT IN visited THEN
                    PUSH neighbour ONTO stack
                ENDIF
            ENDFOR
        ENDIF
    ENDWHILE
ENDPROCEDURE
# Python (recursive)
def dfs_recursive(graph, vertex, visited=None):
    if visited is None:
        visited = set()
    visited.add(vertex)
    print(vertex, end=" ")
    for neighbour in sorted(graph[vertex]):
        if neighbour not in visited:
            dfs_recursive(graph, neighbour, visited)

DFS Trace (recursive, starting from A, alphabetical order)

dfs(A) → visit A, recurse into B (first alphabetical unvisited)
  dfs(B) → visit B, recurse into E (A already visited)
    dfs(E) → visit E, recurse into D (B visited)
      dfs(D) → visit D, no unvisited neighbours → backtrack
    back to E: recurse into F
      dfs(F) → visit F, recurse into C (E visited)
        dfs(C) → visit C, no unvisited neighbours → backtrack
      back to F: done → backtrack
    back to E: done → backtrack
  back to B: done → backtrack
back to A: all visited → done

DFS order: A, B, E, D, F, C

Applications of DFS:

  • Topological sorting — ordering tasks with dependencies
  • Detecting cycles in a graph
  • Solving puzzles with backtracking — mazes, Sudoku, N-Queens
  • Finding connected components
  • Path finding (finds a path, not necessarily the shortest)

When tracing BFS/DFS, always show the state of the queue/stack and visited set at each step. State your ordering convention (e.g., "alphabetical") and follow it consistently. A common mistake with BFS is marking vertices as visited when they are processed rather than when they are added to the queue — marking on add prevents duplicates in the queue.

4.3.2 Tree Traversal

Tree traversals visit every node in a binary tree exactly once. Three depth-first traversals differ in when the root is processed relative to its children.

Example tree:
         20
        /  \
      10    30
     / \      \
    5   15    35
   /
  2

Pre-orderRoot → Left → Right20, 10, 5, 2, 15, 30, 35Copying/serialising a tree; prefix expressions
In-orderLeft → Root → Right2, 5, 10, 15, 20, 30, 35BST sorted output; infix expressions
Post-orderLeft → Right → Root2, 5, 15, 10, 35, 30, 20Deleting tree; postfix expressions
# Pseudocode
PROCEDURE preOrder(node)
    IF node <> null THEN
        OUTPUT node.data         # ROOT first
        preOrder(node.left)
        preOrder(node.right)
    ENDIF
ENDPROCEDURE

PROCEDURE inOrder(node)
    IF node <> null THEN
        inOrder(node.left)       # left first
        OUTPUT node.data         # ROOT in middle
        inOrder(node.right)
    ENDIF
ENDPROCEDURE

PROCEDURE postOrder(node)
    IF node <> null THEN
        postOrder(node.left)
        postOrder(node.right)    # right second
        OUTPUT node.data         # ROOT last
    ENDIF
ENDPROCEDURE
# Python
def pre_order(node):
    if node is not None:
        print(node.data, end=" ")
        pre_order(node.left)
        pre_order(node.right)

def in_order(node):
    if node is not None:
        in_order(node.left)
        print(node.data, end=" ")
        in_order(node.right)

def post_order(node):
    if node is not None:
        post_order(node.left)
        post_order(node.right)
        print(node.data, end=" ")

Key fact: In-order traversal of a binary search tree always produces values in ascending sorted order.

To trace traversals quickly, use the "wall-hugging" method: draw a continuous line around the outside of the tree. For pre-order, note a node when the line passes its left side. For in-order, when passing underneath. For post-order, when passing its right side. This visual method is faster and more reliable than tracing recursion.

4.3.3 Dijkstra's Shortest Path Algorithm

Dijkstra's algorithm: Finds the shortest path from a single source vertex to all other vertices in a weighted graph with non-negative edge weights. It is a greedy algorithm — always processing the unvisited vertex with the smallest known distance.

Algorithm:

  1. Set the distance to the source as 0 and all others as ∞
  2. Mark all vertices as unvisited. Set source as current vertex.
  3. For the current vertex, examine each unvisited neighbour:
    • Calculate: tentative = current distance + edge weight to neighbour
    • If tentative < neighbour's current distance, update it
  4. Mark the current vertex as visited
  5. Select the unvisited vertex with the smallest distance as new current
  6. Repeat from step 3 until all visited or destination reached

Worked Example

Graph:
  A --5-- B --2-- D
  |       |       |
  2       3       4
  |       |       |
  C --7-- E --1-- F

Find shortest paths from A to all vertices.

Reasoning at each step:
Step 1: Process A. Update B=0+5=5, C=0+2=2. Smallest unvisited: C(2)
Step 2: Process C. Update E=2+7=9. Smallest unvisited: B(5)

Step 6: Process F. Done.

Shortest paths from A:
  A→C: A-C (cost 2)
  A→B: A-B (cost 5)
  A→D: A-B-D (cost 7)
  A→E: A-B-E (cost 8)
  A→F: A-B-E-F (cost 9)

Pseudocode

FUNCTION dijkstra(graph, source)
    distances ← {v: ∞ for all vertices v}
    distances[source] ← 0
    previous ← {v: None for all vertices v}
    visited ← {}

    WHILE there are unvisited vertices with distance < ∞
        current ← unvisited vertex with smallest distance
        ADD current TO visited

        FOR EACH neighbour OF current NOT IN visited
            tentative ← distances[current] + weight(current, neighbour)
            IF tentative < distances[neighbour] THEN
                distances[neighbour] ← tentative
                previous[neighbour] ← current
            ENDIF
        ENDFOR
    ENDWHILE

    RETURN distances, previous
ENDFUNCTION
# Python implementation
import heapq

def dijkstra(graph, source):
    distances = {v: float('inf') for v in graph}
    distances[source] = 0
    previous = {v: None for v in graph}
    pq = [(0, source)]    # (distance, vertex) min-heap

    while pq:
        dist, current = heapq.heappop(pq)
        if dist > distances[current]:
            continue       # skip if we already found a shorter path

        for neighbour, weight in graph[current]:
            tentative = dist + weight
            if tentative < distances[neighbour]:
                distances[neighbour] = tentative
                previous[neighbour] = current
                heapq.heappush(pq, (tentative, neighbour))

    return distances, previous

Properties

  • Time complexity: O(V2) with simple array; O((V + E) log V) with min-heap
  • Does NOT work with negative edge weights
  • It is greedy — locally optimal choice at each step yields globally optimal result
  • Produces a shortest-path tree rooted at the source
In the exam, show a clear table with columns for each vertex and rows for each step. At each step: (1) identify the current vertex (smallest unvisited distance), (2) update distances to unvisited neighbours, (3) mark current as visited. You may be asked to reconstruct the actual path by tracking the "previous" vertex. The most common mistake is processing a vertex that has already been visited.

4.3.4 Searching Algorithms

Linear Search

Linear search: Examines each element sequentially from the beginning until the target is found or the end is reached. Works on any list (sorted or unsorted). Also called sequential search. 
# Pseudocode
FUNCTION linearSearch(array, target)
    FOR i ← 0 TO LENGTH(array) - 1
        IF array[i] = target THEN
            RETURN i                 # found at index i
        ENDIF
    ENDFOR
    RETURN -1                        # not found
ENDFUNCTION

# Python
def linear_search(arr, target):
    for i in range(len(arr)):
        if arr[i] == target:
            return i
    return -1

Binary Search

Binary search: Repeatedly divides a sorted list in half, comparing the target to the middle element. Each comparison eliminates half the remaining search space. Precondition: data must be sorted. Requires random access (arrays, not linked lists). 
# Pseudocode (iterative)
FUNCTION binarySearch(array, target)
    low ← 0
    high ← LENGTH(array) - 1

    WHILE low <= high
        mid ← (low + high) DIV 2

        IF array[mid] = target THEN
            RETURN mid                   # found!
        ELSE IF array[mid] < target THEN
            low ← mid + 1              # target in RIGHT half
        ELSE
            high ← mid - 1             # target in LEFT half
        ENDIF
    ENDWHILE

    RETURN -1                             # not found
ENDFUNCTION

# Python
def binary_search(arr, target):
    low, high = 0, len(arr) - 1
    while low <= high:
        mid = (low + high) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            low = mid + 1
        else:
            high = mid - 1
    return -1

4.3.5 Sorting Algorithms

Bubble Sort

Bubble sort: Repeatedly steps through the list, compares adjacent elements, and swaps them if in the wrong order. Each pass "bubbles" the largest unsorted element to its correct position. An optimised version uses a swapped flag to exit early if a pass makes no swaps. 
# Pseudocode
FUNCTION bubbleSort(array)
    n ← LENGTH(array)
    FOR i ← 0 TO n - 2
        swapped ← False
        FOR j ← 0 TO n - 2 - i         # -i: last i elements already sorted
            IF array[j] > array[j+1] THEN
                SWAP array[j] AND array[j+1]
                swapped ← True
            ENDIF
        ENDFOR
        IF NOT swapped THEN
            BREAK                            # early exit: already sorted
        ENDIF
    ENDFOR
    RETURN array
ENDFUNCTION

# Python
def bubble_sort(arr):
    n = len(arr)
    for i in range(n - 1):
        swapped = False
        for j in range(n - 1 - i):
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]
                swapped = True
        if not swapped:
            break
    return arr

Merge Sort

Merge sort: A divide-and-conquer algorithm: (1) divide the list into two halves, (2) recursively sort each half, (3) merge the two sorted halves. Merging two sorted lists is O(n). 
# Pseudocode
FUNCTION mergeSort(array)
    IF LENGTH(array) <= 1 THEN
        RETURN array                       # base case
    ENDIF

    mid ← LENGTH(array) DIV 2
    left ← mergeSort(array[0..mid-1])
    right ← mergeSort(array[mid..end])
    RETURN merge(left, right)
ENDFUNCTION

FUNCTION merge(left, right)
    result ← []
    i ← 0
    j ← 0

    WHILE i < LENGTH(left) AND j < LENGTH(right)
        IF left[i] <= right[j] THEN
            APPEND left[i] TO result
            i ← i + 1
        ELSE
            APPEND right[j] TO result
            j ← j + 1
        ENDIF
    ENDWHILE

    APPEND remaining elements of left
    APPEND remaining elements of right
    RETURN result
ENDFUNCTION
# Python
def merge_sort(arr):
    if len(arr) <= 1:
        return arr
    mid = len(arr) // 2
    left = merge_sort(arr[:mid])
    right = merge_sort(arr[mid:])
    return merge(left, right)

def merge(left, right):
    result = []
    i = j = 0
    while i < len(left) and j < len(right):
        if left[i] <= right[j]:
            result.append(left[i])
            i += 1
        else:
            result.append(right[j])
            j += 1
    result.extend(left[i:])
    result.extend(right[j:])
    return result

Quick Sort

Quick sort: A divide-and-conquer algorithm: (1) select a pivot element, (2) partition so elements < pivot go left, elements > pivot go right, (3) recursively sort each partition. The pivot ends up in its final sorted position. 
# Pseudocode
FUNCTION quickSort(array, low, high)
    IF low < high THEN
        pivotIndex ← partition(array, low, high)
        quickSort(array, low, pivotIndex - 1)
        quickSort(array, pivotIndex + 1, high)
    ENDIF
ENDFUNCTION

FUNCTION partition(array, low, high)
    pivot ← array[high]              # last element as pivot
    i ← low - 1

    FOR j ← low TO high - 1
        IF array[j] <= pivot THEN
            i ← i + 1
            SWAP array[i] AND array[j]
        ENDIF
    ENDFOR

    SWAP array[i+1] AND array[high]    # place pivot
    RETURN i + 1
ENDFUNCTION
# Python
def quick_sort(arr, low=0, high=None):
    if high is None:
        high = len(arr) - 1
    if low < high:
        pivot_idx = partition(arr, low, high)
        quick_sort(arr, low, pivot_idx - 1)
        quick_sort(arr, pivot_idx + 1, high)
    return arr

def partition(arr, low, high):
    pivot = arr[high]
    i = low - 1
    for j in range(low, high):
        if arr[j] <= pivot:
            i += 1
            arr[i], arr[j] = arr[j], arr[i]
    arr[i + 1], arr[high] = arr[high], arr[i + 1]
    return i + 1

Stable sort: Preserves the relative order of elements with equal keys. If two records have the same key and appear in a certain order before sorting, a stable sort keeps that order. Important when sorting by multiple criteria (e.g., sort by name, then by age — stability preserves name order within each age group).
Know the best/average/worst complexities and when each sort is suitable: bubble sort for nearly-sorted small data; merge sort when guaranteed O(n log n) is needed or for external sorting of large files; quick sort for general-purpose in-memory sorting (fastest in practice). A common mistake is forgetting quick sort's worst case is O(n2). Also know that merge sort uses O(n) extra space, while bubble sort and quick sort are in-place.

4.3.6 Reverse Polish Notation (RPN)

Reverse Polish Notation (postfix notation): A way of writing expressions where operators come after their operands, eliminating the need for parentheses and precedence rules.

Infix vs Postfix

(3 + 4) * 23 4 + 2 *"Add 3 and 4, multiply by 2"

Converting Infix to RPN (Expression Tree Method)

Build the expression as a tree (operands as leaves, operators as internal nodes respecting precedence), then perform a post-order traversal to get the RPN.

Expression: (3 + 4) * 2

Tree:        *
           /   \
          +     2
         / \
        3   4

Post-order traversal: 3, 4, +, 2, *
RPN: 3 4 + 2 *

Converting Infix to RPN (Shunting Yard Algorithm)

Uses an output queue and an operator stack:

  1. Read tokens left to right
  2. Number: add to output queue
  3. Operator: pop operators of ≥ precedence from stack to output, then push current operator
  4. (: push onto stack
  5. ): pop to output until ( found, discard (
  6. End: pop all remaining operators to output

3[3][]Number → output
+[3][+]Push operator
4[3, 4][+]Number → output
*[3, 4][+, *]* > + precedence, push
2[3, 4, 2][+, *]Number → output
END[3, 4, 2, *, +][]Pop remaining

Result: 3 4 2 * +

Evaluating RPN Using a Stack

  1. Read tokens left to right
  2. Number: push onto stack
  3. Operator: pop two values, apply operator (second-popped OP first-popped), push result
  4. Final value on stack is the answer

Worked Example: Evaluate 5 3 - 8 2 + *

TokenActionStack
5push 5[5]
3push 3[5, 3]
-pop 3, pop 5: 5-3=2[2]
8push 8[2, 8]
2push 2[2, 8, 2]
+pop 2, pop 8: 8+2=10[2, 10]
*pop 10, pop 2: 2*10=20[20]

Result: 20. Verify: (5 - 3) * (8 + 2) = 2 * 10 = 20.

Another Example: Evaluate 10 3 2 - /

TokenActionStack
10push 10[10]
3push 3[10, 3]
2push 2[10, 3, 2]
-pop 2, pop 3: 3-2=1[10, 1]
/pop 1, pop 10: 10/1=10[10]

Result: 10. Verify: 10 / (3 - 2) = 10.

Critical operand order rule: First popped = right operand, second popped = left operand. So 5 3 - means 5 - 3 = 2 (NOT 3 - 5).

Why Use RPN?

  • No brackets or precedence rules — unambiguous
  • Easy for computers to evaluate using a simple stack algorithm
  • Used in stack-based calculators, PostScript, and internally by compilers
The most common mistake: getting operand order wrong for subtraction and division. For 5 3 -, you pop 3 first (right), then 5 (left), giving 5 - 3 = 2, NOT 3 - 5. Remember: second-popped OPERATOR first-popped.

4.3.7 Time Complexity and Big-O Notation

Big-O notation: Describes the upper bound of an algorithm's growth rate as input size n increases. It expresses how execution time scales, ignoring constants and lower-order terms. Big-O answers: "As n gets arbitrarily large, how does the running time grow?"

How to Determine Big-O

  • Identify the dominant operation (the operation performed most often)
  • Count how many times it executes as a function of n
  • Drop constants: O(3n) = O(n), O(n/2) = O(n)
  • Drop lower-order terms: O(n2 + n) = O(n2), O(n + log n) = O(n)
  • Single loop over n items = O(n)
  • Nested loop (both over n) = O(n2)
  • Halving the problem each step = O(log n)
# O(1) - constant time
def get_first(arr):
    return arr[0]          # always one operation

# O(n) - linear
def find_max(arr):
    max_val = arr[0]
    for item in arr:       # loops n times
        if item > max_val:
            max_val = item
    return max_val

# O(n^2) - quadratic
def has_duplicates(arr):
    for i in range(len(arr)):        # n times
        for j in range(i+1, len(arr)):  # up to n times
            if arr[i] == arr[j]:
                return True
    return False

# O(log n) - logarithmic
def binary_search(arr, target):
    low, high = 0, len(arr) - 1
    while low <= high:    # halves search space each iteration
        mid = (low + high) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            low = mid + 1
        else:
            high = mid - 1
    return -1

Tractable vs Intractable Problems

Tractable problem: A problem that can be solved in polynomial time (O(nk) for some constant k). Algorithms like O(n), O(n2), O(n3) are tractable.
Intractable problem: A problem where the best known algorithm requires exponential or worse time (O(2n), O(n!)). These become practically unsolvable as n grows. Examples: travelling salesman problem, boolean satisfiability.

Halting Problem

Halting problem: Given an arbitrary program and its input, determine whether the program will eventually halt (finish) or run forever. Alan Turing proved in 1936 that no general algorithm can solve this problem for all possible programs. This is an undecidable (non-computable) problem. It places a fundamental limit on what computers can do.

The proof uses a contradiction (diagonalisation): if a halting detector H existed, you could construct a program that does the opposite of what H predicts, creating a logical paradox. Therefore, H cannot exist.

Know the hierarchy: O(1) < O(log n) < O(n) < O(n log n) < O(n2) < O(2n) < O(n!). Be able to determine Big-O by analysing code. Remember that Big-O describes the worst case growth rate. A common error is saying O(n2/2) — constants are always dropped. Also understand the difference between tractable (polynomial time) and intractable (exponential time), and that the halting problem is undecidable — no algorithm can solve it for all inputs.

4.3.8 Optimisation Algorithms

Optimisation algorithms find the best solution from a set of possible solutions. Two key approaches for the AQA specification:

Shortest Path (Dijkstra's)

Covered in detail in section 4.3.3 above. Dijkstra's is the standard algorithm for finding shortest paths in weighted graphs with non-negative edges.

Heuristic Approaches

Heuristic: A "rule of thumb" or approximation technique that finds a good enough solution in reasonable time, without guaranteeing the optimal solution. Used when finding the exact optimal solution is intractable (e.g., travelling salesman problem).

Heuristics trade optimality for speed. They are useful when:

  • The problem is NP-hard or intractable (no known polynomial-time exact algorithm)
  • The input is too large for exact methods
  • A "good enough" solution is acceptable

Examples of heuristic approaches:

  • Nearest neighbour (TSP): always visit the closest unvisited city. Fast (O(n2)) but may not find the optimal tour.
  • A* search: uses a heuristic function to estimate the remaining cost, guiding the search toward the goal more efficiently than Dijkstra's (which explores in all directions).
  • Greedy algorithms: make the locally optimal choice at each step (Dijkstra's is greedy but happens to produce the global optimum for non-negative weights).
Understand that some problems (like the travelling salesman) cannot be solved exactly in polynomial time for large inputs. Heuristics provide practical approximate solutions. Know the difference between an exact algorithm (guaranteed optimal but potentially slow) and a heuristic (fast but only approximately optimal). The AQA spec expects you to understand when and why heuristics are used.
