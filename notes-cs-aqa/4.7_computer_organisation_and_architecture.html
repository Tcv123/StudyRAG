4.7.1 Internal Hardware Components of a Computer

The Processor (CPU)

The Central Processing Unit (CPU) is the primary component responsible for executing instructions. It contains the Arithmetic Logic Unit (ALU), the Control Unit (CU), and a set of registers. Modern CPUs also contain on-die cache memory to speed up data access.

Processor: An electronic circuit that executes instructions from a program by performing arithmetic, logic, control, and input/output operations. It is the "brain" of the computer, but it can only do what its instructions tell it to do.

Key internal components:

  • ALU (Arithmetic Logic Unit) — performs all arithmetic calculations (add, subtract, multiply, divide) and logical operations (AND, OR, NOT, XOR, comparisons). The ALU takes inputs from registers and outputs results to the accumulator.
  • Control Unit (CU) — coordinates all CPU operations. It fetches instructions from memory, decodes them, and sends control signals to the ALU, registers, and other components. It generates timing signals using the system clock. The CU does not process data itself.
  • Registers — ultra-fast storage locations inside the CPU, each holding a single word of data. They are the fastest form of storage but very limited in number. Different registers have specialised roles.
  • Cache — small, fast SRAM memory between the CPU and main memory. It stores copies of frequently accessed data and instructions to reduce the average time to access memory.
  • Clock — an electronic oscillator that generates a regular pulse signal, synchronising all CPU operations. Each pulse is one clock cycle. The clock speed (measured in GHz) determines how many cycles occur per second.

Main Memory

Main memory (primary storage) holds data and instructions that the CPU needs to access quickly. It is directly addressable by the CPU via the address bus.

The System Bus

The system bus is a set of parallel wires that connects the CPU to main memory and I/O controllers. It consists of three distinct sub-buses, each carrying a different type of signal:

BusPurposeDirectionWidth Effect
Address BusCarries memory addresses from the CPU to identify the location being accessedUnidirectional (CPU → memory/IO)n-bit address bus can address 2n unique locations. E.g., 32-bit bus = 4 GB addressable; 64-bit = 16 exabytes.
Data BusCarries actual data between CPU, memory, and I/O devicesBidirectionalWider data bus = more data per transfer. 64-bit data bus transfers 8 bytes per operation.
Control BusCarries control signals to coordinate operationsBidirectionalMultiple individual lines, each carrying a specific signal.

Control signals carried on the control bus include:

  • Memory Read / Memory Write — signals indicating whether data is being read from or written to memory
  • Bus Request / Bus Grant — used by I/O devices requesting and being granted control of the bus (for DMA — Direct Memory Access)
  • Clock — the timing signal that synchronises all operations
  • Interrupt — signals from I/O devices or software requesting CPU attention
  • Reset — resets the CPU to its initial state
The width of the address bus determines how many unique memory locations can be addressed: 2n locations for an n-bit bus. The width of the data bus determines how much data can be transferred in a single operation. If asked "what happens if you increase the address bus from 32 to 36 bits?", the answer is that addressable memory increases from 232 = 4 GB to 236 = 64 GB.

4.7.2 Von Neumann Architecture

Stored Program Concept: The idea (proposed by John von Neumann in 1945) that instructions and data are stored together in the same main memory, and instructions are fetched and executed sequentially (unless a branch instruction is encountered). Before this, computers were rewired for each new program.

Key features of the Von Neumann architecture:

  • A single memory space holds both instructions and data — the CPU cannot distinguish between them by looking at the bits; it is the context (fetch vs data access) that determines interpretation.
  • A single bus system connects the CPU to memory
  • Instructions are fetched and executed sequentially (one after another, in order)
  • The CPU contains an ALU, CU, and registers
  • Programs can modify themselves at runtime (since instructions are data that can be overwritten) — though this is generally avoided in modern practice for security reasons.

Key Registers

RegisterAbbreviationPurpose

Always be precise about what each register holds. The PC holds the address of the next instruction, not the current one. The MAR holds an address; the MDR holds data (or an instruction fetched from that address). The CIR holds the current instruction being decoded. In AQA mark schemes, this precision is required for full marks.

4.7.3 Harvard Architecture

The Harvard architecture uses physically separate memory and separate buses for instructions and data. This allows the CPU to fetch an instruction and read/write data simultaneously, doubling potential throughput.

FeatureVon NeumannHarvard
MemorySingle shared memory for instructions and dataSeparate memory for instructions and data
Bus systemSingle bus (potential bottleneck)Separate buses for instruction and data
Simultaneous accessNo — fetch or data access, not both at onceYes — can fetch next instruction while accessing data for current instruction
SpeedSlower (Von Neumann bottleneck)Faster for specific tasks
FlexibilityMore flexible (programs can be loaded from anywhere; self-modifying code possible)Less flexible (instruction memory size is fixed at design time)
Typical useGeneral-purpose computers (desktops, laptops, servers)Embedded systems (microcontrollers), DSPs (Digital Signal Processors), microcontrollers (Arduino, PIC)
Von Neumann Bottleneck: The limitation that arises because both data and instructions share the same bus, meaning the CPU can be idle while waiting for data to be transferred. The bus becomes a performance bottleneck because the rate of data transfer is much slower than the rate at which the CPU can process data. Harvard architecture avoids this by having separate buses.

Modified Harvard architecture is used in most modern processors: separate L1 caches for instructions (L1i) and data (L1d) provide Harvard-style parallel access at the cache level, but a unified main memory (Von Neumann style) retains flexibility. This hybrid approach provides the speed benefits of Harvard while maintaining the flexibility of Von Neumann.

4.7.4 The Fetch-Decode-Execute Cycle

The CPU continuously repeats three stages to process each instruction. This cycle runs billions of times per second in a modern processor.

1. Fetch

  1. The contents of the PC are copied to the MAR. (PC → MAR)
  2. The PC is incremented (PC ← PC + 1) to point to the next instruction. (This happens early so branch instructions can overwrite the PC if needed.)
  3. The address in the MAR is placed on the address bus and sent to main memory.
  4. The control unit sends a memory read signal along the control bus.
  5. The data at the addressed memory location is transferred along the data bus into the MDR.
  6. The contents of the MDR are copied to the CIR. (MDR → CIR)

2. Decode

  1. The control unit decodes the instruction held in the CIR.
  2. The instruction is split into the opcode (the operation to perform, e.g., ADD, LDR, BRA) and the operand (the data value or address to operate on).
  3. The CU determines what control signals need to be sent and to which components.
  4. If the operand is a memory address, the addressing mode determines how to interpret it (immediate, direct, indirect, or indexed).

3. Execute

  1. The decoded instruction is carried out. This could involve:
    • An ALU operation — arithmetic or logic calculation, with the result stored in the ACC
    • A data transfer — loading data from memory into a register, or storing register contents to memory (requires MAR/MDR and the bus system)
    • A branch/jump — loading a new address into the PC, breaking sequential execution
    • A comparison — comparing two values and setting flags in the status register (Zero, Negative, Carry, Overflow)
    • A halt instruction — stopping the CPU
  2. Results may be stored in the accumulator or written back to memory via MDR and the data bus.
  3. The status register flags are updated to reflect the result.

FDE Cycle Walkthrough Example

  Memory contents:
  Address 100: LDR R0, 200    (Load value at address 200 into R0)
  Address 101: ADD R0, R0, #5  (Add 5 to R0)
  Address 102: STR R0, 201    (Store R0 into address 201)
  Address 200: 42             (Data value)
  Address 201: 0              (Empty, will receive result)

  Cycle 1 (instruction at 100):
    Fetch:   PC=100 → MAR=100, PC incremented to 101
             Memory[100] → MDR = "LDR R0, 200", MDR → CIR
    Decode:  Opcode = LDR, Operand = address 200
    Execute: MAR=200, Memory[200] → MDR=42, MDR → R0
             R0 now contains 42

  Cycle 2 (instruction at 101):
    Fetch:   PC=101 → MAR=101, PC=102
             Memory[101] → MDR = "ADD R0, R0, #5", MDR → CIR
    Decode:  Opcode = ADD, Operand = R0, #5 (immediate)
    Execute: ALU calculates R0 + 5 = 42 + 5 = 47, result → R0

  Cycle 3 (instruction at 102):
    Fetch:   PC=102 → MAR=102, PC=103
             Memory[102] → MDR = "STR R0, 201", MDR → CIR
    Decode:  Opcode = STR, Operand = address 201
    Execute: R0 → MDR=47, MAR=201, MDR written to Memory[201]
             Memory[201] now contains 47
When describing the FDE cycle, always reference the specific registers and buses involved. Saying "the instruction is fetched from memory" is not detailed enough — you must describe the role of PC, MAR, MDR, CIR, and the address/data/control buses. Include the register transfers (e.g., PC → MAR, MDR → CIR).

Interrupts

After the execute phase of each instruction, the CPU checks for interrupts. An interrupt is a signal (sent via the control bus) from hardware or software requesting the CPU's immediate attention.

Common interrupt sources:

  • I/O interrupts: keyboard key pressed, mouse moved, printer ready, disk transfer complete
  • Timer interrupts: the OS scheduler's time quantum has expired (triggers context switch)
  • Software interrupts (traps): division by zero, stack overflow, system call from a program
  • Hardware faults: memory parity error, power failure warning

Interrupt handling process:

  1. The CPU checks the interrupt flag after each execute phase. If interrupts are disabled (the interrupt mask is set), the interrupt is queued.
  2. If an interrupt is detected and interrupts are enabled, the CPU saves its current state — all register contents (PC, ACC, status register, etc.) are pushed onto the system stack.
  3. The PC is loaded with the address of the appropriate Interrupt Service Routine (ISR), found via an interrupt vector table.
  4. The ISR executes to handle the interrupt (e.g., read the keystroke, acknowledge the device).
  5. On completion, the CPU restores its previous state by popping values from the stack and resumes the interrupted program exactly where it left off.

Interrupt priorities: Multiple interrupts can occur simultaneously. Higher-priority interrupts (e.g., power failure) can interrupt lower-priority ISRs. This creates a nested interrupt system.

4.7.5 Processor Instruction Set

Machine Code: The lowest-level programming language, consisting of binary instructions that the CPU can directly execute. Each instruction consists of an opcode (the operation) and typically one or more operands (the data or address).

Instruction Format

| Opcode | Operand |
|  ADD   |  #42    |   ← Add the value 42 to the accumulator (immediate)
|  LDR   |  200    |   ← Load the value at memory address 200 (direct)
|  BRA   |  loop   |   ← Branch unconditionally to label "loop"
|  HALT  |         |   ← No operand needed

Types of Operation

  • Data transfer: LDR (load from memory to register), STR (store from register to memory), MOV (copy between registers)
  • Arithmetic: ADD, SUB, MUL, DIV
  • Logical / Bitwise: AND, ORR, EOR (XOR), MVN (bitwise NOT)
  • Shift: LSL (logical shift left), LSR (logical shift right), ASR (arithmetic shift right)
  • Branch: B (unconditional), BEQ (branch if equal/zero flag set), BNE (branch if not equal), BGT (branch if greater than), BLT (branch if less than)
  • Comparison: CMP (compare two values, sets status register flags without storing the result)
  • I/O: IN, OUT (not in all instruction sets)
  • HALT: Stop execution

Addressing Modes

ModeDescriptionExampleUse Case
ImmediateThe operand is the actual data value. No memory access needed.ADD R0, R0, #5 — add the literal value 5Loading constants, incrementing counters
DirectThe operand is the memory address where the data is stored.LDR R0, 200 — load the value stored at address 200Accessing global variables
IndirectThe operand is the address of a memory location that contains the address of the actual data (a pointer).LDR R0, @200 — go to address 200, read the value there (say 350), then load from address 350Implementing pointers, dynamic data structures
IndexedThe operand is added to the contents of an index register to form the effective address.LDR R0, 200, IX — load from address (200 + value in IX)Iterating through arrays: base address stays constant, IX increments
Indirect addressing is like following a pointer — the memory location holds a reference to another location. Indexed addressing is essential for array access: the base address stays constant while the index register is incremented each iteration. Be ready to trace through instructions using each addressing mode.

4.7.7 Pipelining

Pipelining is a technique where multiple instructions are overlapped during execution. The FDE cycle is divided into stages, and each stage processes a different instruction simultaneously, like an assembly line in a factory.

Clock CycleFetch UnitDecode UnitExecute Unit
1Instruction 1——
2Instruction 2Instruction 1—
3Instruction 3Instruction 2Instruction 1
4Instruction 4Instruction 3Instruction 2
5Instruction 5Instruction 4Instruction 3

Without pipelining: each instruction takes 3 cycles; 5 instructions take 15 cycles.
With pipelining: after the initial fill (3 cycles), one instruction completes per cycle; 5 instructions take only 7 cycles. Throughput is approximately tripled.

Pipeline Hazards — situations that prevent the next instruction from entering the pipeline:

  • Control hazards (branch hazards): A branch instruction makes the destination of the next fetch unknown until the branch is resolved. The pipeline may have to be flushed (all partially completed instructions discarded), wasting the work done so far. Branch prediction guesses the outcome to keep the pipeline full; if wrong, the pipeline is flushed.
  • Data hazards: An instruction depends on the result of a previous instruction that has not yet completed its execute stage. E.g., instruction 2 needs the result that instruction 1 is still computing. Solutions include forwarding (passing the result directly between pipeline stages) and stalling (inserting "bubbles" / NOPs).
  • Structural hazards: Two instructions need the same hardware resource at the same time (e.g., both need to access memory). Resolved by having separate instruction and data caches (Harvard-style at cache level).

4.7.9 Multicore and Parallel Processing

Multicore Processors

A multicore processor has two or more independent processing units (cores) on a single chip. Each core has its own ALU, CU, L1 cache (and often L2 cache), and can execute its own FDE cycle independently and simultaneously.

Advantages:

  • Multiple tasks can be processed truly simultaneously (not just time-sliced)
  • Better performance for multi-threaded applications (web browsers, video editing, compiling)
  • Lower power consumption and heat generation compared to achieving the same throughput by increasing clock speed
  • The OS can assign different processes to different cores

Limitations:

  • Software must be written to use multiple threads to benefit from multiple cores — single-threaded software runs on only one core regardless of how many are available
  • Not all tasks can be parallelised — some computations are inherently sequential (each step depends on the previous result)
  • Amdahl's Law — the overall speedup is limited by the proportion of the task that must be done sequentially. If 25% of a program is sequential, the maximum speedup (with infinite cores) is only 4x.
  • Overhead from coordinating cores: inter-core communication, cache coherency (ensuring all cores see consistent data), synchronisation (preventing race conditions)

Parallel Processing

Parallel processing involves dividing a task into smaller sub-tasks that can be executed simultaneously across multiple cores or processors.

  • Data parallelism: The same operation is applied to different pieces of data simultaneously. Example: applying a brightness filter to every pixel in an image — each core processes a different section of the image.
  • Task parallelism: Different operations are performed simultaneously on the same or different data. Example: one core handles user input while another renders graphics and a third processes audio.

4.7.10 Co-processors (GPUs)

Co-processor: A supplementary processor designed to handle specific types of computation, offloading work from the main CPU. This allows the CPU to focus on general-purpose tasks while the co-processor handles specialised workloads.

Graphics Processing Unit (GPU)

A GPU was originally designed for rendering graphics but is now widely used for general-purpose parallel computation (GPGPU).

Applications of GPU computing:

  • Machine learning / AI: Training neural networks involves millions of matrix multiplications — perfectly suited to GPU parallelism (using frameworks like CUDA, TensorFlow, PyTorch)
  • Cryptocurrency mining: Hash computations are highly parallel
  • Scientific simulations: Weather modelling, molecular dynamics, fluid dynamics, protein folding
  • Image and video processing: Filters, encoding, decoding
  • Real-time ray tracing: Simulating light physics for photorealistic rendering in games
