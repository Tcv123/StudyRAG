4.7.1 Internal Hardware Components

The Processor (CPU)

CPU (Central Processing Unit): An electronic circuit that executes instructions by performing arithmetic, logic, control, and I/O operations. It contains the ALU, Control Unit, registers, cache, and a clock.
  • ALU (Arithmetic Logic Unit) — performs arithmetic (add, subtract, multiply, divide) and logical operations (AND, OR, NOT, XOR, comparisons). Takes inputs from registers, outputs to the accumulator.
  • Control Unit (CU) — coordinates all CPU operations. Fetches instructions, decodes them, sends control signals. Generates timing signals via the system clock. Does not process data itself.
  • Registers — ultra-fast storage inside the CPU, each holding one word of data. The fastest form of storage but very limited in number.
  • Cache — small, fast SRAM between CPU and main memory. Stores frequently accessed data/instructions to reduce average memory access time.
  • Clock — electronic oscillator generating a regular pulse signal. Each pulse = one clock cycle. Speed measured in GHz (billions of cycles/second).

4.7.2 Von Neumann Architecture

Stored Program Concept (von Neumann, 1945): Instructions and data are stored together in the same main memory, and instructions are fetched and executed sequentially. Before this, computers were physically rewired for each new program.
  • Single memory holds both instructions and data
  • Single bus system connects CPU to memory
  • Instructions fetched and executed sequentially
  • Programs can modify themselves (instructions are writable data)

Be precise: PC holds the address of the next instruction (not current). MAR holds an address; MDR holds data. CIR holds the current instruction. This precision is required for full marks in AQA mark schemes.

4.7.3 Harvard Architecture

Uses physically separate memory and buses for instructions and data, enabling simultaneous fetch and data access.

FeatureVon NeumannHarvard
MemorySingle shared memorySeparate instruction and data memory
BusesSingle bus (bottleneck)Separate instruction and data buses
Simultaneous accessNo (fetch OR data, not both)Yes (fetch instruction while accessing data)
FlexibilityMore flexible (programs loaded from anywhere)Less flexible (instruction memory fixed at design)
UseGeneral-purpose (desktops, laptops)Embedded systems, DSPs, microcontrollers (Arduino)
Von Neumann Bottleneck: Both data and instructions share the same bus, so the CPU idles while waiting for data transfer. The bus transfer rate is much slower than CPU processing speed. Harvard avoids this with separate buses.

Modified Harvard architecture (most modern processors): separate L1 caches for instructions (L1i) and data (L1d) provide Harvard-style parallelism, but unified main memory retains Von Neumann flexibility.

4.7.4 The Fetch-Decode-Execute Cycle

1. Fetch

  1. PC → MAR (address of next instruction sent to MAR)
  2. PC ← PC + 1 (increment to point to next instruction)
  3. MAR placed on address bus
  4. CU sends memory read signal on control bus
  5. Data from addressed location → data bus → MDR
  6. MDR → CIR (instruction now in CIR for decoding)

2. Decode

  1. CU decodes instruction in CIR
  2. Split into opcode (operation) and operand (data/address)
  3. CU determines required control signals and addressing mode

3. Execute

  1. Instruction carried out:
    • ALU operation — arithmetic/logic, result to ACC
    • Data transfer — load from memory or store to memory via MAR/MDR
    • Branch/jump — new address loaded into PC
    • Comparison — sets flags in status register
    • Halt — stops the CPU
  2. Status register flags updated
  3. CPU checks for interrupts

FDE Walkthrough

Memory:
  100: LDR R0, 200    (Load value at address 200 into R0)
  101: ADD R0, R0, #5 (Add 5 to R0)
  102: STR R0, 201    (Store R0 to address 201)
  200: 42             (data)
  201: 0              (empty)

Cycle 1 (address 100):
  Fetch:   PC=100 → MAR, PC=101, Memory[100] → MDR → CIR
  Decode:  Opcode=LDR, Operand=address 200
  Execute: MAR=200, Memory[200] → MDR=42 → R0. R0 is now 42.

Cycle 2 (address 101):
  Fetch:   PC=101 → MAR, PC=102, Memory[101] → MDR → CIR
  Decode:  Opcode=ADD, Operands=R0, #5 (immediate)
  Execute: ALU: 42+5=47 → R0

Cycle 3 (address 102):
  Fetch:   PC=102 → MAR, PC=103, Memory[102] → MDR → CIR
  Decode:  Opcode=STR, Operand=address 201
  Execute: R0(47) → MDR, MAR=201, MDR written to Memory[201]
Always reference specific registers and buses. "The instruction is fetched" is insufficient — describe PC → MAR, memory read signal on control bus, data bus → MDR → CIR. This level of detail is required for full marks.

Interrupts

After each execute phase, the CPU checks for interrupts — signals requesting immediate attention.

Sources: I/O devices (key press, disk complete), timer (scheduler quantum expired), software traps (division by zero, system call), hardware faults (memory error).

Interrupt handling:

  1. CPU checks interrupt flag after execute phase
  2. If interrupt detected and enabled: save current state (all registers) onto system stack
  3. PC loaded with address of Interrupt Service Routine (ISR) from interrupt vector table
  4. ISR executes to handle the interrupt
  5. On completion, saved state is restored from the stack (registers popped)
  6. CPU resumes the interrupted program from where it left off

Interrupt priorities: Higher-priority interrupts can interrupt lower-priority ISRs (nested interrupts). The interrupt mask can disable certain interrupts temporarily.

4.7.6 Machine Code and Assembly Language

Machine Code

Machine code is the binary instructions directly executed by the CPU. Each instruction consists of an opcode (what operation) and an operand (data or address). Machine code is processor-specific — different CPU architectures have different instruction sets.

Assembly Language

Assembly language uses human-readable mnemonics for machine code instructions. There is a one-to-one mapping between assembly mnemonics and machine code instructions.

# AQA assembly instruction set examples:
LDR Rd, <memory>     # Load register Rd with value from memory
STR Rd, <memory>     # Store register Rd to memory
ADD Rd, Rn, <op2>    # Rd = Rn + operand2
SUB Rd, Rn, <op2>    # Rd = Rn - operand2
MOV Rd, <op2>        # Rd = operand2
CMP Rn, <op2>        # Compare Rn with operand2 (sets flags)
B <label>             # Branch unconditionally
BEQ <label>           # Branch if equal (Zero flag set)
BNE <label>           # Branch if not equal
BGT <label>           # Branch if greater than
BLT <label>           # Branch if less than
AND Rd, Rn, <op2>    # Rd = Rn AND operand2
ORR Rd, Rn, <op2>    # Rd = Rn OR operand2
EOR Rd, Rn, <op2>    # Rd = Rn XOR operand2
MVN Rd, <op2>        # Rd = NOT operand2
LSL Rd, Rn, <op2>    # Logical shift left
LSR Rd, Rn, <op2>    # Logical shift right
HALT                   # Stop execution

Example: Sum numbers 1 to 5

        MOV R0, #0        ; R0 = sum = 0
        MOV R1, #1        ; R1 = counter = 1
loop:   ADD R0, R0, R1    ; sum += counter
        ADD R1, R1, #1    ; counter++
        CMP R1, #6        ; compare counter with 6
        BLT loop          ; if counter < 6, loop again
        STR R0, result    ; store sum
        HALT
result: 0                 ; memory location for result
; After execution: result = 15 (1+2+3+4+5)

Pipelining

Pipelining overlaps the fetch, decode, and execute stages of sequential instructions. While instruction 1 executes, instruction 2 is decoded, and instruction 3 is fetched — all simultaneously, tripling throughput in ideal conditions. 
Without pipelining:
  Cycle:  1  2  3  4  5  6  7  8  9
  Inst 1: F  D  E
  Inst 2:          F  D  E
  Inst 3:                   F  D  E
  3 instructions in 9 cycles

With pipelining:
  Cycle:  1  2  3  4  5
  Inst 1: F  D  E
  Inst 2:    F  D  E
  Inst 3:       F  D  E
  3 instructions in 5 cycles (nearly 2x faster)

Pipeline hazards:

  • Data hazard — instruction depends on result of a previous instruction not yet complete
  • Control hazard — branch instruction changes PC; instructions already in pipeline are wrong (pipeline flush). Mitigated by branch prediction.
  • Structural hazard — two instructions need the same hardware resource simultaneously

4.7.9 GPUs, Multicore, and Parallel Processing

Multicore Processors

A multicore processor has multiple independent processing cores on a single chip. Each core can execute its own FDE cycle independently, enabling true parallel processing.

  • Dual-core: 2 cores; quad-core: 4 cores; octa-core: 8 cores
  • Not all tasks benefit — only parallelisable workloads speed up
  • Amdahl's Law: the serial portion of a program limits the maximum speedup from parallelism

GPU (Graphics Processing Unit)

A GPU contains thousands of small, simple cores optimised for performing the same operation on many data points simultaneously (SIMD — Single Instruction, Multiple Data). Originally for graphics rendering, now used for general-purpose parallel computing (GPGPU).

4.7.11 I/O Methods: Programmed I/O, Interrupts, and DMA

Three different methods allow the CPU to communicate with I/O devices. Each represents a different trade-off between CPU involvement and efficiency.

1. Programmed I/O (Polling)

In programmed I/O (polling), the CPU repeatedly checks a status register on the I/O device to see whether it is ready. The CPU is entirely in control and actively waits in a busy loop.
  • The CPU enters a polling loop, reading the device status register over and over
  • When the device signals ready, the CPU performs the transfer
  • Simple to implement; no hardware beyond the device controller
  • Disadvantage: the CPU is occupied doing nothing useful while it polls — this is called busy waiting. Wastes CPU cycles that could run other processes.
  • Acceptable for very fast devices or simple embedded systems where there is nothing else to do
Polling loop (pseudocode):
WHILE device_status_register != READY DO
    WAIT      ; CPU stuck here, doing nothing useful
ENDWHILE
; Device is ready, perform data transfer
READ data_register

2. Interrupt-Driven I/O

In interrupt-driven I/O, the CPU starts an I/O operation and then continues executing other instructions. When the device finishes, it sends an interrupt signal on the control bus. The CPU pauses its current task, saves state, and runs the Interrupt Service Routine (ISR) to complete the transfer.
  • CPU does not waste time polling; it runs other processes while the device works
  • Device signals the CPU via the interrupt request line (IRQ) on the control bus
  • The CPU checks for interrupts at the end of each FDE cycle
  • ISR reads data from the device and stores it; then restores CPU state
  • Advantage: efficient CPU use — handles slow devices without blocking
  • Disadvantage: context-switch overhead; interrupt handling complexity; not ideal for very high-speed data streams (many interrupts per second saturate the CPU)

3. Direct Memory Access (DMA)

DMA uses a dedicated DMA controller chip to transfer data directly between main memory and a device, without routing every byte through the CPU. The CPU only initiates and concludes the transfer.

DMA transfer process:

  1. CPU programs the DMA controller with: source address, destination address, number of bytes to transfer, direction (read/write)
  2. CPU continues executing instructions independently
  3. DMA controller takes control of the buses (cycle stealing or burst mode) and transfers data directly to/from RAM
  4. When the transfer is complete, DMA controller sends a completion interrupt to the CPU
  5. CPU checks results and resumes normal operation
FeatureProgrammed I/O (Polling)Interrupt-Driven I/ODMA
CPU involvement during transfer100% — busy waitingLow — only during ISRVery low — only at start/end
CPU efficiencyVery low (blocked)HighHighest
Hardware complexityLowestMedium (interrupt controller)Highest (DMA controller chip)
Best forSimple/fast devices, embeddedSlow devices (keyboard, mouse)High-bandwidth transfers (disk, GPU, network)
Data pathDevice → CPU → MemoryDevice → CPU (ISR) → MemoryDevice → Memory (directly)
The key exam distinction: polling wastes CPU time (busy waiting); interrupt-driven frees the CPU but involves the CPU for every data item; DMA frees the CPU almost entirely for large bulk transfers. DMA is used by hard drives, graphics cards, and network cards because they move large amounts of data at high speed. A 6-mark question may ask you to compare all three methods — cover CPU use, efficiency, hardware needed, and suitability.

4.7.12 BIOS and the Boot Process

The BIOS (Basic Input/Output System) is firmware stored in ROM on the motherboard. It is the first software to run when a computer is powered on, before the OS is loaded. Modern systems use UEFI (Unified Extensible Firmware Interface) which extends BIOS with a GUI, faster boot, and support for large disks.

BIOS / UEFI functions:

  1. POST (Power-On Self-Test) — checks that essential hardware is present and functional (CPU, RAM, keyboard, storage devices). Errors are reported via beep codes or on-screen messages.
  2. Hardware initialisation — sets up CPU, memory controller, and basic I/O chips to a known working state.
  3. Boot device selection — identifies the bootable device (SSD, HDD, USB, network) based on configured boot order.
  4. Loads the bootloader — reads the first sector (Master Boot Record or UEFI boot partition) from the boot device into RAM and transfers control to it. The bootloader then loads the operating system kernel.

BIOS is stored in ROM (non-volatile — survives power-off). Its primary jobs are POST (test hardware) and bootstrapping (loading the OS). If asked why BIOS is in ROM rather than RAM: ROM is non-volatile so the BIOS persists when power is removed — RAM would lose it. UEFI replaces BIOS on all modern systems; the key improvements are GUI, Secure Boot, and support for drives larger than 2 TB.

Worked Example: Find the Maximum of Two Numbers

; Find max of values at address 200 and 201, store in 202
        LDR R0, 200         ; R0 = first number
        LDR R1, 201         ; R1 = second number
        CMP R0, R1          ; compare R0 and R1
        BGT store           ; if R0 > R1, branch to store
        MOV R0, R1          ; otherwise, R0 = R1 (R1 is larger)
store:  STR R0, 202         ; store max value to address 202
        HALT

; If Memory[200]=15, Memory[201]=22:
; CMP sets flags: 15-22 = negative, so BGT not taken
; MOV R0, R1  sets R0 = 22
; STR stores 22 to Memory[202]

Worked Example: Counting Loop (Multiplication by Repeated Addition)

; Compute 4 x 7 using repeated addition
        MOV R0, #0          ; R0 = product = 0
        MOV R1, #4          ; R1 = multiplicand
        MOV R2, #7          ; R2 = counter (multiply 7 times)
loop:   ADD R0, R0, R1      ; product += multiplicand
        SUB R2, R2, #1      ; counter--
        CMP R2, #0          ; has counter reached 0?
        BNE loop            ; if not, repeat
        STR R0, result      ; store result (should be 28)
        HALT
result: 0

Bitwise Operations in Assembly

; Test if bit 3 of R0 is set (masking):
        AND R1, R0, #8      ; #8 = 00001000 in binary
        CMP R1, #0
        BNE bit_set         ; branch if bit 3 was 1

; Set bit 4 of R0:
        ORR R0, R0, #16     ; #16 = 00010000

; Toggle bit 2 of R0:
        EOR R0, R0, #4      ; #4 = 00000100

; Logical shift left by 3 (multiply by 8):
        LSL R1, R0, #3      ; R1 = R0 * 8
In AQA assembly questions, use # for immediate values (e.g., MOV R0, #5 sets R0 to the number 5) and no # for memory addresses (e.g., LDR R0, 200 loads from address 200). The CMP instruction subtracts but does not store the result — it only sets flags. Always trace through each register value at each step to verify correctness.
