4.12.1 The Functional Programming Paradigm

Functional programming is a programming paradigm in which computation is treated as the evaluation of mathematical functions. Programs are constructed by applying and composing functions, rather than by executing a sequence of statements that change the program's state.

Key principles that distinguish functional programming from imperative programming:

  • Functions are first-class objects — they can be assigned to variables, passed as arguments, and returned from other functions.
  • Programs avoid side effects — functions do not modify external state (no changing global variables, no I/O within pure functions).
  • Data is immutable — once created, data cannot be changed. Any "modification" creates a new value.
  • Functions are pure — the same inputs always produce the same output, and nothing else is affected.
  • Computation is achieved through function application and composition rather than through variable assignment and loops.
  • Recursion replaces iteration (loops) as the primary control structure.

Imperative (Python) vs Functional (Haskell) — summing a list:

-- Imperative approach (Python):
total = 0              # mutable variable
for x in [1,2,3,4]:   # loop
    total = total + x  # state mutation
print(total)           # 10

-- Functional approach (Haskell):
mySum []     = 0                      -- base case
mySum (x:xs) = x + mySum xs           -- recursion, no state mutation
-- mySum [1,2,3,4] = 1 + mySum [2,3,4] = 1+2+mySum [3,4] = ... = 10

4.12.2 First-Class Objects

In functional programming, functions are first-class objects (first-class citizens). This means functions have the same status as any other data value — integers, strings, lists. Specifically, a function can be:
  • Assigned to a variable or stored in a data structure.
  • Passed as an argument to another function.
  • Returned as the result of another function.
  • Created dynamically at runtime (anonymous/lambda functions).
-- Assign a function to a name
double = \x -> x * 2           -- lambda (anonymous function) assigned to "double"

-- Pass a function as an argument
applyTwice f x = f (f x)
applyTwice double 3            -- double(double(3)) = double(6) = 12

-- Return a function from a function
makeAdder n = \x -> x + n      -- returns a NEW function
addFive = makeAdder 5           -- addFive is now a function: \x -> x + 5
addFive 10                      -- Result: 15

-- Python equivalent (functions are first-class in Python too):
# double = lambda x: x * 2
# def apply_twice(f, x): return f(f(x))
# apply_twice(double, 3)  # 12

4.12.3 Higher-Order Functions

A higher-order function is a function that either:
  • Takes one or more functions as arguments, or
  • Returns a function as its result (or both).
Higher-order functions are only possible because functions are first-class objects.

The three most important higher-order functions in functional programming are map, filter, and fold (reduce). These replace the need for explicit loops.

map

map applies a given function to every element of a list, returning a new list of results. The original list is unchanged (immutability).

-- Definition:
map f []     = []
map f (x:xs) = f x : map f xs

-- Examples:
map (*2) [1, 2, 3, 4]       -- Result: [2, 4, 6, 8]
map even [1, 2, 3, 4]       -- Result: [False, True, False, True]
map toUpper "hello"          -- Result: "HELLO"
map (\x -> x*x) [1,2,3,4]   -- Result: [1, 4, 9, 16]  (using lambda)

-- Trace through: map (*2) [1, 2, 3]
-- = (*2) 1 : map (*2) [2, 3]
-- = 2 : (*2) 2 : map (*2) [3]
-- = 2 : 4 : (*2) 3 : map (*2) []
-- = 2 : 4 : 6 : []
-- = [2, 4, 6]

filter

filter takes a predicate (a function returning True/False) and a list, returning a new list containing only elements for which the predicate returns True.

-- Definition:
filter p []     = []
filter p (x:xs)
    | p x       = x : filter p xs     -- include x
    | otherwise  = filter p xs         -- skip x

-- Examples:
filter even [1, 2, 3, 4, 5, 6]     -- Result: [2, 4, 6]
filter (>3) [1, 2, 3, 4, 5]        -- Result: [4, 5]
filter (/= ' ') "h e l l o"        -- Result: "hello" (remove spaces)
filter (\x -> x `mod` 3 == 0) [1..15]  -- Result: [3,6,9,12,15]

-- Trace through: filter even [1, 2, 3, 4]
-- even 1? No  -> filter even [2, 3, 4]
-- even 2? Yes -> 2 : filter even [3, 4]
-- even 3? No  -> filter even [4]
-- even 4? Yes -> 4 : filter even []
-- = 2 : 4 : [] = [2, 4]

fold (reduce)

fold (called reduce in Python/JavaScript) combines all elements of a list into a single value by repeatedly applying a binary function. There are two variants:

foldl (fold left) — processes from left to right, accumulating a result:

-- Definition:
foldl f acc []     = acc
foldl f acc (x:xs) = foldl f (f acc x) xs

-- Example: sum a list using foldl
foldl (+) 0 [1, 2, 3, 4]
  = foldl (+) (0+1) [2, 3, 4]     -- acc=1
  = foldl (+) (1+2) [3, 4]         -- acc=3
  = foldl (+) (3+3) [4]            -- acc=6
  = foldl (+) (6+4) []             -- acc=10
  = 10

-- Example: product of a list
foldl (*) 1 [2, 3, 4]
  = foldl (*) (1*2) [3, 4]  = foldl (*) 2 [3, 4]
  = foldl (*) (2*3) [4]     = foldl (*) 6 [4]
  = foldl (*) (6*4) []      = foldl (*) 24 []
  = 24

-- Example: count elements (reimplementing length)
foldl (\acc _ -> acc + 1) 0 [7, 3, 9]
  = foldl f (0+1) [3, 9] = foldl f 1 [3, 9]
  = foldl f (1+1) [9]    = foldl f 2 [9]
  = foldl f (2+1) []     = 3

foldr (fold right) — processes from right to left:

-- Definition:
foldr f acc []     = acc
foldr f acc (x:xs) = f x (foldr f acc xs)

-- Example: concatenate strings
foldr (++) "" ["Hello", " ", "World"]
  = "Hello" ++ foldr (++) "" [" ", "World"]
  = "Hello" ++ (" " ++ foldr (++) "" ["World"])
  = "Hello" ++ (" " ++ ("World" ++ ""))
  = "Hello" ++ (" " ++ "World")
  = "Hello" ++ " World"
  = "Hello World"
You must be able to trace through map, filter, and fold with specific inputs step by step. In exam answers, show each step clearly on a separate line. Remember the pattern: map transforms each element (same length list out); filter selects elements (shorter or equal list out); fold combines elements into a single value (one value out).

4.12.4 Function Application and Composition

Function Application

Function application is the process of applying a function to its arguments to produce a result. In Haskell notation, function application is written by juxtaposition (placing the function name next to its argument, separated by a space) rather than using parentheses as in mathematics. 
-- Haskell function application:
f x          -- apply f to x         (maths: f(x))
add 3 5      -- apply add to 3 and 5 (maths: add(3, 5))
sqrt 16      -- apply sqrt to 16     (maths: sqrt(16))

-- Function application has the HIGHEST precedence:
f 3 + 2      -- means (f 3) + 2, NOT f (3 + 2)
-- If you want f applied to (3+2), use parentheses: f (3 + 2)

Function Composition

Function composition combines two functions to create a new function where the output of one function becomes the input of the next. In Haskell, the composition operator is . (dot). Mathematically: (f . g)(x) = f(g(x)) — g is applied first, then f. 
-- Compose two functions: first negate, then double
doubleNeg = (*2) . negate
doubleNeg 5    -- negate 5 = -5, then (*2) (-5) = -10

-- Check if a list is non-empty
notNull = not . null
notNull []     -- null [] = True, not True = False
notNull [1,2]  -- null [1,2] = False, not False = True

-- Pipeline of transformations:
process = filter even . map (*3)
process [1, 2, 3, 4]
  -- Step 1: map (*3) [1,2,3,4] = [3, 6, 9, 12]
  -- Step 2: filter even [3, 6, 9, 12] = [6, 12]
  -- Result: [6, 12]

-- Composition chains read right-to-left:
transform = show . (+1) . (*2)
transform 5
  -- (*2) 5 = 10
  -- (+1) 10 = 11
  -- show 11 = "11"

Function composition allows complex operations to be built from simple, reusable functions. Each function does one thing well; composition chains them together. This is a key benefit of functional programming — high modularity and readability.

4.12.5 Partial Application and Currying

Partial Application

Partial application is the process of fixing some (but not all) of a function's arguments, producing a new, more specialised function that takes the remaining arguments. 
-- add takes two arguments
add x y = x + y

-- Partially apply: fix the first argument to 5
addFive = add 5            -- addFive is now a function waiting for one more argument
addFive 3      -- Result: 8   (add 5 3)
addFive 10     -- Result: 15  (add 5 10)

-- More examples:
multiply x y = x * y
double = multiply 2        -- partially applied: first arg fixed to 2
triple = multiply 3        -- partially applied: first arg fixed to 3
double 7       -- 14
triple 7       -- 21

-- Powerful with higher-order functions:
map (add 10) [1, 2, 3]       -- Result: [11, 12, 13]
map (multiply 3) [1, 2, 3]   -- Result: [3, 6, 9]
filter (> 5) [3, 7, 2, 9, 1] -- Result: [7, 9]  (> is partially applied with 5)

Currying

Currying is the technique of transforming a function that takes multiple arguments into a sequence of functions, each taking a single argument. In Haskell, all functions are curried by default — a "multi-argument function" is actually a chain of single-argument functions. 
-- These definitions are equivalent:
add x y = x + y
add = \x -> (\y -> x + y)     -- explicit curried form

-- Type signature reveals the currying:
add :: Int -> Int -> Int
-- Read as: add takes an Int, returns a FUNCTION (Int -> Int)
-- The arrows associate to the right: Int -> (Int -> Int)

-- Step-by-step evaluation of add 3 5:
add 3 5
  = (\x -> (\y -> x + y)) 3 5   -- substitute x = 3
  = (\y -> 3 + y) 5              -- now a function waiting for y; substitute y = 5
  = 3 + 5
  = 8

-- This is why partial application works naturally:
add 3        -- returns the function (\y -> 3 + y)
(add 3) 5    -- applies that function to 5, giving 8
The distinction: currying is the underlying technique of decomposing a multi-argument function into a chain of single-argument functions. Partial application is the practical act of supplying fewer arguments than expected, producing a specialised function. Currying enables partial application. In the exam, be able to explain both and show how partial application works with map/filter.

4.12.6 Immutable Data and Referential Transparency

Immutable Data Structures

Immutability means that once a data value is created, it cannot be modified. Any operation that would "change" the data instead produces a completely new data structure, leaving the original unchanged. 
-- In imperative programming (mutation):
x = 5
x = x + 1    -- x is MUTATED to 6; the old value 5 is lost

-- In functional programming (immutability):
-- x = 5 is a PERMANENT BINDING. x is always 5 in this scope.
-- To get 6, create a NEW binding:
y = x + 1    -- y = 6, but x is STILL 5. Both values coexist.

Benefits of immutability:

  • No unexpected changes: Shared data cannot be modified by one part of the program without other parts knowing — eliminates a large class of bugs related to unintended state changes.
  • Easier reasoning: You can understand each part of the program in isolation, because values never change. The value of a variable is the same at the beginning, middle, and end of a function.
  • Naturally thread-safe: Since data cannot be modified, multiple threads can read the same data concurrently without locks, mutexes, or synchronisation. No race conditions are possible.
  • Easier testing and debugging: Pure functions with immutable data always return the same result for the same input, making unit tests simple and deterministic.
  • Enables optimisations: The compiler can cache results, reorder computations, and parallelise execution safely because it knows values will not change.

Side Effects and Pure Functions

A side effect is any observable change that a function makes to state outside its own scope: modifying a global variable, writing to a file, printing to screen, changing a database, sending a network request, or modifying a mutable data structure.

A pure function has two properties:
  • It always returns the same output for the same input (deterministic).
  • It has no side effects — it does not change anything outside itself.
-- Pure function:
square x = x * x
-- square 4 will ALWAYS return 16, no matter when, where, or how many times called.
-- It reads no external state and changes no external state.

-- Impure function (imperative pseudocode):
counter = 0
increment() =
    counter = counter + 1    -- side effect: modifies external variable
    return counter
-- Returns 1, then 2, then 3... Different result each time! Not pure.

Referential Transparency

An expression is referentially transparent if it can be replaced with its value without changing the behaviour of the program. This is a direct consequence of using pure functions and immutable data. 
-- If double x = x * 2, then:
-- Anywhere we see "double 5", we can replace it with "10" and the
-- program behaves identically. This is referential transparency.

-- Benefits for the compiler:
-- * Reorder computations safely (no dependencies on execution order)
-- * Cache results (memoisation) — if double 5 = 10, no need to recompute
-- * Parallelise execution — independent pure expressions can run on different cores
-- * Perform algebraic simplification and inlining

4.12.7 Lists in Functional Programming

The list is the fundamental data structure in functional programming. In Haskell, a list is defined recursively as either the empty list [] or an element cons'd onto another list using :.

-- These are equivalent representations:
[1, 2, 3]  =  1 : [2, 3]  =  1 : 2 : [3]  =  1 : 2 : 3 : []

-- A list is ALWAYS either:
-- 1. The empty list []     (base case)
-- 2. x : xs                (an element x followed by a list xs)

List Processing with Recursion

In functional programming, recursion replaces iteration. Processing a list follows a standard pattern: a base case for the empty list, and a recursive case that processes the head and recurses on the tail.

-- Sum all elements
mySum []     = 0
mySum (x:xs) = x + mySum xs

-- Trace: mySum [3, 5, 2]
--   = 3 + mySum [5, 2]
--   = 3 + (5 + mySum [2])
--   = 3 + (5 + (2 + mySum []))
--   = 3 + (5 + (2 + 0))
--   = 3 + (5 + 2) = 3 + 7 = 10

-- Calculate length
myLength []     = 0
myLength (_:xs) = 1 + myLength xs    -- _ means "ignore the head value"

-- Reverse a list
myReverse []     = []
myReverse (x:xs) = myReverse xs ++ [x]

-- Trace: myReverse [1, 2, 3]
--   = myReverse [2, 3] ++ [1]
--   = (myReverse [3] ++ [2]) ++ [1]
--   = ((myReverse [] ++ [3]) ++ [2]) ++ [1]
--   = (([] ++ [3]) ++ [2]) ++ [1]
--   = ([3] ++ [2]) ++ [1]
--   = [3, 2] ++ [1]
--   = [3, 2, 1]

-- Check membership
myElem _ []     = False
myElem e (x:xs)
    | e == x    = True
    | otherwise  = myElem e xs

-- Implement map using recursion
myMap f []     = []
myMap f (x:xs) = f x : myMap f xs
List recursion questions are very common and carry multiple marks. Always define the base case (usually the empty list []) and the recursive case (process the head x, recurse on the tail xs). Trace through each step showing the substitution clearly. A missing base case is the most common error — without it, the recursion never terminates.

4.12.9 Advantages and Disadvantages

AdvantagesDisadvantages
Easier to reason about and prove correctness — pure functions have predictable, deterministic behaviour. Mathematical proofs of correctness are possible.Steeper learning curve for programmers accustomed to imperative thinking. Concepts like monads, currying, and recursion can be unintuitive at first.
Naturally suited to parallel and concurrent programming — no mutable shared state means no race conditions, no deadlocks, no need for locks. Functions can be evaluated on different cores safely.Less efficient for some problems where in-place mutation is natural — e.g., modifying one element of a large array is O(1) imperatively but O(n) when creating a new immutable array.
More concise and expressive code — higher-order functions, composition, and pattern matching reduce boilerplate. map/filter/fold replace dozens of lines of loop code.Recursive solutions can use more stack space and may cause stack overflow for very deep recursion (mitigated by tail call optimisation in some languages).
Easier to test — pure functions can be tested in complete isolation without setting up state, mocking dependencies, or worrying about test order.I/O and real-world interaction inherently involve side effects. Functional languages handle this with special mechanisms (monads in Haskell, effects systems) which add complexity.
Referential transparency enables powerful compiler optimisations: memoisation (caching results), lazy evaluation (computing values only when needed), safe reordering, and automatic parallelisation.Smaller ecosystem and fewer libraries compared to mainstream imperative languages for some application domains (though this is improving).
No side effects means fewer bugs related to state — a major category of real-world software defects is eliminated.Performance-critical code (e.g., game engines, operating system kernels) often needs fine-grained control over memory layout and mutation, which is more natural in imperative languages.
