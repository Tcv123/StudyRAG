4.12.1 The Functional Programming Paradigm

Functional programming is a programming paradigm in which computation is treated as the evaluation of mathematical functions. Programs are constructed by applying and composing functions, rather than by executing a sequence of statements that change the program's state.

Key principles that distinguish functional programming from imperative programming:

  • Functions are first-class objects — they can be assigned to variables, passed as arguments and returned from other functions.
  • Programs avoid side effects — functions do not modify external state.
  • Data is immutable — once created, data cannot be changed.
  • Functions are pure — the same inputs always produce the same output.
  • Computation is achieved through function application and composition rather than through variable assignment and loops.

4.12.2 First-Class Objects

In functional programming, functions are first-class objects (also called first-class citizens). This means functions have the same status as any other data value. Specifically, a function can be:
  • Assigned to a variable or stored in a data structure.
  • Passed as an argument to another function.
  • Returned as the result of another function.
  • Created dynamically (at runtime).

Example in Haskell-style notation:

-- Assign a function to a name
double = \x -> x * 2

-- Pass a function as an argument
applyTwice f x = f (f x)
applyTwice double 3    -- Result: 12  (double(double(3)) = double(6) = 12)

-- Return a function from a function
makeAdder n = \x -> x + n
addFive = makeAdder 5
addFive 10              -- Result: 15

4.12.3 Higher-Order Functions

A higher-order function is a function that either:
  • Takes one or more functions as arguments, or
  • Returns a function as its result.
Higher-order functions are only possible because functions are first-class objects.

map

map applies a given function to every element of a list, returning a new list of results.

map f []     = []
map f (x:xs) = f x : map f xs

-- Examples:
map (*2) [1, 2, 3, 4]       -- Result: [2, 4, 6, 8]
map even [1, 2, 3, 4]       -- Result: [False, True, False, True]
map toUpper "hello"          -- Result: "HELLO"

filter

filter takes a predicate (a function that returns True/False) and a list, returning a new list containing only the elements for which the predicate returns True.

filter p []     = []
filter p (x:xs)
    | p x       = x : filter p xs
    | otherwise  = filter p xs

-- Examples:
filter even [1, 2, 3, 4, 5, 6]     -- Result: [2, 4, 6]
filter (>3) [1, 2, 3, 4, 5]        -- Result: [4, 5]
filter (/= ' ') "h e l l o"        -- Result: "hello"

fold (reduce)

fold (called reduce in some languages) combines all elements of a list into a single value by repeatedly applying a binary function. There are two variants:

foldl (fold left) — processes from left to right:

foldl f acc []     = acc
foldl f acc (x:xs) = foldl f (f acc x) xs

-- Example: sum a list
foldl (+) 0 [1, 2, 3, 4]
  = foldl (+) (0+1) [2, 3, 4]
  = foldl (+) (1+2) [3, 4]
  = foldl (+) (3+3) [4]
  = foldl (+) (6+4) []
  = 10

foldr (fold right) — processes from right to left:

foldr f acc []     = acc
foldr f acc (x:xs) = f x (foldr f acc xs)

-- Example: concatenate strings
foldr (++) "" ["Hello", " ", "World"]   -- Result: "Hello World"
You must be able to trace through map, filter and fold with specific inputs. Practise step-by-step evaluation. In exam answers, show each step clearly. Remember: map transforms each element, filter selects elements, fold combines elements into a single value.

4.12.4 Function Application and Composition

Function Application

Function application is the process of applying a function to its arguments to produce a result. In Haskell notation, function application is written by juxtaposition (placing the function name next to its argument, separated by a space) rather than using parentheses. 
-- Function application in Haskell:
f x          -- apply f to x
add 3 5      -- apply add to 3 and 5 (result: 8)

-- Compared to mathematical notation:
-- f(x)
-- add(3, 5)

Function Composition

Function composition combines two functions to create a new function. The output of one function becomes the input of the next. In Haskell, the composition operator is . (dot). Mathematically: (f . g)(x) = f(g(x)). 
-- Compose two functions: first negate, then double
-- (Note: g is applied first, then f)
doubleNeg = (*2) . negate
doubleNeg 5    -- negate 5 = -5, then (*2) (-5) = -10

-- More examples:
notNull = not . null
notNull []     -- null [] = True, not True = False
notNull [1,2]  -- null [1,2] = False, not False = True

-- Composition chains:
process = filter even . map (*3)
process [1, 2, 3, 4]
  -- map (*3) [1,2,3,4] = [3, 6, 9, 12]
  -- filter even [3, 6, 9, 12] = [6, 12]

Function composition allows complex operations to be built from simple, reusable functions, improving readability and modularity.

4.12.5 Partial Application and Currying

Partial Application

Partial application is the process of fixing some of a function's arguments, producing a new function that takes the remaining arguments. This allows general functions to be specialised. 
-- add takes two arguments
add x y = x + y

-- Partially apply add by fixing the first argument to 5
addFive = add 5

addFive 3      -- Result: 8   (equivalent to add 5 3)
addFive 10     -- Result: 15  (equivalent to add 5 10)

-- More examples:
multiply x y = x * y
double = multiply 2
triple = multiply 3

double 7       -- Result: 14
triple 7       -- Result: 21

-- Using partial application with map:
map (add 10) [1, 2, 3]    -- Result: [11, 12, 13]
map (multiply 3) [1, 2, 3] -- Result: [3, 6, 9]

Currying

Currying is the technique of transforming a function that takes multiple arguments into a sequence of functions, each taking a single argument. In Haskell, all functions are curried by default.

A function add x y = x + y is actually a function that takes x and returns a new function that takes y:

-- These are equivalent:
add x y = x + y
add = \x -> (\y -> x + y)

-- Type signature shows currying clearly:
add :: Int -> Int -> Int
-- Read as: add takes an Int and returns a function (Int -> Int)

-- Step-by-step:
add 3 5
  = (\x -> (\y -> x + y)) 3 5
  = (\y -> 3 + y) 5
  = 3 + 5
  = 8

Currying is what makes partial application possible. Because add is really a chain of single-argument functions, providing just the first argument naturally produces a new function waiting for the second.

The distinction between currying and partial application: currying is the technique of decomposing a multi-argument function into a chain of single-argument functions. Partial application is the act of supplying fewer arguments than expected, producing a specialised function. Currying enables partial application.

4.12.6 Immutable Data and Referential Transparency

Immutable Data Structures

Immutability means that once a data value is created, it cannot be modified. Any "change" produces a completely new data structure, leaving the original unchanged. 
-- In imperative programming:
x = 5
x = x + 1    -- x is mutated to 6

-- In functional programming:
-- x = 5 is a permanent binding. x is always 5.
-- To get 6, you create a new value:
y = x + 1    -- y = 6, but x is still 5

Benefits of immutability:

  • No unexpected changes to shared data — eliminates a large class of bugs.
  • Simplifies reasoning about programs — values never change, so you can understand each part in isolation.
  • Naturally thread-safe — since data cannot be modified, concurrent access requires no locks.
  • Easier testing and debugging — functions always return the same result for the same input.

Side Effects and Pure Functions

A side effect is any observable change that a function makes to state outside its own scope: modifying a global variable, writing to a file, printing to screen, changing a mutable data structure, or sending a network request.

A pure function has two properties:
  • It always returns the same output for the same input (deterministic).
  • It has no side effects.
-- Pure function:
square x = x * x
-- square 4 will ALWAYS return 16, no matter when or where it is called.

-- Impure function (pseudocode, imperative style):
counter = 0
increment() = counter = counter + 1; return counter
-- Returns different values each time it is called (1, 2, 3, ...)

Referential Transparency

An expression is referentially transparent if it can be replaced with its value without changing the behaviour of the program. This is a direct consequence of using pure functions and immutable data. 
-- If double x = x * 2, then:
-- double 5 can always be replaced with 10 anywhere in the program.

-- This allows the compiler to:
--   * Reorder computations safely
--   * Cache results (memoisation)
--   * Parallelise execution
--   * Perform algebraic simplification

4.12.7 Lists in Functional Programming

The list is the fundamental data structure in functional programming. In Haskell, a list is defined recursively as either an empty list or an element followed by another list.

A list can be decomposed using pattern matching with the cons operator:

-- Any non-empty list [1, 2, 3] can be written as 1 : [2, 3]
-- In general: (x:xs) where x is the head and xs is the tail

-- Pattern matching in a function definition:
myLength []     = 0
myLength (x:xs) = 1 + myLength xs

myLength [4, 7, 2]
  = 1 + myLength [7, 2]
  = 1 + 1 + myLength [2]
  = 1 + 1 + 1 + myLength []
  = 1 + 1 + 1 + 0
  = 3

List Operations and Recursion

In functional programming, recursion replaces iteration (loops). Processing a list typically involves a base case (empty list) and a recursive case (process the head, recurse on the tail).

-- Sum all elements
mySum []     = 0
mySum (x:xs) = x + mySum xs

mySum [3, 5, 2]
  = 3 + mySum [5, 2]
  = 3 + 5 + mySum [2]
  = 3 + 5 + 2 + mySum []
  = 3 + 5 + 2 + 0
  = 10

-- Reverse a list
myReverse []     = []
myReverse (x:xs) = myReverse xs ++ [x]

myReverse [1, 2, 3]
  = myReverse [2, 3] ++ [1]
  = (myReverse [3] ++ [2]) ++ [1]
  = ((myReverse [] ++ [3]) ++ [2]) ++ [1]
  = (([] ++ [3]) ++ [2]) ++ [1]
  = [3, 2, 1]

-- Check if an element is in a list
myElem _ []     = False
myElem e (x:xs)
    | e == x    = True
    | otherwise  = myElem e xs
List recursion questions are very common. Always define the base case (usually the empty list []) and the recursive case (process x, recurse on xs). Trace through each step carefully to show the examiner your working.

4.12.9 Advantages and Disadvantages

AdvantagesDisadvantages
Easier to reason about and prove correctness — pure functions have predictable behaviour.Steeper learning curve for programmers used to imperative thinking.
Naturally suited to parallel and concurrent programming — no mutable shared state means no race conditions.Can be less efficient for some problems (e.g., in-place array mutation is O(1) imperatively but O(n) with immutable structures).
More concise and expressive code — higher-order functions and composition reduce boilerplate.Recursive solutions can use more stack space and may be harder to debug for beginners.
Easier to test — pure functions can be tested in isolation without setting up state.I/O and real-world interaction (which inherently involve side effects) require special handling (e.g., monads in Haskell).
Referential transparency enables powerful compiler optimisations (memoisation, lazy evaluation, reordering).Smaller ecosystem and fewer libraries compared to mainstream imperative languages for some domains.
