4.9.1 Communication Methods

Serial vs Parallel Transmission

FeatureSerialParallel
How it worksBits sent one at a time along a single wire/channel, one after anotherMultiple bits sent simultaneously along multiple wires (e.g., 8 wires for 8 bits at once)
Speed (short distance)Slower (one bit per cycle)Faster (multiple bits per cycle)
Speed (long distance)Faster (no skew problems)Slower (skew limits practical speed)
CostCheaper (fewer wires, simpler connectors)More expensive (many wires, wider cables)
SkewNo skew (single channel)Bits can arrive at different times (skew) because wires have slightly different electrical properties, causing errors at longer distances
CrosstalkMinimalElectromagnetic interference between adjacent wires (crosstalk) corrupts signals
Use casesUSB, SATA, Ethernet, internet links, long-distance communicationInternal CPU-to-memory bus, old printer cables (IEEE 1284, now obsolete for external use)
Skew: In parallel transmission, the phenomenon where bits sent simultaneously arrive at slightly different times due to differences in wire lengths, impedance, or electrical properties, potentially corrupting the data. Limits the maximum usable cable length and clock speed.
Crosstalk: Electromagnetic interference between adjacent parallel wires where the signal in one wire induces a current in a neighbouring wire, corrupting data. Worse at higher frequencies and longer distances.

Synchronous vs Asynchronous Transmission

FeatureSynchronousAsynchronous
TimingData sent in a continuous stream synchronised by a shared clock signal between sender and receiverData sent one character/byte at a time without a shared clock. Each character is framed with start bit (0) and stop bit(s) (1).
ClockBoth sender and receiver share a clock signal (embedded in the data or on a separate wire)No shared clock. The start bit signals "data is coming" so the receiver can synchronise for that byte.
EfficiencyMore efficient for large data transfers (minimal framing overhead)Less efficient (at least 2 extra bits per byte for start/stop = ~20% overhead)
SpeedGenerally faster (continuous stream)Generally slower (gaps between characters)
Use casesHigh-speed network links, video streaming, VoIP, SPI busKeyboard input, RS-232 serial ports, UART, modems, Bluetooth

Bit Rate, Baud Rate, Bandwidth and Latency

Bit rate: The number of bits transmitted per second (bps). This is the actual data throughput.
Baud rate: The number of signal changes (symbols) per second. Each signal change can encode one or more bits.
Bandwidth: The range of frequencies available for data transmission on a medium. Higher bandwidth allows a higher data rate. Measured in Hz but often colloquially used to mean data rate.
Latency: The time delay between sending data and it arriving at the destination. Affected by distance, number of hops (routers), congestion, and processing time.

Relationship between bit rate and baud rate:

Bit rate = Baud rate x Number of bits per signal change (bits per symbol)

Example 1: If baud rate = 1000 and each signal encodes 1 bit:
  Bit rate = 1000 x 1 = 1000 bps

Example 2: If baud rate = 1000 and each signal encodes 4 bits
  (using 16 different voltage/phase levels):
  Bit rate = 1000 x 4 = 4000 bps

Example 3: If baud rate = 2400 and each signal encodes 2 bits:
  Bit rate = 2400 x 2 = 4800 bps

Modern modems and network technologies use techniques like QAM (Quadrature Amplitude Modulation) to encode many bits per symbol. 256-QAM encodes 8 bits per symbol; 1024-QAM encodes 10 bits per symbol.

Bit rate and baud rate are only equal when each signal change encodes exactly one bit. In most modern systems, multiple bits are encoded per symbol, so the bit rate exceeds the baud rate. Questions often ask you to calculate one from the other.

4.9.2 Network Topologies

Topology: The arrangement of devices (nodes) and connections (links) in a network. Physical topology describes the physical cable layout. Logical topology describes how data flows through the network (which can differ from the physical layout).
TopologyDescriptionAdvantagesDisadvantages
Star All devices connect to a central switch or hub via individual cables. The most common LAN topology. Single cable failure only affects one device; easy to add/remove devices; easy to troubleshoot; high performance (no collisions with a switch); centralised management. Central switch is a single point of failure (if it fails, the entire network goes down); requires more cabling than bus; switch hardware can be expensive for enterprise-grade models.
Bus All devices connect to a single shared backbone cable with terminators at each end to prevent signal reflection. Cheap and simple to set up; requires minimal cabling; suitable for small, temporary networks. Backbone failure brings down the entire network; performance degrades with more devices (collisions increase); difficult to troubleshoot; limited cable length and device count; all data visible to all devices (security risk); largely obsolete.
Mesh Full mesh: every device connects to every other device. Partial mesh: some devices have redundant connections, but not all-to-all. Highly redundant — no single point of failure; data can take multiple routes (fault tolerance); reliable; if one link fails, traffic is rerouted. Very expensive (many cables and ports — full mesh of n devices needs n(n-1)/2 connections); complex to set up and manage; impractical for large networks. The internet is a partial mesh.
Hybrid A combination of two or more topologies (e.g., star-bus where multiple star networks are connected by a bus backbone, or star-mesh in enterprise data centres). Flexible and scalable; can be tailored to specific needs; combines advantages of constituent topologies. Complex to design, implement, and manage; can be expensive; fault diagnosis may be more difficult.

4.9.6 The Internet

The Internet: A global network of interconnected networks that communicate using the TCP/IP protocol suite. It is the physical infrastructure of cables, routers, switches, and servers.
The World Wide Web (WWW): A collection of web pages, documents, and resources linked by hyperlinks, accessed via the internet using HTTP/HTTPS and rendered by web browsers. The web is a service that runs on top of the internet — they are not the same thing. Other internet services include email (SMTP/IMAP), file transfer (FTP), DNS, VoIP, and streaming.

Internet Structure

  • The internet is a hierarchical network of networks connected by high-capacity backbone links (fibre optic cables, including transoceanic undersea cables carrying nearly all international internet traffic)
  • Internet Exchange Points (IXPs) are physical locations where different ISPs and networks interconnect and exchange traffic directly (peering), reducing latency and cost
  • Internet Service Providers (ISPs) provide end users with internet access. ISPs operate in a tiered hierarchy:
    • Tier 1: Global backbone providers (can reach any network without paying for transit)
    • Tier 2: Regional ISPs that peer with some networks and pay Tier 1 for global reach
    • Tier 3: Local ISPs that connect end users and buy transit from Tier 1/2
  • DNS servers translate human-readable domain names to IP addresses
  • Web servers host websites and respond to HTTP/HTTPS requests from clients

Packet Switching vs Circuit Switching

4.9.7 TCP/IP Protocol Stack

Protocol: A set of rules that govern how data is formatted, transmitted, received, and acknowledged across a network. Protocols ensure that devices from different manufacturers, running different operating systems, can communicate reliably. They define message format, timing, sequencing, and error handling.

The TCP/IP model has four layers. Data passes down through the layers on the sender's side (encapsulation — each layer adds its own header) and up through the layers on the receiver's side (de-encapsulation — each layer strips its header).

LayerNameFunctionKey ProtocolsData Unit
4 (Top)ApplicationProvides network services directly to user applications. Encodes data in the format required by the application protocol.HTTP, HTTPS, FTP, SMTP, IMAP, POP3, SSH, DNS, DHCP, TelnetMessage / Data
3TransportProvides end-to-end communication between applications on different hosts. Segments data, provides error checking, flow control, and (with TCP) reliable ordered delivery. Uses port numbers to identify specific applications.TCP, UDPSegment (TCP) / Datagram (UDP)
2Internet (Network)Handles logical addressing (IP addresses) and routing. Packets are independently routed across networks towards the destination. Each router examines the destination IP and forwards accordingly.IP (IPv4, IPv6), ARP, ICMPPacket
1 (Bottom)Link (Network Interface)Handles physical transmission of data frames over the network medium. Uses MAC addresses for local (hop-to-hop) delivery. Manages access to the physical medium.Ethernet, Wi-Fi (802.11), PPPFrame

Why use layers?

  • Modularity: Each layer can be developed, tested, and updated independently without affecting other layers.
  • Interoperability: Different vendors can implement the same layer to the same standard.
  • Abstraction: Each layer only needs to know about the layer directly above and below it.
  • Standardisation: Agreed standards ensure universal communication.

TCP vs UDP

FeatureTCP (Transmission Control Protocol)UDP (User Datagram Protocol)
ConnectionConnection-oriented (three-way handshake: SYN → SYN-ACK → ACK establishes session before data transfer)Connectionless (no handshake; just sends data immediately)
ReliabilityReliable — guarantees delivery, correct ordering, and error checking. Uses acknowledgements and retransmits lost packets.Unreliable — no guarantee of delivery, ordering, or retransmission. "Best effort" delivery.
SpeedSlower (overhead from handshake, acknowledgements, retransmission, flow control)Faster (minimal overhead — just header + data, no waiting for ACKs)
Flow controlYes — windowing mechanism adjusts transmission rate to prevent overwhelming the receiverNo — sender transmits at whatever rate it chooses
OrderingYes — sequence numbers ensure correct order at receiverNo — packets may arrive in any order
Use casesWeb browsing (HTTP/HTTPS), email (SMTP/IMAP), file transfer (FTP), SSH — where data integrity is criticalLive streaming, VoIP, online gaming, DNS queries, video conferencing — where speed matters more than perfection (a dropped packet = minor glitch, not a problem)
Header size20-60 bytes (larger overhead)8 bytes (minimal overhead)

The OSI (Open Systems Interconnection) model is a seven-layer reference model for understanding network communication. It is more detailed than TCP/IP and used primarily for teaching and design, while TCP/IP is the practical implementation.

Know how the OSI model maps to the TCP/IP model — the AQA spec requires understanding of both. The OSI Application, Presentation, and Session layers all map to the TCP/IP Application layer. The OSI Data Link and Physical layers both map to the TCP/IP Link layer. When answering questions about protocols, state which layer each protocol operates at.

DNS Resolution Process (How a URL Becomes an IP Address)

  1. User types www.example.com in their browser.
  2. Browser checks its local cache for a recent lookup. If found, uses it.
  3. If not cached, the OS checks the host file and its own DNS cache.
  4. If still not resolved, the query is sent to the recursive DNS resolver (usually provided by the ISP or a service like Google DNS 8.8.8.8).
  5. The resolver queries the root DNS server, which directs it to the TLD (Top-Level Domain) server for .com.
  6. The TLD server directs to the authoritative DNS server for example.com.
  7. The authoritative server returns the IP address (e.g., 93.184.216.34).
  8. The resolver caches the result (with a TTL) and returns it to the browser.
  9. The browser connects to the web server at that IP address.

Public vs Private IP Addresses

  • Public IP addresses are globally unique, routable on the internet, and assigned by ISPs. Every device directly reachable on the internet needs one.
  • Private IP addresses are used within local networks and are NOT routable on the internet. Multiple separate LANs can reuse the same private ranges without conflict. Standard private ranges:
    • 10.0.0.0 – 10.255.255.255 (Class A — 16.7 million addresses)
    • 172.16.0.0 – 172.31.255.255 (Class B — 1 million addresses)
    • 192.168.0.0 – 192.168.255.255 (Class C — 65,536 addresses)

NAT (Network Address Translation)

NAT: A technique used by routers to translate private IP addresses on a local network to the router's single public IP address (and vice versa) when communicating with the internet. This allows many devices to share one public IP and helps conserve the limited IPv4 address space.

The router maintains a NAT table that maps each internal device's private IP:port combination to a unique port on the external public IP. When a response arrives from the internet, the router looks up the destination port in the NAT table and forwards the packet to the correct internal device.

  Example NAT Table:
  Internal IP:Port       External IP:Port       Destination
  192.168.1.5:52431  →  86.134.10.20:52431  →  google.com:443
  192.168.1.8:49872  →  86.134.10.20:49872  →  bbc.co.uk:80
  192.168.1.12:61000 →  86.134.10.20:61000  →  youtube.com:443

Subnet Masking

A subnet mask divides an IP address into a network part (identifying the network/subnet) and a host part (identifying the specific device on that network).

IP Address:   192.168.1.100
Subnet Mask:  255.255.255.0

Binary mask: 11111111.11111111.11111111.00000000
Network Part: 192.168.1     (where mask bits are 1 — identifies the network)
Host Part:    .100           (where mask bits are 0 — identifies the device)

All devices with IPs 192.168.1.x (where x = 1 to 254) are on the same subnet.
.0 is the network address (cannot be assigned to a device).
.255 is the broadcast address (sends to all devices on the subnet).

CIDR notation: 192.168.1.0/24  (the /24 means the first 24 bits are the network part)

Subnetting divides a large network into smaller, more manageable sub-networks. Benefits: improved security (traffic isolated between subnets), reduced congestion (broadcast traffic contained within each subnet), and easier administration.

4.9.11 Firewalls, Proxy Servers, and Encryption

Firewalls

A firewall filters network traffic based on a set of security rules, acting as a barrier between trusted (internal) and untrusted (external) networks.

  • Packet filtering: Inspects packet headers (source/destination IP, port numbers, protocol) and allows or denies based on rules. Fast but cannot inspect packet contents.
  • Stateful inspection: Tracks the state of active connections (established, new, related) and makes decisions based on the context of traffic, not just individual packets. More intelligent than simple packet filtering.
  • Application-layer filtering (proxy firewall): Inspects the actual content of packets at the application level (deep packet inspection). Can block specific URLs, file types, or malware signatures. Slowest but most thorough.

Proxy Servers

Proxy Server: An intermediary server that sits between client devices and the internet. Client requests go to the proxy first; the proxy forwards them to the destination on the client's behalf, then returns the response.

Functions:

  • Anonymity: Hides the client's real IP address from the destination server
  • Caching: Stores frequently accessed web content locally; subsequent requests are served from cache without contacting the origin server (faster response, less bandwidth)
  • Content filtering: Blocks access to specific websites or content categories (used by schools and businesses to enforce acceptable use policies)
  • Logging: Records all web activity for monitoring, auditing, and compliance
  • Security: Can scan content for malware before it reaches the internal network
  • Load balancing: Distributes requests across multiple servers (reverse proxy)

Encryption — TLS (Transport Layer Security)

TLS (Transport Layer Security) is a cryptographic protocol that provides secure, encrypted communication over a network. It is the successor to SSL and is used by HTTPS. It provides confidentiality (encryption), integrity (detecting tampering), and authentication (verifying server identity).

TLS Handshake (simplified):

  1. Client Hello: Client sends supported TLS versions, cipher suites, and a random number.
  2. Server Hello: Server responds with chosen TLS version, cipher suite, its digital certificate (containing its public key, signed by a Certificate Authority), and a random number.
  3. Certificate Verification: Client verifies the server's certificate against its list of trusted Certificate Authorities (CAs). This confirms the server is who it claims to be.
  4. Key Exchange: Client generates a pre-master secret, encrypts it with the server's public key (asymmetric encryption), and sends it. Only the server can decrypt it with its private key.
  5. Session Key Derivation: Both client and server use the pre-master secret and the random numbers to independently compute the same symmetric session key.
  6. Secure Communication: All subsequent data is encrypted with the symmetric session key. Symmetric encryption is used because it is much faster than asymmetric for bulk data transfer.

This hybrid approach uses asymmetric encryption (slow but solves key distribution) for the initial key exchange, then symmetric encryption (fast) for the actual data transfer.

4.9.12 Socket Programming Concepts

Socket: An endpoint for communication between two processes over a network. A socket is uniquely identified by the combination of an IP address and a port number (e.g., 192.168.1.5:8080). Sockets provide the interface between the application layer and the transport layer.

Key Concepts

  • Port number: A 16-bit number (0–65535) that identifies a specific application or service on a device. Well-known ports (0-1023): 80 (HTTP), 443 (HTTPS), 25 (SMTP), 22 (SSH), 53 (DNS). Ephemeral ports (49152-65535) are dynamically assigned to client applications.
  • Server socket: Binds to a specific port and listens for incoming connections.
  • Client socket: Initiates a connection to a server's IP address and port.

TCP Socket Communication Flow

  1. Server: Creates a socket, binds it to an IP:port, and starts listening.
  2. Client: Creates a socket and connects to the server's IP:port.
  3. Server: Accepts the connection, creating a new socket for this specific client.
  4. Both: Exchange data by reading from and writing to their sockets.
  5. Both: Close their sockets when communication is complete.
# Python TCP socket example — Server
import socket

server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)  # TCP socket
server.bind(('0.0.0.0', 8080))    # Bind to all interfaces, port 8080
server.listen(5)                    # Queue up to 5 pending connections
print("Server listening on port 8080...")

client_socket, address = server.accept()   # Block until a client connects
print(f"Connection from {address}")
data = client_socket.recv(1024)            # Receive up to 1024 bytes
print(f"Received: {data.decode()}")
client_socket.send(b'Hello from server')   # Send response
client_socket.close()                      # Close this client connection
server.close()                             # Close the server socket
# Python TCP socket example — Client
import socket

client = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
client.connect(('192.168.1.5', 8080))      # Connect to server
client.send(b'Hello from client')           # Send data
response = client.recv(1024)                # Receive response
print(f"Server replied: {response.decode()}")
client.close()                              # Close connection

A very common exam question: "Why is client-side validation alone insufficient?" Answer: Users can bypass client-side code by disabling JavaScript in their browser, or by crafting HTTP requests directly (e.g., using developer tools or command-line tools). Server-side validation is essential because the server is under the developer's control and cannot be bypassed by the user. Best practice is to validate on BOTH sides: client-side for user experience (immediate feedback), server-side for security.

4.9.14 Packet Structure

When data is transmitted across a packet-switched network, it is broken into packets. Each packet is routed independently and may take different paths to the destination. Each packet contains three parts:

ComponentContentsPurpose
Header
  • Source IP address
  • Destination IP address
  • Packet sequence number
  • Protocol identifier (TCP/UDP)
  • TTL (Time to Live)
  • Source and destination port numbers
  • Header checksum
  • Packet length
  • Flags (e.g., fragmentation flags)
 Provides all the routing and control information needed to deliver the packet to its destination and reassemble the original data correctly.
Payload (Data) A portion of the actual data being transmitted The useful content. Size is limited by the Maximum Transmission Unit (MTU) — typically 1500 bytes for Ethernet. Larger messages are split across multiple packets.
Trailer
  • Error detection code (CRC — Cyclic Redundancy Check)
  • End-of-packet marker
 Allows the receiver to detect transmission errors. If the CRC does not match, the packet is discarded and retransmission is requested (by TCP).

Key Header Fields Explained

  • TTL (Time to Live): A counter (typically starting at 64 or 128) that is decremented by 1 each time the packet passes through a router. When TTL reaches 0, the packet is discarded. This prevents packets from circling the network indefinitely if there is a routing loop. The traceroute command uses TTL to discover the path packets take.
  • Checksum: A value calculated from the packet's contents using an algorithm (e.g., CRC-32). The sender computes and includes the checksum. The receiver recalculates it and compares. If they differ, the packet has been corrupted during transmission and is discarded.
  • Sequence number: Because packets may take different routes and arrive out of order, the sequence number allows the receiving device to reassemble packets in the correct order. It also allows detection of missing packets (gaps in the sequence).

Complete Data Transmission Process

  1. Application generates data (e.g., an HTTP request).
  2. Transport layer (TCP) segments the data into chunks, adds port numbers and sequence numbers.
  3. Internet layer (IP) adds source and destination IP addresses, creating packets.
  4. Link layer adds source and destination MAC addresses, creating frames.
  5. Frames are transmitted as electrical/optical/radio signals over the physical medium.
  6. Each router along the path: receives the frame, strips the frame header, reads the IP destination, consults its routing table, adds new frame headers for the next hop, forwards the packet.
  7. At the destination: the link layer checks the CRC, strips the frame header. The IP layer verifies the checksum and passes to TCP. TCP reassembles packets in order, checks for missing packets (requests retransmission), and delivers the complete data to the application.
When describing data transmission, cover the full process: data is segmented into packets; each packet has a header (source/destination IP, sequence number, TTL, checksum), payload, and trailer (CRC). Packets may take different routes. At the destination, packets are reassembled using sequence numbers, checksums are verified, and missing/corrupted packets are retransmitted (TCP). Always mention specific header fields by name.
