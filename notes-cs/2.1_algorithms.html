2.1.1 Computational Thinking

Computational thinking is a problem-solving approach that involves three key techniques:

TechniqueDefinitionExample
DecompositionBreaking a complex problem into smaller, more manageable sub-problems that can be solved independently. Each sub-problem is simpler to understand, implement, and test.Building a game: decompose into sub-problems: graphics rendering, user input handling, scoring system, sound effects, level design, save/load functionality. Each can be developed and tested by a different team member.
AbstractionRemoving unnecessary detail and focusing only on the relevant information needed to solve the problem. Hides complexity behind a simpler representation.A road map abstracts away individual buildings, trees, and people — it shows only roads and junctions because that is all a driver needs. A weather app shows temperature and rain icons, abstracting away the complex atmospheric models behind them.
Algorithmic thinkingDesigning a precise, unambiguous, step-by-step solution that can be followed mechanically to solve the problem. The steps must be in the correct order and cover all cases (including edge cases).Writing exact instructions to find the largest number in a list: (1) set max to first element, (2) for each remaining element, if it is greater than max, update max, (3) return max.

Pseudocode Constructs (OCR Style)

// Selection (IF-ELSE)
if score >= 50 then
    print("Pass")
else
    print("Fail")
endif

// SWITCH / CASE (for multiple specific values)
switch grade:
    case 'A':
        print("Excellent")
    case 'B':
        print("Good")
    default:
        print("Other")
endswitch

// Count-controlled loop (FOR) — runs a known number of times
for i = 1 to 10
    print(i)
next i

// Condition-controlled loop (WHILE) — may run 0 or more times
while password != "abc"
    password = input("Enter password: ")
endwhile

// Post-condition loop (DO-UNTIL) — runs at least once
do
    x = input("Enter a positive number: ")
until x > 0

Trace Tables

A trace table tracks the values of variables at each step of an algorithm's execution. It is an essential debugging tool and is frequently required in exam questions.

// Algorithm:
x = 3
for i = 1 to 4
    x = x + i
next i
print(x)

// Trace table:
| Step | i | x  |
|------|---|----|
| Init | - | 3  |
| 1    | 1 | 4  |  (3 + 1)
| 2    | 2 | 6  |  (4 + 2)
| 3    | 3 | 9  |  (6 + 3)
| 4    | 4 | 13 |  (9 + 4)
Output: 13

2.1.3 Searching Algorithms

Linear Search

Linear search checks each item in the list one by one from the start until the target is found or the end of the list is reached. It works on both sorted and unsorted data. 
function linearSearch(list, target):
    for i = 0 to length(list) - 1
        if list[i] == target then
            return i           // found at index i
        endif
    next i
    return -1                  // not found

// Trace: searching for 25 in [3, 7, 12, 19, 25, 31, 44, 58]
// Check index 0: 3 != 25
// Check index 1: 7 != 25
// Check index 2: 12 != 25
// Check index 3: 19 != 25
// Check index 4: 25 == 25 → FOUND at index 4 (5 comparisons)
FeatureDetail
Data requirementWorks on unsorted AND sorted data
Best caseO(1) — target is the first element (1 comparison)
Worst caseO(n) — target is last or not present (n comparisons)
Average caseO(n/2) ≈ O(n)
AdvantagesSimple to implement; works on unsorted data; no pre-processing needed
DisadvantagesSlow for large datasets; checks every element in the worst case

Binary Search

Binary search repeatedly halves the search space by comparing the target to the middle element. If the target is smaller, search the left half; if larger, search the right half. The data must be sorted first. 
function binarySearch(list, target):
    low = 0
    high = length(list) - 1
    while low <= high
        mid = (low + high) DIV 2
        if list[mid] == target then
            return mid         // found
        elseif list[mid] < target then
            low = mid + 1      // search right half
        else
            high = mid - 1     // search left half
        endif
    endwhile
    return -1                  // not found

// Trace: searching for 25 in [3, 7, 12, 19, 25, 31, 44, 58]
// Step 1: low=0, high=7, mid=3, list[3]=19. 25 > 19 → low=4
// Step 2: low=4, high=7, mid=5, list[5]=31. 25 < 31 → high=4
// Step 3: low=4, high=4, mid=4, list[4]=25. 25 == 25 → FOUND (3 comparisons)
// Much faster than linear search's 5 comparisons!
FeatureDetail
Data requirementMust be sorted first (critical!)
Best caseO(1) — target is the exact middle element
Worst caseO(log n) — very efficient even for huge datasets
AdvantageExtremely fast for large datasets. Searching 1 billion items takes at most ~30 comparisons.
DisadvantageData must be sorted first (sorting takes O(n log n) time); more complex to implement
Binary search requires sorted data — always state this explicitly. For a list of 1,000,000 items: linear search needs up to 1,000,000 comparisons; binary search needs at most 20 (log2 of 1,000,000 ≈ 20). You do NOT need to memorise algorithm code — only be able to apply and trace algorithms on a given dataset.

2.1.4 Sorting Algorithms

Bubble Sort

Bubble sort repeatedly compares adjacent pairs of elements and swaps them if they are in the wrong order. After each complete pass through the list, the next largest unsorted element "bubbles up" to its correct position at the end. Passes repeat until no swaps are made in a complete pass (meaning the list is sorted). 
// Bubble sort trace: [5, 3, 8, 1, 9, 2]

Pass 1:
  Compare 5,3 → swap → [3, 5, 8, 1, 9, 2]
  Compare 5,8 → no swap
  Compare 8,1 → swap → [3, 5, 1, 8, 9, 2]
  Compare 8,9 → no swap
  Compare 9,2 → swap → [3, 5, 1, 8, 2, 9]   // 9 is now in final position

Pass 2:
  Compare 3,5 → no swap
  Compare 5,1 → swap → [3, 1, 5, 8, 2, 9]
  Compare 5,8 → no swap
  Compare 8,2 → swap → [3, 1, 5, 2, 8, 9]   // 8 is now in final position

Pass 3:
  Compare 3,1 → swap → [1, 3, 5, 2, 8, 9]
  Compare 3,5 → no swap
  Compare 5,2 → swap → [1, 3, 2, 5, 8, 9]   // 5 in position

Pass 4:
  Compare 1,3 → no swap
  Compare 3,2 → swap → [1, 2, 3, 5, 8, 9]   // 3 in position

Pass 5:
  No swaps made → list is sorted! STOP.
  Final: [1, 2, 3, 5, 8, 9]

Merge Sort

Merge sort uses a divide and conquer strategy: (1) recursively split the list in half until each sub-list has one element, (2) merge the sub-lists back together in sorted order. It is much more efficient than bubble sort for large datasets. 
// Merge sort trace: [5, 3, 8, 1, 9, 2]

Split phase (divide until single elements):
  [5, 3, 8, 1, 9, 2]
  [5, 3, 8]        [1, 9, 2]
  [5] [3, 8]       [1] [9, 2]
  [5] [3] [8]      [1] [9] [2]

Merge phase (merge pairs in sorted order):
  [3, 8]  (merge [3] and [8])        [2, 9]  (merge [9] and [2])
  [3, 5, 8]  (merge [5] and [3,8])   [1, 2, 9]  (merge [1] and [2,9])
  [1, 2, 3, 5, 8, 9]  (merge [3,5,8] and [1,2,9])

Merging [3,5,8] and [1,2,9]:
  Compare 3 and 1 → take 1 → [1]
  Compare 3 and 2 → take 2 → [1, 2]
  Compare 3 and 9 → take 3 → [1, 2, 3]
  Compare 5 and 9 → take 5 → [1, 2, 3, 5]
  Compare 8 and 9 → take 8 → [1, 2, 3, 5, 8]
  Take remaining 9 → [1, 2, 3, 5, 8, 9]

Insertion Sort

Insertion sort builds the sorted list one element at a time. It takes each element from the unsorted portion and inserts it into its correct position within the already-sorted portion, shifting elements as needed. 
// Insertion sort trace: [5, 3, 8, 1, 9, 2]

Start: sorted portion = [5], unsorted = [3, 8, 1, 9, 2]

Insert 3: 3 < 5, shift 5 right, insert 3 → [3, 5, | 8, 1, 9, 2]
Insert 8: 8 > 5, insert at end        → [3, 5, 8, | 1, 9, 2]
Insert 1: 1 < 3, shift 3,5,8 right    → [1, 3, 5, 8, | 9, 2]
Insert 9: 9 > 8, insert at end        → [1, 3, 5, 8, 9, | 2]
Insert 2: 2 > 1, 2 < 3, shift 3,5,8,9 → [1, 2, 3, 5, 8, 9]

Comparison

AlgorithmBest CaseWorst CaseKey Characteristic
Bubble sortO(n) — already sorted (one pass, no swaps)O(n²) — reverse sortedSimplest to understand and implement; least efficient for large data; good for teaching
Merge sortO(n log n)O(n log n) — always the sameMost efficient and consistent; works well on large datasets; requires extra memory for merging; uses divide and conquer
Insertion sortO(n) — already sortedO(n²) — reverse sortedGood for small datasets or nearly-sorted data; simple; sorts in-place (no extra memory)
You do not need to memorise code for these algorithms. You DO need to be able to trace through each algorithm on a given dataset, showing the state of the list after each pass/step. For merge sort, show both the complete split phase AND the merge phase. For bubble sort, show each comparison and swap within each pass.
