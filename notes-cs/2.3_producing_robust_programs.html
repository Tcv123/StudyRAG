2.3 Producing Robust Programs

OCR GCSE Computer Science (J277) — Paper 2

A robust program handles unexpected inputs and situations gracefully without crashing, is thoroughly tested, and is easy for other programmers to understand and maintain.

2.3.1 Defensive Design

Defensive design means writing code that anticipates problems before they occur. The goal is to ensure the program behaves correctly even when users make mistakes or deliberately try to break it.

TechniqueDescriptionExample
Anticipating misuseDesign the program to handle unexpected or malicious input without crashing. Assume users will enter wrong data types, leave fields blank, or try to exploit the system.If a form asks for age, what if the user enters "abc", -5, 999, or leaves it blank? The program must handle all these cases with clear error messages, not crash.
Input validationCheck that data meets all requirements before processing it. Common checks: range check, type check, length check, presence check, format check.Range check: age must be 0-120. Type check: age must be an integer (not text). Length check: password must be 8-64 characters. Presence check: email field must not be empty. Format check: email must contain '@' and a domain.
AuthenticationVerify the identity of a user before granting access to the system or specific features.Username/password login screen. Two-factor authentication (2FA): password + SMS code or authenticator app. Biometric: fingerprint or facial recognition.
// Input validation example (OCR pseudocode):
do
    age = int(input("Enter your age (0-120): "))
    if age < 0 OR age > 120 then
        print("Error: age must be between 0 and 120.")
    endif
until age >= 0 AND age <= 120

// Python equivalent with error handling:
while True:
    try:
        age = int(input("Enter your age (0-120): "))
        if 0 <= age <= 120:
            break
        else:
            print("Error: age must be between 0 and 120.")
    except ValueError:
        print("Error: please enter a whole number.")

2.3.2 Maintainability

Code that is easy to read, understand, and modify by other programmers (or by yourself months later) is said to be maintainable. This is critical in professional software development where teams work on shared codebases over years.

TechniqueWhy It MattersExample
Meaningful variable/subroutine namesSelf-documenting code: the name tells you what it does. totalScore is clear; ts or x is not.calculateTax(income) vs ct(i). studentName vs sn. isGameOver vs flag.
CommentsExplain what a block does and WHY. Future programmers (including your future self) need context. Do not comment obvious code (x = x + 1 // add 1 to x is pointless).// Calculate VAT at 20% and add to subtotal before the relevant code block. // Binary search requires sorted data to explain a pre-condition.
IndentationVisually shows the structure of loops, conditions, and subroutines. Makes it easy to see which code is inside which block. Logic errors become easier to spot.All code inside an IF block is indented one level; code inside a nested loop is indented two levels.
Use of subroutinesBreak code into small, focused modules. Each subroutine does one thing. Individual parts are easier to test, debug, reuse, and update independently.Instead of 200 lines in main(), have readData(), processScores(), displayResults(). If the display format changes, only displayResults() needs updating.
Constants instead of "magic numbers"Give meaningful names to fixed values. const MAX_LIVES = 3 is clearer than using 3 throughout the code. If the value needs to change, update one line instead of searching for every occurrence.const VAT_RATE = 0.2 instead of writing 0.2 in multiple places.

2.3.3 Testing — Types of Test Data

Every program must be tested with carefully chosen test data to verify it works correctly. There are three essential types:

TypePurposeExample (valid range: 1-100)
Normal (valid) dataConfirm the program works correctly with typical, expected input that falls well within the valid range.Enter 50 — should be accepted and processed normally. Also test other typical values like 25, 75.
Boundary (edge) dataTest values at the very edges of the acceptable range — both just inside and just outside. These are where off-by-one errors are most likely.Enter 1 (lowest valid) and 100 (highest valid) — should be accepted.
Enter 0 (just below range) and 101 (just above range) — should be rejected.
Erroneous (invalid) dataConfirm the program rejects bad input gracefully with a clear error message, without crashing or producing wrong results.Enter -5 (negative), 200 (way out of range), "hello" (wrong data type), or leave blank (empty input) — should display a clear error message and ask again.
// Test plan for a program that accepts ages 1-120:
| Test # | Input    | Type      | Expected Result              |
|--------|----------|-----------|------------------------------|
| 1      | 25       | Normal    | Accepted, processed normally |
| 2      | 65       | Normal    | Accepted, processed normally |
| 3      | 1        | Boundary  | Accepted (lower boundary)    |
| 4      | 120      | Boundary  | Accepted (upper boundary)    |
| 5      | 0        | Boundary  | Rejected with error message  |
| 6      | 121      | Boundary  | Rejected with error message  |
| 7      | -5       | Erroneous | Rejected with error message  |
| 8      | "abc"    | Erroneous | Rejected with error message  |
| 9      | ""       | Erroneous | Rejected with error message  |
| 10     | 3.5      | Erroneous | Rejected (not a whole number)|

2.3.4 Testing Strategies

StrategyWhenDescriptionTools Used
Iterative testingThroughout development (ongoing)Each module/subroutine is tested as it is written, before moving on to the next. Errors are caught early when they are cheaper and easier to fix. Individual components are tested in isolation.Print statements (to check variable values at key points); trace tables; debugger stepping; breakpoints; unit tests.
Final (terminal) testingAfter development is completeTesting of the complete, integrated program. Checks that all parts work together correctly. Tests every execution path through the program. Includes user acceptance testing (UAT) where real users test the system.Full test plan with all test data types; integration tests; system tests; UAT; performance/stress testing.

Types of Errors

Error TypeWhat It IsWhen DetectedExample
Syntax errorThe code breaks the rules of the programming language. The program cannot run at all.At compile/interpretation time (before the program runs)Missing colon, misspelled keyword (prnit instead of print), mismatched brackets, missing endif.
Logic errorThe code runs without crashing but produces wrong results. The algorithm is incorrect.During testing (comparing actual output to expected output)Using + instead of -; wrong comparison operator (> instead of >=); loop runs one too many or too few times (off-by-one error).
Runtime errorThe program crashes during execution due to an operation that cannot be performed.During executionDivision by zero; accessing an array index that does not exist; trying to open a file that does not exist; running out of memory.
Always include all three types of test data (normal, boundary, erroneous) in test plans. For boundary testing, test BOTH sides of every boundary — if the valid range is 1-10, you need to test 0 (invalid), 1 (valid boundary), 10 (valid boundary), and 11 (invalid). This is where bugs hide. For error types: syntax = cannot run (detected by translator); logic = runs but wrong answer; runtime = crashes during execution.
