2.5.1 Levels of Programming Language

Programming languages exist at different levels of abstraction from the hardware. Higher-level languages are easier for humans; lower-level languages give more control over hardware.

LevelDescriptionExamplesAdvantagesDisadvantages
Machine code (lowest level) Binary instructions (patterns of 0s and 1s) executed directly by the CPU. No translation needed. Each instruction is a specific operation for a specific processor architecture. 01001000 10000001 Fastest possible execution (no translation overhead); direct hardware control; smallest possible code size Extremely difficult to read, write, and debug; processor-specific (not portable — code for Intel x86 will not run on ARM); very error-prone
Assembly language (low level) Human-readable mnemonics (short abbreviations) representing machine code instructions. Each mnemonic maps to exactly one machine code instruction (one-to-one correspondence). Translated by an assembler. MOV AX, 5
ADD AX, BX
LDR R0, 200		 More readable than machine code; direct hardware control (can access specific registers and memory addresses); useful for device drivers, OS kernels, embedded systems, performance-critical code Still processor-specific (not portable); difficult and time-consuming to write complex programs; requires detailed knowledge of the processor's architecture and instruction set
High-level language (highest level) English-like syntax with abstractions (variables, loops, functions, objects) that hide hardware details. One high-level statement may translate to many machine code instructions. Must be translated by a compiler or interpreter. Python, Java, C#, C++, JavaScript, Haskell Easy to read, write, and debug; portable (same code runs on different processors after recompilation); large standard libraries; faster development time; easier to maintain Slightly slower execution (translation overhead); less direct control over hardware; requires a translator to be installed

Why do we still need low-level languages? Some tasks require direct hardware control that high-level languages cannot provide: writing device drivers, operating system kernels, embedded system firmware, real-time systems, and performance-critical code (game engines, cryptographic algorithms). Assembly gives the programmer total control over exactly which instructions the CPU executes, how memory is used, and how hardware registers are manipulated.

2.5.2 Translators

A translator converts source code written by a programmer into machine code that the CPU can execute. There are three types: assemblers, compilers, and interpreters.
TranslatorInputHow It WorksOutput
AssemblerAssembly languageConverts each assembly mnemonic to its corresponding machine code instruction. One-to-one translation. Specific to one processor architecture.Machine code (object code) for a specific processor
CompilerHigh-level languageTranslates the ENTIRE source code at once (before execution). Performs lexical analysis, syntax checking, optimisation, and code generation. Produces a standalone executable file that can run independently.Executable file (.exe, binary)
InterpreterHigh-level languageTranslates and executes ONE LINE AT A TIME. No separate executable is produced. The source code is re-translated every time the program runs.No permanent output file — code is executed directly

Compiler vs Interpreter — Detailed Comparison

FeatureCompilerInterpreter
TranslationEntire program translated before any of it runsOne line/statement at a time, during execution
Execution speedFast — compiled executable runs directly as machine codeSlower — must re-translate the code every time it runs
Error reportingLists all errors found during compilation (can be overwhelming but comprehensive)Stops at the first error encountered (useful for fixing one error at a time)
DistributionDistribute the executable — source code stays private (intellectual property protected)Source code must be distributed to run the program (visible to everyone)
Development cycleSlower: edit → compile (wait) → run → debugFaster: edit → run immediately — great for prototyping and learning
PortabilityCompiled executable is platform-specific (need to recompile for each OS/architecture)Source code is portable — runs on any platform with the interpreter installed
OptimisationCan optimise across the entire program (dead code removal, loop unrolling, inlining)Limited optimisation (only sees one line at a time)
RequirementCompiler NOT needed on the user's machine (only the executable)Interpreter MUST be installed on every machine that runs the program
ExamplesC, C++, Rust, Go, Java (to bytecode)Python, JavaScript, Ruby, PHP

Hybrid approach: Many modern languages use both techniques. Java compiles source code to bytecode (intermediate representation), which is then interpreted (or JIT-compiled) by the Java Virtual Machine (JVM) at runtime. Python compiles to bytecode (.pyc files) which is interpreted by the CPython interpreter. This combines the error-checking benefits of compilation with the portability of interpretation.

2.5.3 IDE (Integrated Development Environment) Features

An IDE (Integrated Development Environment) is a software application that provides a comprehensive set of tools for writing, testing, and debugging code, all within a single interface. Examples: Visual Studio Code, PyCharm, Eclipse, IDLE, Thonny.
FeatureWhat It DoesWhy It Helps
Syntax highlightingColours different parts of the code differently: keywords in one colour, strings in another, comments in another, variables in another, numbers in another.Makes code much easier to read at a glance. Helps spot typos immediately — if a keyword is not highlighted in the expected colour, you know it is misspelled. Mismatched string quotes become visually obvious.
Error diagnostics / detectionFlags syntax errors, warnings, and potential problems as you type (red underlines, error markers in the gutter). Shows descriptive error messages explaining what is wrong and often suggesting fixes.Catches mistakes before running the program, saving time. Points to the exact line and position of the error. Some IDEs can auto-fix common errors.
Run / ExecuteCompile and run the program directly from within the IDE with a single button click or keyboard shortcut.No need to switch to a command line. Output is displayed in an integrated console/terminal within the IDE.
Debugger (stepping)Allows you to execute code one line at a time (step over, step into, step out), inspecting the state of variables and the call stack at each step.Identifies exactly where and why logic errors occur. You can watch variables change value step by step, which is far more powerful than adding print statements everywhere.
BreakpointsMark specific lines where execution should pause automatically. When the program reaches a breakpoint, it stops, and you can inspect all variable values at that precise moment.Allows you to run the program at full speed until the interesting part, then pause and investigate. You can set conditional breakpoints that only trigger when a specific condition is true.
Auto-completion / IntelliSenseSuggests variable names, function names, method names, parameters, and keywords as you type. Shows function signatures and documentation on hover.Reduces typos; speeds up coding; helps discover available functions and methods without consulting documentation; ensures correct spelling of identifiers.
Version control integrationBuilt-in support for Git/GitHub: commit changes, view history, compare versions, create branches, merge, push/pull.Revert to previous working versions if something breaks; track who changed what and when; essential for team collaboration; safely experiment with new features on branches.
Code refactoring toolsAutomated renaming of variables/functions across the entire project; extracting code into subroutines; reorganising code structure.Safely rename a variable everywhere it is used (not just the current file). Reduces risk of errors during code reorganisation.
Compiler: translates everything first → fast to run, source not distributed, all errors at once. Interpreter: line by line → easy to debug (stops at first error), slower to run, source must be distributed. For IDE questions, always name the specific feature AND explain why it helps the programmer — examiners award marks for both parts. Simply listing features without explaining their benefit will not get full marks.
