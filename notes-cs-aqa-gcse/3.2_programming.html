3.2.1 Data Types

A data type is a classification that tells the computer what kind of value a variable holds, how much memory to allocate for it, and what operations can legally be performed on it. Choosing the correct data type is a fundamental part of writing a reliable program — using the wrong type causes bugs, wastes memory, or produces unexpected results (for example, concatenating instead of adding).

Every value a program handles — a price, a name, a password, a temperature reading, a win/lose flag — must have a data type. The AQA specification requires you to know five basic types: integer, real (float), Boolean, character and string. You should be able to recognise each in pseudo-code, choose the appropriate type for a given scenario, and convert (cast) between them when needed.

The Five AQA Data Types

Note that a character is not the same as a string, even when the string contains only one letter. 'A' (in single quotes) is a character; "A" (in double quotes, with length 1) is a string. Similarly, the digit '7' as a character is not the same as the integer 7; the character holds the symbol 7, not the quantity 7, and they have different binary representations (the character uses its ASCII / Unicode code point).

When asked "which data type is most appropriate for X?" choose the smallest or most specific type that can hold the value. A pupil's age is an integer, not a real (you do not need 12.47 years). A question answered yes/no is a Boolean, not a string "yes"/"no". A postcode is a string, not an integer (it contains letters). Marks are lost for storing numeric data as strings, as this blocks arithmetic without casting.

Casting (Type Conversion)

Casting (or type conversion) is converting a value from one data type to another. It is needed when a value arrives in one form (e.g. text from keyboard input) but must be used in another (e.g. arithmetic on a number). AQA pseudo-code uses functions named after the destination type: int(), float() (or real()), str(), bool().

The most common place casting appears is reading input from the user. Keyboard input always arrives as a string, even if the user typed digits. If you want to do arithmetic you must cast it first. Forgetting to do so is one of the most common beginner bugs and is frequently tested in AQA exam questions.

# Without casting — BUG
age ← USERINPUT         # age is the string "16"
next_age ← age + 1       # ERROR or "161" depending on language

# With casting — correct
age ← int(USERINPUT)    # age is the integer 16
next_age ← age + 1       # next_age is 17

# Casting for output
price ← 9.99
OUTPUT "Total: £" + str(price)   # concatenation requires a string

3.2.2 Programming Concepts

Computers follow instructions in a strictly ordered way. AQA requires you to understand the three core building blocks of every algorithm — sequence, selection and iteration — plus the ideas of variables, constants, assignment, and meaningful identifiers.

Variables and Constants

A variable is a named memory location that holds a value which can change while the program runs. A constant is a named memory location that holds a value which cannot change after it is first set. Both have a name (identifier) and a current value; the only difference is whether the value is allowed to change.

Think of a variable as a labelled box. The label (score, user_name) never changes, but the value inside the box can be replaced as the program runs. A constant is a labelled box that has been locked shut after you put its value in — you (and future readers of the code) can still read the value but you cannot change it. Constants are used for values that should not change, such as mathematical constants (PI = 3.14159), configuration values (VAT_RATE = 0.20, MAX_SCORE = 100) or conversion factors.

# Variables (changeable)
score ← 0
score ← score + 10        # now 10
score ← score + 5         # now 15

# Constants (unchangeable, usually written in UPPERCASE by convention)
CONSTANT VAT_RATE = 0.20
CONSTANT MAX_LIVES = 3
price_with_vat ← price * (1 + VAT_RATE)

Using constants instead of "magic numbers" has three benefits: (1) the code is more readable because VAT_RATE explains what 0.20 means; (2) it is easier to update because you only change the value in one place; (3) it prevents accidental changes that would cause hard-to-find bugs.

Assignment

Assignment stores a value in a variable. AQA pseudo-code uses the left-arrow ← (read as "becomes" or "is assigned"). Many textbooks and most programming languages use = or :=. The important thing is the direction: the value on the right is computed first, then stored in the variable on the left.

x ← 5             # x becomes 5
x ← x + 1         # right side first: (5 + 1) = 6, then stored in x. x is now 6.
name ← "Ada"     # name is a string
ready ← True     # ready is a Boolean

x ← x + 1 is not an equation — it would be nonsense in mathematics. It means: take the current value of x, add 1 to it, and put the result back into x. This pattern is the basis of counters, accumulators and almost every loop.

Meaningful Identifiers

An identifier is the name given to a variable, constant, subroutine or other program element. A meaningful identifier clearly describes what the item represents. Good identifiers make code self-documenting — a reader can understand the purpose of a variable from its name alone, without needing comments.

Loop counters (i, j, k) are the one accepted exception where single letters are still considered meaningful.

The Three Programming Constructs

Every algorithm can be built from just three types of flow of control. Being fluent in all three is central to writing correct programs.

1. Sequence

Sequence is the most basic: statements are executed one after another, from top to bottom, in the order they are written. Every piece of code is a sequence unless selection or iteration changes the flow.

OUTPUT "Welcome"
name ← USERINPUT
OUTPUT "Hello " + name
OUTPUT "Goodbye"

2. Selection (IF / ELSE IF / ELSE)

Selection means choosing which block of code to execute based on a condition. AQA uses IF condition THEN … ELSE IF condition THEN … ELSE … ENDIF. Only one branch is executed each time the statement is reached — whichever one has the first condition evaluating to True (or the ELSE branch if none do). 
INPUT mark
IF mark ≥ 80 THEN
    OUTPUT "A"
ELSE IF mark ≥ 70 THEN
    OUTPUT "B"
ELSE IF mark ≥ 60 THEN
    OUTPUT "C"
ELSE
    OUTPUT "Fail"
ENDIF

Note that the branches are mutually exclusive: for mark = 85 only the first branch runs even though 85 is also ≥ 70 and ≥ 60. This is why the order of ELSE IF tests matters — the most specific / highest condition must come first.

3. Iteration (Loops)

Iteration means repeating a block of code. AQA distinguishes two kinds: count-controlled loops (FOR) that repeat a known number of times, and condition-controlled loops (WHILE) that repeat while a condition is True (an unknown number of times).

Count-Controlled Loops (FOR)

A FOR loop uses a loop counter that starts at a known value, increases by a step (usually 1) each time, and stops when it passes a known end value. Use it when you know in advance how many iterations are required.

# Print the numbers 1 to 5
FOR i ← 1 TO 5
    OUTPUT i
ENDFOR

# Sum the first 10 whole numbers
total ← 0
FOR i ← 1 TO 10
    total ← total + i
ENDFOR
OUTPUT total        # 55

# Step other than 1 (AQA uses TO … STEP)
FOR i ← 0 TO 20 STEP 2      # even numbers 0, 2, 4, … 20
    OUTPUT i
ENDFOR

Condition-Controlled Loops (WHILE)

A WHILE loop keeps repeating while a condition is True. It is checked before each iteration, so the loop body might not run at all if the condition starts as False. Use it when you do not know in advance how many iterations are needed — for example, keep asking the user for a valid password, keep processing file lines until the end, keep applying interest until a balance exceeds a target.

# Keep asking until the user enters the correct password
password ← ""
WHILE password ≠ "hello"
    OUTPUT "Enter password: "
    password ← USERINPUT
ENDWHILE
OUTPUT "Access granted"

Choose FOR when you know the number of iterations up-front (e.g. "process all 30 pupils" or "draw a 10×10 grid"). DIV and MOD — Why They Matter

Together, DIV and MOD split a division into its whole-number part and its remainder, exactly as you were taught to do long division in primary school. 23 ÷ 5 = 4 remainder 3 → 23 DIV 5 = 4 and 23 MOD 5 = 3.

These two operators have many uses:

  • Test for even/odd: n MOD 2 = 0 is true for even numbers, = 1 for odd.
  • Test divisibility: n MOD 5 = 0 tests whether n is a multiple of 5.
  • Extract the last digit: n MOD 10 gives the last digit of n (e.g. 2468 MOD 10 = 8).
  • Strip the last digit: n DIV 10 removes the last digit of n (e.g. 2468 DIV 10 = 246).
  • Convert total seconds to minutes and seconds: total DIV 60 = minutes, total MOD 60 = leftover seconds.
  • Wrap around a range: i MOD n always gives a value 0 … n−1, useful for circular array indices.
# Convert 215 seconds to minutes and seconds
seconds ← 215
mins ← seconds DIV 60     # 3
secs ← seconds MOD 60     # 35
OUTPUT mins + " min " + secs + " s"   # "3 min 35 s"

When an expression mixes several operators, the computer follows the same precedence rules as mathematics (BIDMAS / PEMDAS). Use brackets to force a different order or simply to make intent clear.

A classic exam pitfall is writing something like IF grade = "A" OR "B". This is wrong — "B" on its own is not a condition. It must be written as IF grade = "A" OR grade = "B". Each side of an AND / OR must be a complete Boolean expression.

3.2.6 Data Structures — Arrays and Records

A variable holds a single value. To handle collections of related values — the 30 marks of a class, the 7 days' rainfall, the 9 squares of a noughts-and-crosses board — we need a data structure. AQA requires you to know 1D arrays, 2D arrays, and (in some materials) records.

One-Dimensional Arrays

A one-dimensional array is an ordered collection of items of the same type, stored under a single identifier. Each item is accessed by its index (position). AQA uses zero-based indexing: the first item is at index 0, the second at index 1, and so on. An array of length n has valid indices 0 to n−1. 
# Declaring and using a 1D array
scores ← [72, 85, 64, 90, 58]

OUTPUT scores[0]          # 72
OUTPUT scores[3]          # 90
scores[2] ← 70          # change the third item

# Iterating over an array
FOR i ← 0 TO length(scores) - 1
    OUTPUT scores[i]
ENDFOR

In AQA pseudo-code, length(list) returns the number of items. The loop therefore goes from 0 to length(scores) - 1 inclusive — one of the most common array-traversal patterns you will see in the exam.

Two-Dimensional Arrays

A two-dimensional array is an array of arrays — a grid of values organised into rows and columns. Items are accessed with two indices: row first, then column. 2D arrays are used for tables of data, game boards, spreadsheets, images, matrices — anything with a natural rectangular shape. 
# 3 rows, 4 columns — class scores for 4 tests
scores ← [ [72, 85, 64, 90],
            [58, 77, 81, 66],
            [91, 84, 73, 88] ]

OUTPUT scores[0][0]       # 72 (row 0, col 0 — top-left)
OUTPUT scores[2][3]       # 88 (row 2, col 3 — bottom-right)
scores[1][2] ← 85       # change row 1, col 2

# Nested loop to read every element
FOR row ← 0 TO 2
    FOR col ← 0 TO 3
        OUTPUT scores[row][col]
    ENDFOR
ENDFOR
first, column index second. • Indexing starts at 0. A 3×4 array uses rows 0–2 and columns 0–3. • An out-of-range index (e.g. scores[3][0]) is a run-time error.

Records

A record is a data structure that groups together a fixed set of named fields, which may be of different types. For example a Student record might contain a name (string), date of birth (string), year_group (integer), and is_present (Boolean). Unlike arrays — which hold many items of the same type accessed by an index — records hold one item of each type accessed by its field name. 
# A record representing a student
student ← {
    name: "Ada Lovelace",
    year: 10,
    form: "10B",
    is_present: True
}

OUTPUT student.name         # Ada Lovelace
OUTPUT student.year         # 10
student.is_present ← False

In AQA pseudo-code, a record field is accessed with a dot (student.name). Records are very useful for representing real-world "things" whose properties are well-defined: a Book (title, author, ISBN, year), a Transaction (date, amount, payee, category), a Flight (code, origin, destination, departure_time, gate). An array of records is a common way to represent a table of data — e.g. an array of 30 Student records for a whole class.

3.2.7 Input / Output and File Handling

Programs need to communicate with the outside world — reading input from the user or from a file, and writing output to the screen or a file. AQA covers three channels: keyboard input, screen output, and text file read/write.

Keyboard Input and Screen Output

OUTPUT "Please enter your name: "
name ← USERINPUT          # reads a line typed by the user
OUTPUT "Hello, " + name

# Numeric input requires casting (USERINPUT returns a string)
OUTPUT "Enter your age: "
age ← int(USERINPUT)
OUTPUT "Next year you will be " + str(age + 1)

OUTPUT displays text on the screen. USERINPUT (sometimes just INPUT) reads a line of text from the keyboard, including anything the user types right up to when they press Enter. The value returned is always a string, even if they typed digits, so numeric input must be cast before arithmetic.

Text File Handling

A text file stores data as a sequence of lines of characters. AQA requires you to know four basic file operations: open, read (a line), write (a line), and close.

Three essentials in every file-handling exam answer: (1) open the file (specifying read or write), (2) process it line by line (usually inside a WHILE NOT endOfFile loop for reading), (3) close the file at the end. Forgetting to close is a common mark-loser; it can leave the file locked or data unwritten.

String to Integer and Back

Because user input and file data arrive as strings, converting between strings and numbers is extremely common. Always cast as soon as the type is wrong for the operation you want to do.

price_str ← "9.99"
price ← float(price_str)       # 9.99 as a real number
total ← price * 3              # 29.97

# Going the other way — building output
mins ← 3
secs ← 35
OUTPUT str(mins) + " min " + str(secs) + " s"    # "3 min 35 s"
Two strings that look the same but are typed differently are not equal. Watch out for trailing spaces ("hello " vs "hello"), different cases ("Hello" vs "hello"), and "7" versus the integer 7. To compare ignoring case, convert both sides with toLower() first. To compare with any whitespace removed, use a trim() function.

3.2.9 Random Number Generation

A random number is a value chosen unpredictably from a specified range. Programs use random numbers to simulate chance events — dice rolls, card shuffles, lottery draws, procedurally-generated levels, AI behaviour — and to pick test data, encryption keys, or unique IDs. Truly random numbers are hard to produce by software alone, so most languages provide pseudo-random generators that behave randomly enough for most purposes.

AQA pseudo-code uses a random(low, high) function that returns an integer chosen uniformly from low up to and including (or up to but not including, depending on the spec) high. Check the exam rubric; this guide uses inclusive both ends.

# Simulate a single six-sided die roll
dice ← random(1, 6)
OUTPUT "You rolled " + str(dice)

# Pick a random letter of the alphabet
letter_code ← random(65, 90)         # ASCII codes for 'A' to 'Z'
letter ← CHR(letter_code)

# Choose a random element of an array
favourites ← ["Dog", "Cat", "Rabbit", "Fish", "Hamster"]
idx ← random(0, length(favourites) - 1)
OUTPUT "Today's pet of the day is: " + favourites[idx]
A frequent exam question is: "Write code to simulate rolling a pair of dice." The answer is two separate calls to random(1, 6), not one call to random(2, 12). Calling once gives 2–12 with equal probability, which is wrong — 7 should be six times more likely than 2 on real dice. Two dice combined give a triangular distribution.

3.2.10 Subroutines: Procedures and Functions

A subroutine is a named block of code that performs a specific task. You can call (invoke) a subroutine by its name from elsewhere in your program, passing in any values it needs as parameters. AQA distinguishes two kinds: a procedure carries out a task (e.g. prints a menu, writes to a file) but does not return a value; a function returns a value to its caller.

Procedures

PROCEDURE greet(name)
    OUTPUT "Hello, " + name + "!"
    OUTPUT "Welcome to the program."
ENDPROCEDURE

# Calling the procedure
greet("Alice")               # prints "Hello, Alice!" then "Welcome to the program."
greet("Bob")                 # prints "Hello, Bob!"  then "Welcome to the program."

Functions

FUNCTION circle_area(radius)
    CONSTANT PI = 3.14159
    RETURN PI * radius * radius
ENDFUNCTION

area ← circle_area(5)                # area is now 78.53975
OUTPUT "Area = " + str(area)
OUTPUT "Area of r=10 is " + str(circle_area(10))       # functions can be used in expressions

A RETURN statement inside a function both produces the result and ends the function early, so code after a taken RETURN is not executed. Procedures usually do not have a RETURN and simply end when the last statement runs (or optionally end early with RETURN on its own with no value).

Parameters and Arguments

A parameter is a variable named in the subroutine definition (the placeholder). An argument is the actual value supplied when the subroutine is called. In circle_area(radius) above, radius is the parameter; in circle_area(10), 10 is the argument.

Subroutines may have zero, one, or many parameters. Each parameter has a name and (implicitly) a type inferred from the argument passed in.

FUNCTION rectangle_area(width, height)
    RETURN width * height
ENDFUNCTION

OUTPUT rectangle_area(4, 6)         # 24
OUTPUT rectangle_area(10, 2.5)      # 25.0

Local and Global Variables

A local variable is declared inside a subroutine and exists only while that subroutine is running — it is not visible to code outside. A global variable is declared outside all subroutines and is visible throughout the whole program. Using local variables where possible is strongly preferred because it keeps each subroutine self-contained and avoids unintended interactions. 
GLOBAL high_score ← 0          # global — visible everywhere

PROCEDURE play_round()
    round_score ← random(1, 100)     # local to play_round
    IF round_score > high_score THEN
        high_score ← round_score     # we may update the global
    ENDIF
    OUTPUT "You scored " + str(round_score)
ENDPROCEDURE

play_round()
OUTPUT "Best so far: " + str(high_score)
# OUTPUT round_score              # ERROR — round_score is not visible here

Benefits of Subroutines

Subroutines are one of the most important tools in good programming. The benefits appear in almost every exam:

  • Reusability. A subroutine is written once and can be called as many times as needed, with different arguments. Repeated code becomes a single definition.
  • Modularity. A large program can be broken into small, focused subroutines — one job per subroutine. Each can be understood, tested, and replaced on its own.
  • Readability. A line like total ← calculate_bill(items, discount) instantly communicates what is happening, even without reading the underlying code.
  • Maintainability. When a rule changes (e.g. the VAT rate), you change it once in the subroutine instead of in many places throughout the program.
  • Easier debugging and testing. Each subroutine can be tested with its own inputs, isolating bugs to smaller pieces of code.
  • Team working. Different programmers can write different subroutines in parallel, as long as they agree on the parameters and return values.
If a question asks "give three benefits of using subroutines", the safest answers are reusability, readability, and easier to maintain / debug. Always give why each helps, not just the word itself — e.g. "Subroutines make code reusable, because the same procedure can be called multiple times instead of copying the code each time."

3.2.11 Structured Programming

Structured programming is an approach to writing programs that uses only the three basic constructs (sequence, selection, iteration), avoids unrestricted jumps (like the classic GOTO), and decomposes programs into self-contained subroutines each solving a clear sub-problem. Structured programs have an obvious logical flow and are made of small, well-named blocks.

The term dates from the 1960s, when programs were often written as long tangles of GOTO statements jumping backwards and forwards — so-called "spaghetti code". Structured programming pioneers (Dijkstra, Wirth, Hoare) argued that almost all control flow can be expressed with sequence, selection, and iteration alone, and that doing so makes programs far easier to reason about, maintain, and debug. Today every mainstream programming language supports structured programming, and most of the AQA pseudo-code style is explicitly structured.

Advantages of Structured Programming

  • Easier to read and understand. Anyone familiar with IF, FOR, WHILE and subroutine calls can follow the logic without tracing unexpected jumps.
  • Easier to maintain. Each subroutine does one job, so changes are localised rather than rippling through the whole program.
  • Easier to test and debug. Small, self-contained subroutines can be tested individually. When a bug appears, the area to search is small.
  • Promotes code reuse. Well-designed subroutines can be called from many places, or even reused in future programs.
  • Supports teamwork. Different team members can work on different subroutines independently as long as their parameters and return values are agreed.
  • Reliability. Removing GOTO and other unconstrained jumps eliminates a whole class of subtle bugs.

Putting It All Together — A Worked Example

The program below reads a list of test scores from a file, calculates the average, and writes a short report. It combines almost everything in Topic 3.2: file handling, arrays, iteration, arithmetic (including DIV where appropriate), selection, string handling, casting, and subroutines with parameters and return values.

# ---------- Subroutines ----------

FUNCTION read_scores(filename)
    scores ← []
    f ← openRead(filename)
    WHILE NOT f.endOfFile()
        line ← f.readLine()
        scores.append(int(line))       # each line is one integer
    ENDWHILE
    f.close()
    RETURN scores
ENDFUNCTION

FUNCTION average(list)
    total ← 0
    FOR i ← 0 TO length(list) - 1
        total ← total + list[i]
    ENDFOR
    RETURN total / length(list)
ENDFUNCTION

PROCEDURE write_report(filename, avg)
    f ← openWrite(filename)
    IF avg ≥ 70 THEN
        f.writeLine("Class average: " + str(avg) + " - excellent")
    ELSE IF avg ≥ 50 THEN
        f.writeLine("Class average: " + str(avg) + " - satisfactory")
    ELSE
        f.writeLine("Class average: " + str(avg) + " - needs improvement")
    ENDIF
    f.close()
ENDPROCEDURE

# ---------- Main program ----------

scores ← read_scores("scores.txt")
avg    ← average(scores)
write_report("report.txt", avg)
OUTPUT "Report written. Average = " + str(avg)

Notice how the main program is only four lines long and reads almost like English. All the detail is hidden inside small, single-purpose subroutines — exactly the pattern structured programming encourages.

When asked "state three advantages of structured programming", a strong answer pairs each advantage with a reason: "easier to debug because each subroutine can be tested on its own; easier to maintain because changes are localised to the relevant subroutine; supports teamwork because different people can work on different subroutines in parallel." Avoid vague answers like "it is better" with no reason.
